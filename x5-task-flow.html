03 / 拆开

任务流的
5 个解剖位。

一步任务 = 进入条件 + 行动 + 副作用 + 退出条件 + 状态迁移。 把这 5 件事拉出来 —— 点下面流程图的任意一步看它的 5 个槽位。

点流程图任意一步 —— 弹出它的 5 个槽位：进入条件 / 行动 / 副作用 / 退出条件 / 状态迁移。

STEP 03 · 校验

已查看 0 / 5 步 · 完整查看才能进入下一节
副作用最容易被忘掉

很多设计稿只描述 UI——按钮长什么样、放哪、按下去跳到哪一页——但不描述"按了这个按钮后端发生什么"。

一句话 提交后发送邮件给审批人 + 锁定金额 + 创建审批任务 就避免一周返工。

副作用 = 用户看不见、系统在做的事。在设计稿里写出来，工程师才知道要不要起 background job、起几个。

5 步 × 5 槽位 = 25 个决策。
任何一格空着，工程师就要追问。

04 / 看更多

「新员工入职」任务流，
看人家怎么拆 5 步。

HR 要做"新员工入职"流程。下面直接给你 5 步 + 行动 + 副作用 + 退出条件—— 每一步说明"谁做、做什么、做完触发什么"。

新员工入职 · ONBOARDING · 5 步任务流

已经填好——重点在每一步的"谁做 / 副作用 / 退出条件"。

1
HR 操作
创建入职单
行动：HR 在系统里录入新员工基本信息（姓名/部门/入职日期/工号）。
副作用：系统不发邮件、不建账号——这只是占位记录。
退出 = 单子保存
→
2
新员工操作
填写资料
行动：新员工收到链接，填学历 / 银行卡 / 紧急联系人。
副作用：每填一栏自动 autosave，关页面不丢。
退出 = 必填全填
→
3
新员工操作
上传材料
行动：上传身份证 / 学历证书 / 体检报告 PDF。
副作用：OCR 自动识别 + 提示"是否一致"。
退出 = 全部上传 ✓
→
4
主管操作
审批
行动：直接主管收到通知，看资料 → 同意 / 驳回 / 补材料。
副作用：同意 → 锁定工号 + 发欢迎邮件。
退出 = 主管点同意
→
5
系统自动
开权限
行动：系统调用 IT API，建邮箱 / VPN / OA 账号。
副作用：账号信息发给员工 + HR 看板状态变"已入职"。
退出 = 自动 · 无人触发
⚠ 容易踩 3 个坑：
① 第 1 步就建账号——HR 一录信息就开邮箱权限，员工还没审批通过就有了账号 → 泄密风险。账号要等审批后才建。
② 第 4 步退出条件搞错——以为"入职日期到了"就算审批通过 → 主管没点同意也走完流程。退出条件 = 主管显式点同意，不是时间。
③ 第 5 步没标"系统自动"——工程师以为还要 HR 手动点一下 → 多做一个按钮 + 一遍审批 UI。"系统自动"是一种步骤类型，要标出来。
05 / 把这件事变成习惯

下次接需求，
先画"流程条"。

从这一篇开始，"页面思维"和"任务流思维"的差距会出现在你每一次需求评审的发言里。

i

画稿子前先画"流程条"

任务从哪开始、有几步、每步谁做、什么时候完成。

一句话能说清的流程，page mockup 才有锚点。

ii

副作用 ≠ UI

按钮按下去后端做什么、什么时候发邮件、什么时候锁定金额——这些不在 UI 里但必须在设计稿里。

"副作用"是任务流的隐形车道。

iii

每一步都问"退出条件"

这一步什么时候算完成？是用户点'下一步'？是后端校验过？是审批人同意？

退出条件不清楚，每个团队会做出不一样的实现。

划重点 · TL;DR

同一个任务可以全程一页（modal）、可以 5 页、可以全程零页（API 触发）。先设计任务，再决定页面承载方式——而不是反过来。

需求评审第一句不是"怎么画"，是"这是几步任务，每步谁做、做完什么时候算完"。语言一变，所有人都跟着变。
