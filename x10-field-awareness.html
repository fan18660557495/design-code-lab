01 / 问题

"加个下拉框"
背后藏着5 个问题。

设计师以为字段 = "一个输入框 + 一个标签"。其实字段 = 必填? + 空值含义 + 取值范围 + 可改性 + 是否派生 5 维度。 少标一维 = 工程师猜 1 次。

字段当输入 · INPUT

"稿子里有个'状态'下拉框"

设计稿画了一个 dropdown，4 个选项。看起来清楚。

工程师：必填吗？null 时显示啥？这 4 个选项是写死还是从后端取？运营能加新选项吗？status 能改吗（已发货还能改回未支付吗）？是计算字段（从其它字段派生）吗？

设计师：呃……我再回去问下产品。

字段当 SCHEMA

"状态字段，规则如下"

5 维度全部标清楚——工程师直接落 schema。

状态 status: enum (4 选项 + 1 个 not-set) · 必填 · 空值显示"未设置" · 运营可加但要后台 · 只读（只能通过状态机迁移）· 不是计算。

结果：追问 → 0 次。

05 / 把这件事变成习惯

下次需求评审，
第一句问这 5 件事。

从这一篇开始，"输入框思维"和"schema 思维"的差距会出现在每一次字段定义里。

i

每个字段标 5 维度

必填? · 空值含义 · 取值范围 · 可改性 · 是否计算。

不画 = 工程师追问 5 次。

ii

"空"不是"没有"——是设计语言

"—" / "未设置" / "0" / 隐藏 / "未联系"——每种语义不同。null（没填）和 0（填了零）必须分开画——库存 null vs 库存 0，决策完全两回事。

"空"是一种字段状态，要被设计。

iii

"计算字段"不出现在 form 里

计算字段不该有 input — 它是派生的，写入毫无意义。

看到 form 里包计算字段，是 schema 设计错。

划重点 · TL;DR

需求评审拿到字段，第一句问 5 维度——必填、空值含义、范围、可改性、是否计算。

把这 5 件事写进稿子，工程师能直接落 schema。否则你交付的不是设计稿——是5 个未解之谜。
