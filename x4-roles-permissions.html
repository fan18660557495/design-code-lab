02.5 / 想清楚

权限有 三层，
不只是"显示/隐藏"。

看到 4 张稿不一样之后，先抽出一层概念—— 可见 / 行 / 列，这三层决定了下面那张矩阵里每一格到底是怎么"不一样"。

第 1 层 · WHO CAN SEE IT

可见性

visibility · 看得到 / 看不到

最粗的一层——整块东西的有无。导航条、菜单、卡片、按钮，对某些角色根本不存在。

UI 表现 客服界面里压根没有"成单分析"这个模块；客户本人界面里没有删除客户按钮——不是 disabled，是不渲染。
背后：route 守卫 / nav 项隐藏 / 按钮 unmount。设计稿要列出每个角色的"入口清单"。
第 2 层 · WHICH ROWS

行权限 · 筛选

row-level · 看哪些条

你看得到这个列表，但只能看到属于你的那些行。同一个客户列表页，销售只看自己的客户，老板看全公司。

UI 表现 列表数量（"我的客户 12"vs"全部客户 4,820"）、空态文案（"暂无属于你的客户"vs"全公司没有客户"——两件事差别巨大）。
背后：query WHERE owner_id = me / RLS / filter middleware。设计稿要画"我的视角"的空态、计数、默认筛选。
第 3 层 · WHICH FIELDS

列权限 · 裁切

column-level · 看哪些字段

你看得到这条记录，但有些字段被遮罩、脱敏、或整个不显。详情页里那个 `—` 可能是"没填" / "*** 脱敏" / "你没权限看"——三件事。

UI 表现 客服看到客户电话 138****1234（脱敏）；销售看到完整 13812341234；普通访客这一字段整行消失——三种语义，三种 UI。
背后：field-level 过滤 / mask 函数 / API select 黑名单。设计稿要明说每个字段在每个角色下的呈现，别留给"前端按需隐藏"。
下面那张矩阵的每一格，背后都是这三层的组合——
能看的字段 = 列权限 + 可见性 · 能做的操作 = 可见性 + 行权限 · 空态文案 = 行权限的不同结果集。
三层是骨架，矩阵是展开图。先画三层，再填矩阵——别把"加个隐藏开关"当作权限设计。

04 / 看更多

新角色 「审计员」 来了——
看人家怎么配矩阵。

审计员的工作语境是 「独立 · 只读 · 可追溯」——审查交易、查看金额、确认合规，但不改任何数据，也不让客户感知审计存在。 下面直接给你 5 维度配置 + 每一格的"为什么"。

审

审计员 · AUDITOR

5 个维度 × 已经选好 · 看每一格的为什么

维度 审计员看到 / 能做
① 字段可见 看金额、看合同、看跟进备注 · 不看电话邮箱原文 审计的工作是"核对数字"——金额、开票、流水必须全开。但电话邮箱是隐私，看到无意义反而违规。列权限只开"金额财务" + 脱敏"个人联系"。
② 操作可做 只读所有客户数据 · 但能"加审计备注"和"导出报告" 写操作（改 / 删 / 加商机）一律不开。但她要留下审计意见——添加备注 / 标记合规 / 导出报告。这些是审计自己的产物，不修改客户数据。
③ 行权限 / 看哪些 看全公司所有客户 · 不按 owner 筛 审计不是销售——不能"我的客户"。她必须看全部，才能抽样。行权限放到顶 = 看全公司，反着销售的"只看自己"。
④ 状态摘要 / 顶部信息 显示"是否抽审" + "上次审计时间" 其他角色看 SLA、跟进进度——审计员只关心"这条记录被审过吗 / 何时"。状态摘要是给当前角色的工作语境定调。
⑤ 空态文案 "本月暂无待审记录"（不是"暂无数据"） 空态最容易被复制粘贴。每个角色的"空"含义不同——销售看"暂无客户"是焦虑，审计看"暂无待审"是好消息（说明公司合规）。文案不该一套通用。
⚠ 最容易掉的坑：把审计员当"管理员的只读版"——给同样的字段、同样的状态摘要、同样的空态文案，只是按钮置灰。这不是设计，是偷懒——审计员的工作语境完全不同，状态摘要应该是"上次审计时间"，不是"SLA 剩余"。
05 / 把这件事变成习惯

下次接到"角色权限"，
先画矩阵。

权限不是"加几个 if"。每个角色都有自己的工作语境，要为它单独跑一遍设计。

i

角色是"维度"，不是"开关"

每个角色都是一份独立的工作语境。"隐藏几个字段"是症状，"为每个角色画一张矩阵"才是设计。

多角色 ≠ 多 if-else。

ii

画一个角色之前问"它在解决什么问题"

销售在卖、财务在算账、客服在救火、审计在查证。语境不同，整张页面的字段 / 操作 / 语言都该不同。

语境先于布局。

iii

空态是角色最容易被忽略的差异

同一个空列表，文案在不同角色眼里意义完全不同。空态文案是判断你是否真的为这个角色设计过的试金石。

稿子里有 4 段空态？说明你跑过 4 遍。

划重点 · TL;DR

权限不是"详情页底部加个开关"。同一份数据 × N 个角色 = N 套字段 / 操作 / 状态 / 空态 / 跳转。每个角色都是从对象的同一池里取一个完全不同的子集。

下次接到"角色权限"需求，第一句不是 "要加什么 if"，是 "要拉一张 5×N 的差异矩阵"。
