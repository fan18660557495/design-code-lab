03 / 拆开

组合 4 件事。

组件组合时要回答 4 件事——容纳关系 / 数据下行 / 事件上行 / 状态归属。 点每张卡片展开看坏 vs 好的代码对比。

1 容纳关系contain

谁是父，谁是子？哪些组件该嵌套，哪些该平级？

反例：把 Filter 放 Table 内部——以后想做"一个 Filter 控制多个 Table"就废了。Filter 和 Table 被绑死，复用性归零。
经验：可独立使用的组件分开放（Filter 独立于 Table，便于多 Table 共用一个 Filter，或者 Filter 独立用于地图视图）。
BAD
// Filter 嵌死在 Table 内
<Table>
  <Filter />
  <Rows />
</Table>
GOOD
// 平级，独立复用
<FilterBar />
<Table />
<MapView />
2 数据下行data flow down

数据从哪一层加载？再往下传到哪？

反例：Card / Table / Timeline 每个子组件自己 fetch——一个页面打 N 个请求（N+1 问题），网络瀑布、数据不一致。
经验：找最近的"共同父"——它持有数据，往下传 props。子组件保持"哑组件"（只渲染，不取数）。
BAD
<Page>
  <Card/>  // fetch
  <Table/> // fetch
  <Tl/>    // fetch
</Page>
GOOD
<Page> // 1 次 fetch
  <Card data={c}/>
  <Table data={opps}/>
  <Tl data={acts}/>
</Page>
3 事件上行events bubble up

子组件的事件通知哪一层？谁负责"做出反应"？

反例：Table 自己处理"删除"按钮 + 自己 refetch——父不知道发生了什么，CustomerCard 显示的还是删除前的客户数。
经验：事件冒泡到"拥有 data 的那一层"，由它决定下一步（refetch / 跳转 / 弹窗）。子组件只负责"告诉爸爸"。
BAD
// Table 自己 refetch
<Table>
  onDelete = () =>
    api.del(); refetch()
</Table>
GOOD
// 父决定下一步
<Table onDelete=
  {(id) => {
    api.del(id);
    page.refetch()
  }}/>
4 状态归属state ownership

哪些状态归子（私有），哪些归父（共享）？

反例：currentPage 放 Pagination 内部——切 Tab 后又回第 1 页（因为 Pagination 被 unmount 了状态丢了）；想从 URL 同步分页也做不到。
经验：跨 tab 共享的状态归 Tab；跨 Tab 不共享的归 Page。"谁要看见这个状态？state 放在能覆盖所有看见的最低祖先。"
BAD
// page 归 Pagination
<Pagination>
  useState('page', 1)
  // unmount = 丢
</Pagination>
GOOD
// page 归 Tab
<Tab>
  useState('page', 1)
  <Pagination
    value={page}/>
</Tab>
4 件事每件都是设计决策。
决定了组件能不能复用、能不能维护、改一处会不会牵连十处。

04 / 看更多

「订单管理」页——
看人家拆的组件树。

换个场景：订单管理。8 个组件——⚠ 标的几行是设计师最容易交叉错的地方。

OrderListPage · 组件树 + 来源 + 复用度
8 nodes · 3 levels
组件 来源 复用度 为什么这样放 / 何时该自定义
<PageLayout /> shadcn 高 · 全站共用 页面骨架——header + sidebar + 主内容区。所有列表页共用，永远不要在这里加业务字段。
<ToolBar /> shadcn + slot 高 顶部操作条 = 标题 + 批量按钮 + 导出。批量按钮塞 slot 进去，别 fork ToolBar 写死。
<FilterBar /> 自定义 中 · 列表页共用 独立组件，不嵌在 Table 里——以后想"一套 Filter 控制多个 Table"、或"切到地图视图共用 Filter"都不用重写。
<OrderTable /> shadcn DataTable 中 · 表格变种 列：单号 / 客户 / 金额 / 状态 + 行内 <ActionMenu/>。基础 DataTable 不动，列定义和行操作通过 props 传。
<ActionMenu /> shadcn DropdownMenu 高 · 全站 行内"⋯"。注意：菜单项由父传，ActionMenu 自己不知道业务——同一个 ActionMenu 在订单页、客户页用法不同，但组件本身不 fork。
<Pagination /> shadcn 高 · 全站 页码 + 每页条数。state 归 Page，不归 Pagination——切 Filter 重置页码靠的是 Page 同步管两个。
<DetailDrawer /> shadcn Sheet 中 · 列表页共用 点行展开。放 Page 层级，不要嵌进 OrderTable——选中状态归 Page，否则切页 / 切 Filter 抽屉就废了。
<EmptyState /> shadcn 高 · 全站 列表为空时显示。嵌在 OrderTable 内——它是 Table 的内部状态，不是 Page 的。
⚠ FilterBar 不要嵌进 Table 很多稿子画成 <Table> <Filter/> <Rows/> </Table>——一旦想"一个 Filter 控制 Table + Map"就废了。独立放，平级用。
⚠ ActionMenu 不要写死业务 同一个"⋯"在订单页是"取消 / 退款 / 备注"，在客户页是"加标签 / 转移"。菜单项作为 prop 传进去——别为每页 fork 一个 OrderActionMenu。
⚠ DetailDrawer 不要嵌进 Table 抽屉是 Page 的状态——它显示"被选中的那一行"。嵌进 Table 后，切页或切 Filter，Table 重渲染就把抽屉 unmount 了。
⚠ Pagination 的 state 归 Page 稿子上 Pagination 在 Table 下面，很多人以为页码 state 也在 Table 内。其实"第几页"要跟 Filter 一起重置——归 Page 才对。
05 / 把这件事变成习惯

下次画稿子，
右下角先画一棵树。

从这一篇开始，"整页一张图"和"组件树 + 流向"的差距会出现在你每一次交付的稿子里。

i

稿子右下角画组件树

即使粗略，也比没强。工程师能立即知道嵌套关系——哪些是组件，哪些是 div，哪个组件包哪个。

纸上画 30 秒，开发省 30 分钟。

ii

state 归属 = 共享范围

谁要看见这个状态？state 放在能覆盖所有看见的最低祖先。

不共享就放子内部；共享一个区域就放该区域父；全局共享就放最外层（或 store）。

iii

复用组件先问"能不能独立"

Filter 能不能脱离 Table 单独用？能 = 平级；不能 = 应该嵌套。

这一问能避免 80% 的"为什么这个组件改一处坏一处"。

划重点 · TL;DR

页面是组件的舞台。组件树画清楚 + 流向标清楚 = 工程师 0 追问。

下次需求评审，试一次——画完页面布局后顺手在右下角画一棵 5 行的组件树，标 2 条数据流、1 条事件流。这一动作会让对面工程师的脸由皱变松。
