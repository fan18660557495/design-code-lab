01 / 问题

为什么"一张大稿子"
工程师无从下口？

设计师把"复杂页面"当一个整体去画。其实它是组件组合——已有组件嵌套 + 数据从某层灌进去 + 事件冒泡到某层。 组合规则决定了页面好不好维护。

整页画稿 · BAD

"稿子是一张大图"

所有内容堆在一起。Card、Table、Filter、Pagination、ActionBar 平铺在画板上——看起来像一个完整的产品页面。

工程师追问：Card 是组件吗？还是 div？Filter 改变时 Table 自动 refetch 吗？分页放 Table 里还是外面？切 Tab 后筛选器要不要清空？

你：呃……要再画一份组件嵌套关系？

组件树 · GOOD

"稿子右侧标注一棵组件树"

稿子右侧画一棵树：<Page><Header/><Filter onChange=>/><Table data=/><Pagination/></Page>，旁边箭头标 data 流向、event 冒泡方向。

工程师收到：组件树 + 数据流图。每个组件的 props、events、状态归属一目了然。

结果：0 追问，直接开搭。

幕二 · 拆开
03 / 拆开

组合 4 件事。

组件组合时要回答 4 件事——容纳关系 / 数据下行 / 事件上行 / 状态归属。 点每张卡片展开看坏 vs 好的代码对比。

1 容纳关系contain

谁是父，谁是子？哪些组件该嵌套，哪些该平级？

反例：把 Filter 放 Table 内部——以后想做"一个 Filter 控制多个 Table"就废了。Filter 和 Table 被绑死，复用性归零。
经验：可独立使用的组件分开放（Filter 独立于 Table，便于多 Table 共用一个 Filter，或者 Filter 独立用于地图视图）。
BAD
// Filter 嵌死在 Table 内
<Table>
  <Filter />
  <Rows />
</Table>
GOOD
// 平级，独立复用
<FilterBar />
<Table />
<MapView />
2 数据下行data flow down

数据从哪一层加载？再往下传到哪？

反例：Card / Table / Timeline 每个子组件自己 fetch——一个页面打 N 个请求（N+1 问题），网络瀑布、数据不一致。
经验：找最近的"共同父"——它持有数据，往下传 props。子组件保持"哑组件"（只渲染，不取数）。
BAD
<Page>
  <Card/>  // fetch
  <Table/> // fetch
  <Tl/>    // fetch
</Page>
GOOD
<Page> // 1 次 fetch
  <Card data={c}/>
  <Table data={opps}/>
  <Tl data={acts}/>
</Page>
3 事件上行events bubble up

子组件的事件通知哪一层？谁负责"做出反应"？

反例：Table 自己处理"删除"按钮 + 自己 refetch——父不知道发生了什么，CustomerCard 显示的还是删除前的客户数。
经验：事件冒泡到"拥有 data 的那一层"，由它决定下一步（refetch / 跳转 / 弹窗）。子组件只负责"告诉爸爸"。
BAD
// Table 自己 refetch
<Table>
  onDelete = () =>
    api.del(); refetch()
</Table>
GOOD
// 父决定下一步
<Table onDelete=
  {(id) => {
    api.del(id);
    page.refetch()
  }}/>
4 状态归属state ownership

哪些状态归子（私有），哪些归父（共享）？

反例：currentPage 放 Pagination 内部——切 Tab 后又回第 1 页（因为 Pagination 被 unmount 了状态丢了）；想从 URL 同步分页也做不到。
经验：跨 tab 共享的状态归 Tab；跨 Tab 不共享的归 Page。"谁要看见这个状态？state 放在能覆盖所有看见的最低祖先。"
BAD
// page 归 Pagination
<Pagination>
  useState('page', 1)
  // unmount = 丢
</Pagination>
GOOD
// page 归 Tab
<Tab>
  useState('page', 1)
  <Pagination
    value={page}/>
</Tab>
4 件事每件都是设计决策。
决定了组件能不能复用、能不能维护、改一处会不会牵连十处。

05 / 把这件事变成习惯

下次画稿子，
右下角先画一棵树。

从这一篇开始，"整页一张图"和"组件树 + 流向"的差距会出现在你每一次交付的稿子里。

i

稿子右下角画组件树

即使粗略，也比没强。工程师能立即知道嵌套关系——哪些是组件，哪些是 div，哪个组件包哪个。

纸上画 30 秒，开发省 30 分钟。

ii

state 归属 = 共享范围

谁要看见这个状态？state 放在能覆盖所有看见的最低祖先。

不共享就放子内部；共享一个区域就放该区域父；全局共享就放最外层（或 store）。

iii

复用组件先问"能不能独立"

Filter 能不能脱离 Table 单独用？能 = 平级；不能 = 应该嵌套。

这一问能避免 80% 的"为什么这个组件改一处坏一处"。

划重点 · TL;DR

页面是组件的舞台。组件树画清楚 + 流向标清楚 = 工程师 0 追问。

下次需求评审，试一次——画完页面布局后顺手在右下角画一棵 5 行的组件树，标 2 条数据流、1 条事件流。这一动作会让对面工程师的脸由皱变松。
