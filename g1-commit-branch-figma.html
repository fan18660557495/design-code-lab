Git 的 branch
不是 Figma 的那种。

你在 Figma 里点过 branch —— 那是"另存一份再合回来"。
git 的 branch 完全不是同一回事 —— 它是一条平行的时间线。
这一篇把三件事一次拆开：commit / branch / merge，读完你看版本树不再发懵。

01 / 快照

commit
是每一次保存的快照

每次 git commit，git 给你当前所有文件拍一张照，存起来 —— 附带一段你写的"说明文字"。一条线穿起来，就是项目的历史。

互动下面这条时间线上每一个圆点都是一次 commit。把鼠标放上去看完整信息 —— hash、时间、改了哪些文件。

FIGMA · 类比
点 "Add to history" 主动留版本

在 Figma 里你可以点"添加到版本历史"留一个有名字的快照。

≈
GIT · 但有个关键差别
必须主动 commit，git 不替你保存

Figma 会自动保存历史，git 不会 —— 你不 commit，就什么都没存。

关键commit 不是"备份" —— 它是有名字、有作者、有时间、有改动清单的一次正式留痕。 所以好的 commit message 很重要 —— §04 专门讲。
02 / 分叉

branch
是一条平行的时间线

从主线某一个 commit 上"分叉"出去，独立往前走 —— 互不影响。 这不是"备份"，也不是"另存一份" —— 是真的多了一条时间线。

视觉下面这张图：黑色是 main 主干，橙色是新功能分支 feature/login，蓝色是紧急修 bug 分支 hotfix/btn。三条线同时往前走，互不干扰。
MAIN · 主干
永远是"能上线的"

主线只接受合并过来的稳定代码，不直接改。

FEATURE/LOGIN · 新功能
还在做，没合回

分支上的 c4 / c5 / c7 完全不影响 main。

HOTFIX/BTN · 修 bug
改完合回，分支就完成使命

在 c8 处合并回 main —— 这就是 merge。

FIGMA · BRANCH

是"另存为一份草稿"

本质上是把整个文件复制了一份，让你在副本里改、不影响主文件。改完了再"合回主文件"。

  • 每个 branch 是一份独立的文件副本
  • 主要解决"多人改同一个文件"
  • 合并 = 把整份草稿覆盖回去
GIT · BRANCH

是"分叉一条时间线"

不复制文件，只是新加一个指针，标记"从这个 commit 开始往这边走"。所有 commit 共享同一个仓库。

  • 每个 branch 是一个commit 指针
  • 解决的是"同一个项目并行做几件事"
  • 合并 = 把两条时间线的改动汇成一条
记住"在 git 的 branch 里改了"不等于"在 main 里改了"。你的改动关在自己分支里，必须 merge 回 main，main 才会变。

04 / 留言

commit message
怎么写不被嫌弃

设计师以为这是"随便写写"的字段，工程师每次 commit 都看得见 —— 写得清楚的 message，是项目 5 年后还能读懂的关键。

看对照左边是好的 commit message，右边是反例。把鼠标放上去对比。
好的 · GOOD MESSAGES

短而清楚 · 一眼能懂

  • ✓ fix: 修复登录按钮在 iPhone SE 上溢出
  • ✓ add: 新增订单详情页 skeleton 状态
  • ✓ update: 把主品牌色从 #ff5e1a 调到 #ff4d2e
  • ✓ refactor: 抽离 Card 组件的 hover 状态逻辑
  • ✓ docs: 补 README 里的本地启动步骤
坏的 · BAD MESSAGES

三个月后看不懂

  • ✕ 改了一下 改了哪？
  • ✕ 更新文件 哪个文件？
  • ✕ 1234 ？？？
  • ✕ 按 PM 要求改 改了什么？
  • ✕ . 真的有人这样写
RULE · 01
动词开头

fix / add / update / refactor / remove / docs —— 一眼看出做了哪种改动。

RULE · 02
讲"做了什么"

不是"为什么做"。why 在 PR 描述里写，commit 只讲事实：动了哪个组件、哪个文件、改成了什么。

RULE · 03
一句话能讲完

50 字以内最舒服。讲不完？那这一次 commit 可能装了太多事 —— 拆成两次。

小贴士很多团队用 type: 描述 这种格式（叫 conventional commits）—— 不是必须的，但写起来思路清晰、git log 也好读。
05 / 上手

设计师最该掌握的
5 个 git 命令

不是要你成为 git 专家。这 5 个命令能覆盖 80% 的设计师日常 —— 知道什么场景用哪个，遇到事不慌。

CMD · 01

看现在啥情况

git status

什么时候用：不知道当前在哪个分支、改了哪些文件、还没 commit 的有多少 —— 这一条全告诉你。开干前先打这一条，永远不会错。

输出长这样 On branch feature/login
Changes not staged for commit:
  modified: Button.tsx
  modified: styles.css
CMD · 02

保存这次的改动

git add .
git commit -m "fix: 修复按钮溢出"

什么时候用：改完一件事 —— 比如"按钮颜色调好了" —— 用这两条把它存为一次 commit。add 是把改动放进暂存区，commit 是真正留快照。

输出长这样 [feature/login a3f8b21] fix: 修复按钮溢出
 1 file changed, 4 insertions(+), 2 deletions(-)
CMD · 03

推到远程

git push

什么时候用：commit 之后只是存在你电脑上。push 一下才会同步到 GitHub / GitLab，同事才能看到。commit 是本地、push 是上传。

输出长这样 To github.com:team/project.git
  a1b2c3d..a3f8b21  feature/login -> feature/login
CMD · 04

同步别人改的

git pull

什么时候用：每天开干之前。把同事昨天 push 的改动拉到你的本地 —— 这样你后面动的就是最新版本，少很多冲突。

输出长这样 Updating a3f8b21..7c1d9e2
Fast-forward
  Header.tsx | 8 +++++++-
  1 file changed, 7 insertions(+)
CMD · 05

开一条新分支

git checkout -b feature/login-redesign   # 创建并切到新分支

什么时候用：开始做一个新功能 / 试一个新方案前。永远不要直接改 main —— 先开一条分支，在分支上随便折腾，做完了再合回 main。 -b 的意思是 "branch"（创建一条新分支），分支名一般叫 feature/做的事 或 fix/修的 bug。

输出长这样 Switched to a new branch 'feature/login-redesign'
够用记住这 5 条 —— status / add+commit / push / pull / checkout -b —— 你已经能跟工程师正常协作了。其他命令（rebase、cherry-pick、reset）等你真的撞上再学。

CARD 01 · 心智

Figma vs Git

同一个词，意思不一样
version
历史
Figma 自动存。
git 必须 commit。
branch
分支
Figma 是"另存草稿"。
git 是 分叉的时间线。
merge
合并
Figma 是"草稿覆盖回去"。
git 是 两条时间线汇合。
conflict
冲突
Figma 直接覆盖。
git 让你 逐行选（→ 见 G2）。
