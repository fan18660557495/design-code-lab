状态机的
3 件事。

节点（状态）+ 边（迁移）+ 守卫（条件）—— 画完这 3 件，状态机就完整。点图里的节点和边看属性。

点 节点 看该状态的 UI 摘要 · 点 边 看迁移属性（触发方 / 条件 / 副作用 / 反方向）。

点节点或边查看属性

状态有 UI 形态
迁移有触发条件
1

状态被遗漏

例：忘了"待付款超时自动关闭"——订单永远卡在 submitted。

3 个月后客服收到一堆"为什么还能付款"的工单。

2

边被遗漏

例：忘了"已付款也能退款"——用户点进订单只看到"申请售后"灰色禁用。

客服只能手动改数据库。

3

守卫被遗漏

例："标记发货"按钮没检查库存——运营点了就发不出去，最后还得回退状态。

状态机最隐蔽的坑，全靠条件兜底。

状态机不画清楚，每个 PM / 前端 / 后端理解都不一样——产品就裂了。
画清楚 = 大家在看同一张图说话。

迁移 触发方 守卫条件 (Guard) 副作用
submit员工必填项齐 + 余额够发邮件给主管 + 锁定假期余额
approve主管员工身份有效发邮件给员工 + 创建日历事件
reject主管必填驳回理由发邮件给员工 + 解锁余额
withdraw员工必须在主管未审批前解锁余额 + 通知主管
cancel员工 / 主管批准后 + 假期未开始解锁余额 + 删日历事件 + 通知双方
start (auto)系统 cron到达开始日期状态变化 + 邮件提醒 + 同步到 HR 系统
finish (auto)系统 cron到达结束日期归档 + 同步考勤记录
resubmit员工已 rejected带回原数据，版本号 +1，回 drafting
⚠ 3 个最容易漏的：
① 没有 "saved" 状态——保存是动作不是状态，对应的状态是 drafting。
② withdraw 的守卫 容易漏——审批后撤回 ≠ 撤回，那是 cancel。
③ cancel 的副作用 容易写"解锁余额" 就结束——但日历事件不删，HR 同步不撤，几周后看到"鬼假期"。

05 / 把这件事变成习惯

拿到新需求，
先画状态机。

从这一篇开始，"对象有什么字段"会让位给"对象的一生经过哪些合法状态"。 UI 是从图里推导出来的——不是设计师拍脑袋画的。

i

每个对象画一张状态机

状态机是对象的"时间轴 schema"。不画 = 不知道哪个时刻显示什么。

主对象先画，子对象（评论 / 附件 / 任务）也各画各的。

ii

没有"草稿"和"保存"

"保存"是动作，对应 drafting 状态；"草稿"是状态，不是动作。

区分动词（迁移）和名词（状态）——一字之差，状态机就清楚了。

iii

守卫 = 隐形 UI 约束

某条迁移有条件——意味着相关 UI 按钮要带禁用态 + 解释文案。

守卫不画 = 用户按了按钮然后被拒绝 = 客服工单。

划重点 · TL;DR

拿到新需求第一件事：写出主对象的状态名 + 状态之间的合法迁移。其余都从这张图派生——UI 形态、按钮可见性、权限规则、自动化时机。

下次需求评审，试一次——第一句不是"我打算这么画"，而是"这个对象有几个状态，每两个状态之间是怎么跳的"。会议的语言就变了。
