02.5 / 想清楚

"5 种标签"其实是
2 个维度的合并。

local / api / push / 3rd / build 是经验切片——好记、好上手、能直接标在稿子上。 但底下其实是 位置 × 同步 两个正交维度的交点。看一眼 2D 图，下面 4 张诊断 card 就更明白。

"API"和"PUSH"都是从你家服务端来的——区别在同步方式（一个拉、一个推）。 "LOCAL"和"BUILD"都不打接口——区别在什么时候定下来（运行时存的 vs 编译时打包的）。 把这两件事拆开看，5 个标签是 5 个交点，不是 5 个分类。

维度 1 · 纵轴
位置 · WHERE

这个值的真理 (truth) 在谁家？决定了：失败时谁负责兜底、改字段要找谁、合规风险归谁。

维度 2 · 横轴
同步 · HOW

值怎么到达前端？决定了：什么时候刷、loading 怎么画、stale 怎么标、断线怎么办。

位置 ↓ / 同步 → pull · 主动拉 push · 推送 static · 一次性 computed · 派生
浏览器本地 读 localStorage storage event LOCAL ✓ 本地派生
自家服务端 API ✓ PUSH ✓ — 服务端派生
第三方服务端 3RD ✓ 少见 — —
构建产物 — — BUILD ✓ —
怎么用：下面 4 张诊断 card 用 5 个标签教你"在稿子上标个色"——这是入门动作，记 5 个色就行。
深一层：当你想清楚为什么 API 和 PUSH 的 "truth owner" 一样但"失败模式"完全不同——是因为它们位置同、同步异。所有的 stale 行为、fallback 策略、降级方案都从这个 2D 图来。

03 / 拆开

4 种来源的
诊断特征。

每种来源都有 4 个维度：truth owner / 新鲜度 / 失败模式 / 缓存策略。 点卡片底部"展开"看"如果设计稿不标记来源——最坏情况长这样"。

01 LOCAL · 本地
浏览器自己存的 — localStorage / sessionStorage / cookie
Truth owner
客户端浏览器
新鲜度
永久 / 直到用户清缓存
失败模式
不存在 → fallback 默认值
缓存策略
localStorage 永久 · sessionStorage 会话 · cookie 按需
展开 · 最坏情况 →
如果设计稿没标 · WORST CASE 工程师每次都打接口 → 性能死。
用户每打开一次页面 → 10 次完全可以走 LOCAL 的请求被推到后端。
02 API · 接口拉取
前端主动调一次 — GET /api/xxx
Truth owner
服务端 DB
新鲜度
拉取那一刻的快照
失败模式
超时 / 5xx → 错误态 + 重试
缓存策略
客户端 + SWR · CDN · no-cache（按数据）
展开 · 最坏情况 →
如果设计稿没标 · WORST CASE 后端不知道什么时候拉 → 数据可能过期。
用户看到一份 30 分钟前的"库存"信息，下单时才发现没货。
03 SERVER-PUSH
服务端主动推 — WebSocket / SSE / 实时推送
Truth owner
服务端 + 事件总线
新鲜度
实时（毫秒级）
失败模式
断开 → 自动重连 + 期间状态 stale
缓存策略
最后一次推送的值
展开 · 最坏情况 →
如果设计稿没标 · WORST CASE 工程师写成 polling（每 N 秒拉一次） → 延迟差。
"实时聊天"变成"5 秒后才看到对方的消息"——体感像断网。
04 3RD-PARTY · 第三方
别人的接口 — 天气 / 汇率 / 地图 / 邮件
Truth owner
别人的系统
新鲜度
看他们的承诺
失败模式
服务降级（不可控）→ 必须有 fallback
缓存策略
你自己做（不要每次都打他们）
展开 · 最坏情况 →
如果设计稿没标 · WORST CASE 工程师没做 fallback → 第三方挂全站挂。
和风天气接口挂 → 你的 dashboard 整块白屏 → CEO 看到截图：是不是我们出事了？
标记来源 = 给工程师一份 "数据 schema" —— 避免猜。
猜不一定对，标记一定对。

05 / 把这件事变成习惯

下次画稿子，
每个字段角落标个色。

绿 / 蓝 / 紫 / 黄。从这一篇开始，"字段在哪生"和"字段挂了显示啥"会出现在你每一次评审里。

i

每个字段标来源

设计稿直接标。LOCAL / API / PUSH / 3RD-PARTY 一目了然，工程师不用追问。

颜色编码比文字描述快 10 倍——绿色的字段不用打接口，蓝色的可以缓存。

ii

fallback 比来源更重要

"拿不到数据时显示啥"是设计决策——不是工程师的锅。

错误态 / 上次的值 / 默认值 / 不显示——4 种语义完全不同。是要让用户看到错误，还是悄悄降级，看产品需求。

iii

第三方必须有降级

别人的服务挂了你也要能活。fallback 不是 nice-to-have——是 must。

"天气挂了不显示这一块"和"天气挂了整页白屏"差距在你的设计稿。

划重点 · TL;DR

下次画稿子，每个字段角落标个色——绿 / 蓝 / 紫 / 黄。绿是本地、蓝是接口、紫是推送、黄是第三方。

工程师看一眼就知道：这块要不要打接口、要不要写订阅、要不要做 fallback。你少答 10 个问题，他少踩 10 个坑。
