03 / 拆开

规则说明书
解剖。

一份完整的说明书包含 5 个标准模块。每个模块都有 checklist—— 每张卡片展开看里面写什么、勾选"本模块就绪"。

1
描述
OVERVIEW
包含内容
  • 这是个什么组件 / 页面 / 功能
  • 目标用户 + 主要使用场景
  • 业务价值（为啥要做）
示例
  • 「确认删除 Modal」用在删除高风险数据前的二次确认
  • 用户 = 普通操作员；场景 = 列表勾选 → 删除 → 弹出
本模块就绪
2
视觉
VISUALS
包含内容
  • Figma 链接 + 标注（spacing / color / font / 8px 网格）
  • 所有状态截图：default / hover / active / disabled / loading / error / empty
  • 响应式 breakpoints（mobile / tablet / desktop）
常被漏
  • loading spinner 大小 / disabled 色值 / focus 描边
本模块就绪
3
行为
BEHAVIOR
包含内容
  • 交互流程（状态机 / 用户事件 → 系统反馈）
  • 异常处理：超时 / 冲突 / 双击 / 权限不足
  • 副作用：发邮件 / 写日志 / 刷新缓存 / 跳页面
示例
  • draft → submitting → success | fail
  • 双击提交按钮 = 仅触发一次
本模块就绪
4
数据
DATA
包含内容
  • 字段 schema：name · type · required · default · validation · computed · source
  • API 契约：endpoint / request / response / error codes
  • 状态住址：URL 参数 / 本页 state / 服务端 / 浏览器缓存
常被漏
  • 空值显示什么 / 错误返回什么 code / 字段长度上限
本模块就绪
5
验收
ACCEPTANCE
包含内容
  • 用户场景：Given / When / Then 三段式
  • 边界场景：empty / max / unauthorized / network fail
  • 性能要求：首屏 < N 秒 / 交互响应 < N ms
作用
  • QA 拿到这层 → 直接写测试用例
  • PM 拿到这层 → 直接照单验收
本模块就绪

06

陷阱 · 设计师最常漏的

anti-pattern
⚠ 陷阱 1只给 Figma 链接 = 上线 1 周后被工程师追问 50 个问题。视觉稿只覆盖 default 态 + 主路径。状态 / 权限 / 异常 / 跳转 / 键盘焦点没说 —— 工程师只能拍。
⚠ 陷阱 2"empty / error 让工程师自己处理"。空态和报错态是设计的事。空态插图 + 文案 + CTA 必须画。 错误态文案 + 重试 + 兜底也必须画。 漏了就是产品上线后用户看到"undefined"。
⚠ 陷阱 3权限规则口头说。"客服看不到手机" 这种规则要写在 spec 里，不是开会口头说。 半年后没人记得，新来的工程师就把手机号显示全了 —— 等于泄漏。
⚠ 陷阱 4键盘焦点 / Esc 行为不写。设计师以为"只有键盘党才用"——其实所有人都会按 Esc 关弹窗、按 Tab 在 form 里跳。 不写 = 工程师按经验做 = 每个表单跳一套，无人能预测。
⚠ 陷阱 5跳转回程不写。"点公司 → 公司详情"很好 —— 但用户从公司详情怎么回客户详情？面包屑？返回按钮？保留滚动位置吗？不写就是工程师按"浏览器后退"做，体验不连贯。
·· 最后一句 ·· Figma 链接 + 这 6 张表 + 5 个陷阱清单 = 1 次交付到位。
工程师 1 次写完代码 / QA 1 次写完测试 / PM 1 次验收完。不再有"半夜叫醒补稿子"。

05 / 把这件事变成习惯

下次交付，
不发稿子链接。

从这一篇开始，把"发 Figma" 换成"过一遍 4 层 + 5 模块"。 工程师 / QA 会感谢你——返工 0 次的稿，第一次出现。

i

稿子下面附 4 段

视觉 · 交互 · 数据 · 验收。哪怕一句话——也比没强。

4 段都附，工程师 0 追问。

ii

验收先写 Given/When/Then

不写场景的设计 = 不可测。可测 = 可验收。

先写场景，再画稿——稿子反而更准。

iii

交付不是 send Figma

把稿子发出去后，跟工程师过一遍每个模块。对不上的地方现场补——比事后撕扯快 10 倍。

是确认对齐，不是丢链接。

划重点 · TL;DR

稿子是 视觉 × 一个组件 的产物——只占交付的 1/4。剩下的 3/4（交互、数据、验收）不写，就只能由工程师猜、QA 漏、PM 撕。

下次交付，附一张 5 项的 checklist——勾完才发。工程师 / QA 都会感谢你。
