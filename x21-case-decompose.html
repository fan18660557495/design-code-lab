03 / 拆开

4 张图，
每张的画法。

每张图都有自己的"最小要素"。会画这 4 张 = 你不只是设计师，是架构师。 点卡片展开看每张图的画法 + 缺这张时工程师怎么搞。

1

对象图

Object Model

数据维度——这个系统有哪些对象、谁引用谁、谁是 1:N。

最小要素
节点（对象名）+ 边（cardinality）+ 字段（核心 5–10 个）
工具
SVG · Mermaid · dbdiagram.io · Whimsical
用途
工程师拿来直接落 DB schema · 后端建表 · 写接口
展开看缺它时怎么搞 →
如果缺这张：后端自己看稿瞎猜——"详情页上写了'合同 (2)'，所以合同应该是 customer 的子表？还是另存？联系人和合同共用一个表吗？"
结果：DB schema 改 3 版才稳定，前期写的接口全得重写。
2

任务流

Task Flow

行为维度——用户要做的几件事，每件事的步骤 / 分支 / 异常。

最小要素
步骤 + 用户/系统行动 + 分支 + 异常 + 状态迁移注解
工具
Figma · Whimsical · Mermaid · FigJam
用途
工程师拿来设计路由+ 写测试 + 产品讨论 · QA 拿来写用例
展开看缺它时怎么搞 →
如果缺这张：前端只能看稿上有几个按钮反推路径——"转分配"按钮后面接什么？需要确认弹窗吗？接收方接受前 owner 字段是不是已经变了？
结果：关键路径漏分支、漏异常态，上线后才发现一堆 corner case，回炉。
3

状态机

State Machine

时间维度——对象有几个状态、什么 trigger 让它跳转。

最小要素
状态节点 + 边 + 触发条件 + 副作用（写入 / 通知 / UI 变）
工具
XState · Mermaid · 手画拍照（足够好）
用途
工程师拿来写 enum + transition logic + UI 显隐规则
展开看缺它时怎么搞 →
如果缺这张：前端定义 enum 时凭感觉——dormant 是不是和 inactive 一回事？lost 之后还能不能重新激活？标签颜色用什么？
结果：不同模块对同一状态名理解不一致，列表筛选、详情页徽章、邮件通知里的状态都对不上。
4

组件树

Component Tree

代码结构维度——稿上每一块对应哪个组件、嵌套层级、数据流方向。

最小要素
节点（组件名）+ 嵌套关系 + 数据/事件流箭头
工具
手画 · Figma frame · Storybook hierarchy
用途
工程师拿来定文件结构+ props 接口 + state 归属
展开看缺它时怎么搞 →
如果缺这张：前端自己拆——是把 ActionBar 放在 Header 里还是和 Card 平级？state 放在 CustomerDetail 还是 Tabs 内？
结果：state 放错层级 → 切 tab 数据清空、转分配后页面不刷新、组件复用率为 0，2 周后整个详情页要重构。
4 张图 = 4 份独立合同。
任何一份能脱离设计稿被工程师读懂——这才是"架构图"的标准。

04 / 看更多

4 张图，
每张都画完整。

4 张图各 1 张演示——左边是图，右边是为什么这样画。每张配 ⚠ 陷阱卡。

A · 对象图

DIAGRAM 1 / 4
Customer + 4 邻居 + cardinality
为什么这样画：客户居中，4 个最常用邻居挂上。Contact 是 M:N（一个联系人可能在多公司），其余 1:N。每条边标 cardinality——工程师据此建外键 / 关联表。
⚠ 陷阱：把 Project 误挂上去——这是项目管理工具的概念，CRM 客户没有 Project 邻居。多画一个对象 = 多一张表 + 多一组接口 = 工期翻倍。

B · 状态机

DIAGRAM 2 / 4
Active / Inactive / Archived · 3 条边
为什么这样画：每条边写 trigger + 副作用——active→inactive 是系统自动（cron job 扫 last_contact）；inactive→active 是任何 activity 触发；active→archived 必须用户手动操作。
⚠ 陷阱：设计师只画"active / inactive / archived"3 个节点 + 箭头，不写 trigger → 工程师只能凭直觉写——一半边写错，dormant 客户永远跳不回 active。每条边都必须标 trigger。

C · 角色权限矩阵

DIAGRAM 3 / 4
4 角色 × 5 维度
维度 \ 角色销售财务客服客户本人
字段·金额✓✓✗✗
字段·电话完整脱敏脱敏完整
操作·编辑✓✗备注改密
状态·删除✗✗✗✗
跳转·新建商机✓✗✗✗
为什么这样画：矩阵 = 每格一条权限规则，工程师按格写。脱敏（m）必须发生在后端而不是前端。"删除"只有 admin 能做——不在这 4 角色里。
⚠ 陷阱：用 if (role === 'admin') { ... } 一句话判断权限——加新角色（例如"区域经理"）时全站漏改。权限永远是矩阵，不是 if 链。

DIAGRAM 4 / 4
CustomerDetailPage · 3 层嵌套

为什么这样画：组件树 = 嵌套 + 数据流。父组件传 props ↓ 子组件 emit events ↑。InfoCard 和 RelatedSections 平级（不嵌套），因为它们在视觉上是上下两块。每个 Tab 对应一种关系（呼应对象图 A）。
⚠ 陷阱：把 4 个 Tab 都嵌进 InfoCard——视觉上看似紧凑，但 Tab 切换会导致整个 InfoCard 重渲染。组件树画错 = 性能 / 状态管理后患。

05 / 把这件事变成习惯

下次交付前，
先画 4 张图。

画不出 = 设计还没想完。画出来 = 你已经在做架构师做的事。

i

稿 + 4 张图 = 最终交付

不要再发"Figma 链接 + 标注"——加上对象图、任务流、状态机、组件树。

工程团队体感天差地别。

ii

工具不重要，画出来就行

Mermaid 文本、Whimsical 拖拽、Figma frame、甚至手画拍照——只要工程师能读懂就够了。

不要为了画图本身陷进工具选择。

iii

画的过程就是发现盲点

画对象图发现少了一个关系；画状态机发现忘了"流失重激活"；画组件树发现 state 放错了层。

画的过程，就是 review 的过程。

划重点 · TL;DR

一张设计稿背后藏着 4 张架构图。设计师只画稿——工程师必须自己重画 4 张，每个工程师补的版本都不一样。

设计师如果直接画好 4 张 + 稿——工程师拿到当合同直接落。这 4 张图就是设计师 → 架构师的实际产出。

下次设计交付前，先自己画 4 张图。画不出 = 设计还没想完。
