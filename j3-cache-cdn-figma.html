01 / 缓存为什么存在

缓存
是好事。

在骂它之前，先承认一件事 —— 没有缓存，整个网络都会慢成你不想用。

NO CACHE · 没有缓存的世界

每一次刷新 = 全量下载

你打开任何一个网站，浏览器都得把所有图片、字体、JS、CSS 全部从服务器拖一遍，每次都从头来。

第 1 次访问 → 下载 3 MB 资源
第 2 次访问 → 又下载 3 MB
第 3 次访问 → 还在下载 3 MB (无情)
WITH CACHE · 有缓存的世界

第二次起 = 直接从就近的"小卖部"取

资源被存到了你电脑里 / 离你最近的 CDN 节点 / 服务端的内存。下一次只问"还是那个吗"，就秒返回。

第 1 次 → 完整下载 3 MB
第 2 次 → 0 KB · 直接毫秒级命中
第 N 次 → 仍然 0 KB · 网页起飞

便利店 vs 超市 —— 类比

把"取一个文件"想成"买一瓶水"。同一瓶水，跑超市远，便利店近。缓存就是你家楼下的便利店。但便利店货可能是上周的 —— 这就是缓存过期 / 过旧。

从你到数据库

挑战 3 题 · CHALLENGE

玩完了来对几道

这 3 道题就是设计师上线时最常遇到的 3 个坑。如果都能答对 —— 这一篇可以收工了。

Q1你在 GitHub push 了改完颜色的新 CSS，但访问页面还是旧的。第一步该怎么办？

对了。第一反应永远是 Cmd+Shift+R (Mac) / Ctrl+Shift+R (Win)。这一步 30 秒，能解决八成 "改了没生效"。如果不奏效，再看下一题。

Q2你 hard refresh 完，还是旧的。这说明问题在哪？

对。hard refresh 只清了你这一台电脑的 L0。CDN 那一份还在缓存有效期里，根本不会去问源。这时候要么去 Vercel / Cloudflare 面板手动 purge cache，要么等它过期 (默认几小时到几天)。

Q3下一次发版，怎么从机制上保证所有用户都立刻看到新版？

对。这是行业默认做法 —— Next / Vite / webpack 都内置了。改完颜色 → 文件名变 → 浏览器看到的是新文件名，所有缓存都不认，自然回源拿新的。
C 选项看着对，但会让网站慢得像 1995 年 —— 缓存的好处全没了。
04 / 设计师的 3 把武器

3 个清缓存操作
绝大多数能解决。

排查顺序：先清自己的 (L0)，不行再让平台清 (L1)，再不行才是整个流程出错了。

01

Hard Refresh

跳过 L0 浏览器缓存，逼浏览器至少去问一次 CDN。每天多用 — 设计师上线后第一反射动作。

MacCmd + Shift + R
Win / LinuxCtrl + Shift + R
SafariCmd + Option + E → Cmd + R
› 关键键位
Cmd+Shift+R
02

平台 Purge

登 Vercel / Cloudflare / Netlify 后台，找 Cache 或 Purge 按钮，点一下 — L1 整个边缘节点的缓存被强制丢弃。

VercelProject → Settings → Data Cache → Purge Everything
CloudflareCaching → Configuration → Purge Cache
NetlifyBuilds → Trigger deploy → Clear cache
› 谁能点
有平台账号 权限 的人 — 通常是工程师 / 团队 admin。
03

无痕窗口

无痕 / 隐身模式不会用任何旧缓存。3 秒切一个，看到的就是真用户视角。验证页面用它最稳。

ChromeCmd / Ctrl + Shift + N
SafariCmd + Shift + N
FirefoxCmd / Ctrl + Shift + P
› 关键键位
Cmd+Shift+N
还有一个 (技巧) DevTools 打开时，可以勾上 Network → Disable cache。只要这个 DevTools 窗口开着，浏览器一律不读 L0。设计师做大改的时候建议常开。
05 / cache header

几个字段
设计师该看得懂。

你不用会写、不用会改，但 —— 看到 DevTools Network 里的 Headers，知道哪些字段在控制缓存。 点左边切换。

互动左边是 4 个真实出现的 cache header，点一个 → 右边讲它在干什么。
RESPONSE HEADERS · 响应头节选
Cache-Control: max-age=3600
缓存 1 小时再回源
Cache-Control: no-cache
每次都回源问一遍
Cache-Control: public, max-age=31536000, immutable
永远不再问 — 给打包文件用的
ETag: "a3f7c8e2"
文件指纹 — 没变就 304
↑ POINT TO A HEADER

max-age — 缓存有效期

告诉浏览器 / CDN：拿到这个文件后，3600 秒 (1 小时) 内别再来问我，直接用就行。过了 1 小时，下一次请求会去回源验证。

设计师视角

如果你刚发版，但用户那 max-age=3600 还没走完 —— 他就是会看到旧的，直到这小时过完。这是为什么平台要 purge。

在哪看

DevTools Network → 选一个请求 → 看 Response Headers → 找 Cache-Control 这一行。

Headers Cache Preview Response Timing
Request URL: https://fan.com/_next/static/style.a3f7c.css
Status Code: 200 OK (from disk cache)
Cache-Control: public, max-age=31536000, immutable
ETag: "a3f7c8e2"
Age: 14523 (≈ 4 hours since CDN cached)
x-vercel-cache: HIT
" 看到 x-vercel-cache: HIT 或 200 OK (from disk cache) —— 就心里有数了：这次根本没回源，看到的就是缓存。 —— 一行字胜过一万句"为什么没生效"

DECISION TREE · 决策树

"改了没生效" 排查 5 步

START你 push 了新代码 / 改了样式，访问网址 仍是旧的。
↓
STEP 1先 hard refresh (Cmd+Shift+R) → 解决八成情况。
↓ 仍旧
STEP 2开无痕窗口访问 — 跳过本机所有缓存。还旧 → 不是你电脑的问题。
↓ 仍旧

↓ 仍旧

↓ 仍旧

CHEATSHEET · 速查卡

4 层缓存 + 4 个 header + 3 个动作

层 / 字段 速度 它在干什么 谁来清 / 怎么用
L0 BROWSER < 1 ms 你电脑里的浏览器缓存。每个用户一份。 Cmd+Shift+R · 你自己
L1 CDN ~ 50 ms Cloudflare / Vercel Edge — 离你最近的"便利店"。 平台后台 Purge· 工程师
L2 SERVER ~ 200 ms 服务端的 Redis / 反向代理 — 公司机房里的仓库。 flush redis · 后端工程师
L3 ORIGIN ~ 1 s 真正的源头 (代码 / 数据库)。每次发版第一个到这。 git push (会自动)
— 4 个 cache-control 值 —
max-age=3600 1 h 缓存这个文件 1 小时，过期再回源。 普通页面常用
no-cache 每次问 每次都问源 (但还是会带 ETag 校验)。 动态数据 / API
no-store 不存 完全不缓存，每次都重新下载。 密码 / 隐私页面
immutable 永久 永远不再问源 — 配合文件名 hash 才能用。 打包后的 JS / CSS
— 3 个清缓存动作 —
hard refresh 10 s 跳过 L0，强制问 CDN 一次。 Cmd+Shift+R
purge CDN 30 s 把 L1 整个边缘缓存丢掉。 Vercel / Cloudflare 后台
无痕窗口 3 s 用真用户视角访问，跳过所有本机缓存。 Cmd+Shift+N
看到 "为什么我改了网站没生效" 的瞬间，先问自己："我清到第几层了？" — 这一句话就能让你和工程师沟通顺畅 100 倍。
