状态
放在哪里。

不从 Redux、Zustand 这些词开始。先看页面里会变的东西：只影响自己，还是影响整个页面？ 是用户操作出来的，还是接口给回来的？答案决定状态应该放在哪里。

新能力判断状态范围
关键词local / page / URL / server
例子筛选列表
前置B1 / B2 信息归属
01 / 先别背名词

先问影响范围，
再决定放哪里。

状态管理不是“找一个高级工具”。它先是一个归属判断：这个变化只有一个按钮知道，还是列表、统计、URL、接口都要知道。

只影响自己？比如一个下拉菜单是否展开。它通常是 local state，不需要惊动整页。
影响同一页？比如筛选条件会改变列表、统计、空状态。它通常要上提到 page state。
要被记住或分享？比如搜索词、排序、分页。它可能应该进 URL，刷新和分享后还在。
02 / 一个筛选列表

同一个页面里，
状态有不同位置。

下面不是技术题，是产品判断题。你只需要把“谁受影响”说清楚，前端自然知道要往哪里放。

下拉菜单展开

它只影响这个控件自己。关掉页面、刷新页面，都不需要记住“刚才是不是展开”。

放哪里local state
谁能改这个下拉控件
影响谁只影响自己

筛选条件

筛选一变，列表、结果数量、空状态都会变。它不应该藏在某个按钮里。

放哪里page state
谁能改筛选器 / 清空按钮
影响谁列表、统计、空状态

排序和页码

用户刷新、复制链接、返回上一页时，排序和页码最好还在。它适合变成 URL 状态。

放哪里URL state
谁能改排序控件 / 分页器
影响谁页面地址和数据请求

商品列表数据

商品名称、价格、库存不是前端自己编的。它来自接口，所以要按 server state 处理。

放哪里server state
谁能改接口 / 后台 / 权限
影响谁列表、详情、缓存
