03 / 拆开

3 种行为的
决策树。

每个区块在小屏上只能选三种之一。怎么选？看内容性质——可重排 → 换布局；可截 → 截断；不能省 → 溢出。 点卡片看反例。

1
→

换布局 · reflow

REARRANGE
适用
可重新排列且无信息丢失的内容（卡片、字段网格）
实现
grid → 1 列 / flex direction 改 column
风险
纵向滚很长
例
KPI 4 列 → mobile 1 列
反例
12 列表格强行 reflow 成 1 列 → 每行变 12 个堆叠的 label/value，60 行表格变 720 屏的瀑布流。表格不是字段网格，不能 reflow。
2
…

截断 · truncate

CLIP
适用
列表 / 表格的单行 / 单元格——只要内容能让用户识别"是不是这条"
实现
text-overflow: ellipsis / 单行 / multi-line clamp / hover 看全
风险
截掉关键信息 → 用户认错
例
表格只显示 name + status，其它列截掉 + … 展开
反例
金额、状态这种"决策依据"也被截掉 → 用户看到 "¥48..." 不知道是 48 元还是 4800 万。可识别字段可截，决策字段不能截。
3
⋯

溢出 · overflow

DEFER
适用
操作密集 / 不能省略 / 不能重排（toolbar、tab 数量、表格全列）
实现
横滚 / 更多菜单 / 抽屉 / 模态
风险
用户看不到隐藏的入口
例
ActionBar 主按钮显示 + ⋯收纳次要
反例
所有按钮都收进 ⋯ 菜单 → 用户找不到"新建"。溢出菜单是放次要操作的，主操作必须留在外面。

三选一决策树

内容可重排？
→ 换布局
内容能截、能识别？
→ 截断
不能省、不能重排？
→ 溢出
三选一不是绝对——但每个区块要主动选一个，
不能默认（浏览器会替你选，结果一定是坏体验）。

04 / 看更多

「客户列表」3 断点——
看人家填的行为表。

7 列 + 标题 + 操作。每行预填好了：桌面 / 平板 / 手机各显示什么、怎么截、超了怎么办。⚠ 标的行是设计师最容易漏标的地方。

客户列表 · 响应式行为表（交给工程师的"列优先级 + 截断 + 溢出"规格）
7 列 · 3 断点
列 / 区块 桌面 ≥ 1024 平板 768–1023 手机 < 768 规则 / 为什么
姓名 · name
P1		 显示 显示 卡片主标题 永远显示。单行截断 + ellipsis，最多 12 字。"刘洋…" 用户能识别就行——hover 看全。
公司 · company
P2		 显示 截断 ellipsis 卡片副标题 第二重要字段。平板宽度不够，截到 10 字 + …。手机做卡片副标题，独立一行。
职位 · title
P3		 显示 隐藏 → "详情" 隐藏 辅助信息。平板就开始消失——点开行/卡片才看到。
状态 · status
P1		 徽章显示 徽章显示 卡片右下角小字 决策字段——用户靠它判断"要不要联系"。永远不省略，只换样式。
跟进人 · owner
P3		 显示 隐藏 → "详情" 隐藏 辅助信息。注意：销售本人视角时，owner 升级为 P1——他要看哪些归自己。响应式规则可能因角色变化。
合作金额 · deal
P1		 右对齐 mono 右对齐 mono 卡片右上角 永远显示。金额 ≥ 10 位（¥4,800,000,000）→ 缩字号 + 单位"亿"，不要换行——换行后表格行高崩。
最近联系 · lastTouch
P3		 显示 隐藏 → "详情" 隐藏 辅助信息。显示时用相对时间（3 天前 / 1 周前），不要绝对日期。
整体布局 · layout 表格 7 列 表格 4 列 + ⋯ 卡片视图 morph 手机不是缩小表格——是换组件：表格 → 卡片列表，每张卡 2-3 个字段。
页面标题 · title "客户管理 · 仪表盘" 完整显示 截 2 行 · clamp-2 长标题用 line-clamp: 2，不能单行截到看不懂。"客户管…" 用户不知道是什么页。
⚠ 没标"列优先级" = 工程师瞎砍 稿子上 7 列你只画了桌面态——窄屏时工程师不知道哪几列先消失。结果：他可能把"金额"砍了留"职位"。P1 / P2 / P3 必须在稿子上写。
⚠ 金额超 10 位 桌面看着 ¥480k 没事——但客户里有人是 ¥4,800,000,000 万亿大单。溢出规则要在稿子上演示一次："超 8 位 → 改 ¥48 亿 + 缩字号"。
⚠ 手机不是"缩小表格" 很多人画 mobile 稿时还画表格——只是更窄。错了。手机要 morph 成卡片——一行 = 1 张卡，2-3 个字段。表格本身不适合 375 宽。
⚠ 同一列在不同角色下优先级不同 "跟进人"对管理员是 P3（辅助）；对销售本人是 P1（核心）。响应式行为可能因角色而变——稿子上记得标"哪个角色看的"。
05 / 把这件事变成习惯

下次接响应式，
桌面稿旁边
画一张行为表。

从这一篇开始，"响应式"不再是"再画一稿"，而是每个区块的"行为标签"—— 交付时附在每个块上，工程师不用猜。

i

每个区块单独选三选一

不要"整页用一个规则"。导航、表格、卡片各有最优解——主导航和侧栏可能都用"溢出"，但表格用"截断"。

按区块决策，不按页面决策。

ii

截断必须能 hover 看全

截掉信息但留路径——用户能恢复就 OK。tooltip / 点开详情 / 长按展开都行，完全丢就是 bug。

把"截断"和"删除"分清楚：截断是视觉省略，数据还在。

iii

溢出菜单的入口要看得见

⋯ 按钮、hamburger、横滚条都要让用户知道"被收进哪了"。看不见的入口 = 不存在的功能。

横滚加 fade 渐变、⋯ 加角标数字、hamburger 用动画展开——都是给用户"还有东西"的信号。

划重点 · TL;DR

响应式不是"按比例缩小"，是 每个区块的小屏行为决策——换布局 / 截断 / 溢出三选一。可重排选换布局；可识别选截断；不能省、不能重排选溢出。

下次接到响应式需求，桌面稿旁边画一张"每个区块的移动行为"。每个块标一个 tag：reflow / truncate / overflow + 具体方案。工程师拿到不用猜，整页 morph 而不是 shrink。
