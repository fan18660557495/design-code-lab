01 / 问题

"详情页里的一块"，
其实是另一个对象。

客户详情页里那个"相关合同 section"——你画稿时把它当成详情页的一部分。 工程师看到这一块的瞬间想的是：这是哪个对象，怎么挂回来的，删了会同步删吗？

页面思维 · PAGE

"我在详情页里加了'相关合同'"

稿子上画了一个 section：3 行合同，每行有金额、状态、日期。看起来挺完整。

工程师追问：这"合同"是什么？是把字段复制一份过来？还是引用 Contract 对象？删一个合同会不会同步把客户上的引用也删掉？合同列表页的同一行 UI 要不要复用？

你：呃……我没想这么细。

对象思维 · OBJECT

"客户 has-many 合同，反向引用挂回来"

稿子配一句话：Customer has-many Contract，通过 Contract.customer_id 反向引用。

工程师收到：schema 直接落——一张 contracts 表带外键，详情页 section 只是 Contract.where(customer_id=X) 渲染。删客户的级联策略是另一道决策。

结果：追问 → 0 次，外加你赢得了"懂数据模型"的标签。

幕二.5 · 拆开
03.5 / 拆开

引用还分 强弱。

cardinality 讲多少个。强弱讲另一件事——它能不能脱离父对象独立活着。两者是引用的两个正交维度。

弱引用 · WEAK / ASSOCIATION

客户 → 合同

两端各有独立生命周期
EXAMPLE 客户「星河科技」可以归档，合同仍然存在（只是变成了"归档客户"的合同）。合同有自己的列表页 /contracts、自己的详情页、自己的筛选轴。
  • 有独立详情页——/contracts/c-2401 能单独打开
  • 有独立列表/筛选——按金额、状态、签约时间筛
  • 父对象删除时能保留（设为 archived 或 reassign）
  • 典型：客户↔合同、用户↔订单、文档↔评论
强引用 · STRONG / COMPOSITION

订单 → 订单行

寄生在父对象上，没有独立生命周期
EXAMPLE 订单的"订单行 / 子订单"不能脱离订单单独提交。后台没有 /order-lines 列表，也没有"全公司订单行"这种页面——删订单它们级联跟着没。
  • 没有独立详情页——只在父对象详情里展开
  • 没有独立列表——不存在跨父的全量页面
  • 父对象删除时级联删除（cascade delete）
  • 典型：订单→订单行、表单→表单字段、问卷→题目
3 问判断 · IS-IT-STRONG TEST

看到一个对象拿不准强还是弱——问 3 个问题。

1
它有独立的详情页吗？ /foo/123 能直接打开？
YES / NO
2
它能被列表页单独筛选/查询吗？ "所有 foo" 的页面存在？
YES / NO
3
父对象删除时，它会跟着没吗？ cascade delete？
YES / NO

前两个 YES + 第三个 NO = 弱引用——必须配独立列表 + 详情 + 反向筛选。 前两个 NO + 第三个 YES = 强引用——画在父对象详情页里就行，没有独立入口。 混着出现的？很可能你建模建错了——回去重审"它到底是不是一个对象"。

强引用的子对象——几乎不可能成为主对象（接X3 主对象选择）；
弱引用的子对象——换个视角，它就能当主对象。
3 问 → 1 个结论

05 / 把这件事变成习惯

下次看到"列表"、"相关"，
先问引用问题。

从这一篇开始，需求评审里出现"列表"、"相关"、"关联"这些词， 你脑子里的第一反应应该是引用图，不是 section 布局。

i

看到列表先问"这是谁的列表"

一个 section 显示 N 条记录——先问：这 N 条是什么对象？通过什么字段反向引用回来？

两个问题没问之前不画稿子。问完再画，工程师不需要追问。

ii

Cardinality 是设计决策，不只是数据问题

1:N 还是 M:N 直接影响 UI：单选还是多选？能不能撤销关联？删一个会不会同步删？

需求评审的第一句直接问 cardinality ——比"按钮在哪、什么色"重要 10 倍。

iii

self-reference 不是 bug，是常见模式

任务有子任务、组织有上级组织、项目有依赖项目。看到"层级"或"依赖"两个词，立刻想 self-reference。

设计时要画清楚：哪些是父、哪些是子，能不能多个父，循环依赖怎么挡。

划重点 · TL;DR

需求里出现"列表"或"相关"，立即问 3 件事：是另一个对象吗？怎么引用回来？cardinality（几比几）+ 强弱（能不能独立活）？

语言一变，整个对话就不一样了——你不再是在描述"详情页里有什么"，你是在设计一张可被工程师直接落 schema 的引用图。
