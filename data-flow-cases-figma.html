1商品卡片数据流
02 / 案例 1

这些数据，
分别从哪一层传？

最经典的电商场景 —— 商品列表里的每张卡片。商品名、图、价、按钮状态， 看似都在 ProductCard 里，但它们的来源天差地别。

题目 下面是商品页的组件树：App → ProductList → ProductCard → PriceTag / AddButton。 把右侧 5 个数据节点拖到正确的层级，使其在那一层产生（或被传递起点为该层）。
↓ 拖动数据节点
products[]
filterValue
product
discount
loading
= 跨层传递的 props
= 在该层产生的 state
→ 商品页组件树
<App /> 根 · 数据来源
<ProductList /> 列表 · 含筛选
<ProductCard /> 单个商品卡
<PriceTag /> 价格 + 折扣
<AddButton /> 加购按钮
已放对 0 / 5 个
答案解析

每个数据为什么放在那一层 —

  • products[] 放在 App：整个页面共享的数据，从 API 取来后，作为顶层数据源往下传。
  • filterValue 放在 ProductList：只有列表自己关心的筛选状态。放 App 太高（其他模块用不上），放 ProductCard 太低（卡片之间无法共享筛选）。
  • product 放在 ProductCard：每张卡片接收一个 product 对象（在 ProductList 里 .map 出来）。
  • discount 放在 PriceTag：只有价格组件关心的字段，从 product 里取出来传给它。
  • loading 放在 AddButton：加购按钮自己的临时状态（点击后到 API 返回前）。其他组件都不用知道。

规律：跨多个组件用的数据，放在它们的"共同祖先"层；某个组件独有的临时状态，留在它自己里面。

2找出 props drilling + 状态归属
03 / 案例 2

filterValue
到底归 谁 管？

FilterPanel 是产生筛选值的地方，但它跟 ProductList 是兄弟 —— 兄弟不能直接互传。state 该放在哪？这就是 "lifting state up"。

题目 FilterPanel 让用户选筛选条件，ProductList 要拿这个值去过滤商品。 下面 3 个组件中，把 filterValue state 拖到正确的层。 放错变红，放对变绿。
<ProductPage /> 两个子组件的共同父
↓ 把 state 拖到这里
<FilterPanel /> 用户在这里选筛选
↓ 把 state 拖到这里
<ProductList /> 需要 filterValue 过滤商品
↓ 把 state 拖到这里
↓ 拖动:
filterValue
提示： FilterPanel 和 ProductList 是兄弟。兄弟之间的数据共享，
必须放到它们的共同父，再分别 props 传下去。这叫 lifting state up。
答案解析

为什么放在 ProductPage —

× FilterPanel： 它能产生 filterValue 没错。但 FilterPanel 内部的 state，ProductList 看不到（兄弟之间无法直接通信）。

× ProductList： 它能消费 filterValue 没错。但用户操作的是 FilterPanel —— 你没法在被消费的地方"凭空生出来"这个状态。

√ ProductPage： 两个组件的共同父。state 放这里，往下传给 FilterPanel（含 setter）和 ProductList（含值）。这就是 lifting state up。

口诀： 兄弟想共享 → 提到最近的共同祖先。

3主题嵌套与覆盖
04 / 案例 3

嵌套 3 层 Provider，
叶子读 哪 个？

全站 Light，营销区 Dark，里面某个按钮 Brand —— 这是真实落地里 最常踩坑的场景。Context 离得越近，覆盖优先级越高。

题目 左边 3 个 Provider 节点（Light · Dark · Brand）可以拖动调换嵌套顺序。 右边树里的 5 个叶子组件会实时显示自己读到的最终主题。 规则：从内向外找，最近的 Provider 赢。
→ 嵌套顺序（顶到底 = 外到内）
⋮⋮ ThemeProvider light
⋮⋮ ThemeProvider dark
⋮⋮ ThemeProvider brand
当前嵌套（外 → 内）：
light → dark → brand
规则： 离叶子最近的 Provider 赢
↓ 5 个叶子读到的主题
<Header /> → 读 light
└ <HeroBanner /> → 读 dark
　└ <BannerTitle /> → 读 dark
　　└ <CTAButton /> → 读 brand
<Footer /> → 读 light

试试把 Brand 拖到最外层 —— 你会发现所有叶子都变成 brand 了。 因为这时它没被任何更深的 Provider 覆盖。

规律

Context 嵌套：最近的赢 —

就近原则： 一个组件 useTheme() 时，React 从它往外找，碰到的第一个 ThemeProvider 就是答案。更外层的同名 Provider 都被屏蔽。

这跟 Figma 里 Variable Mode 的覆盖完全一致： 当前 Frame 设了 dark mode，里面的元素就读 dark；外层 frame 是 light 也不影响。

实战意义： 想让营销区整体变 Dark，包一层 ThemeProvider value="dark" 就行 —— 不用改任何子组件代码。

4局部配置 vs 全局
05 / 案例 4

圆角到底用
哪一层？

全站圆角 8px，后台页所有元素 4px（更"严肃"），里面有个 Pill 徽章固定 999px —— 这个真实需求要靠三层覆盖叠出来。

题目 下面三个值，你能说出每个该来自哪一层吗？ 点下方按钮 toggle 各层 on/off，看 pill 徽章的圆角怎么变化。
①
全局 (theme.json / CSS variable)

整个产品的默认圆角。一改全产品都跟着改。--radius: 8px

8px
②
区域 context (后台页 ThemeProvider)

包一层 Provider，覆盖后台页所有元素的圆角，更"严肃"。radius: 4px

4px
③
组件特例 (Pill 徽章)

这种组件视觉上必须是胶囊形，不参与全局圆角语言。borderRadius: 999 写在组件里。

999px
商品页 Button
默认按钮
读到 → 8px
后台 Button
后台按钮
读到 → 4px
后台里的 Pill 徽章
已发布
读到 → 999px
规律

三层覆盖：从远到近，越近越赢 —

① 全局： 用 design tokens / CSS variables。改一处全产品都变 —— 这是设计师在 Figma 里改 Variable 的对应物。

② 区域 context： 用 Provider 包一层。子树里所有元素自动读到新值，不用改组件代码。"营销区 dark" "后台 4px" 都是这种。

③ 组件特例： 直接在组件里写死。某些视觉就是要打破系统（Pill / FAB / 圆形头像）—— 这些不该跟全局圆角联动。

判断口诀： "改一处影响多大？" 全产品都该变 → 全局；只改一片区域 → context；只这一种组件特立独行 → 写死。

5OrderItem 数据传递
06 / 案例 5

传整个 order，
还是只传 需要 的？

订单列表项内部有 4 个子组件：状态徽章、价格、头像、菜单。 要不要把整个 order 对象一股脑塞给每个子组件？这是最小知识原则。

题目 下面是同一个 OrderItem 的两种实现 —— BAD 把整个 order 传给每个子组件， GOOD 只传子组件需要的字段。哪种更好，为什么？
王
#OD-2026-0892 王女士的订单
已发货 ¥ 689.00
⋮
BAD · 反例

每个子组件都收完整 order

// OrderItem.tsx — 反例
<OrderItem order={order} />

// 内部
<StatusBadge order={order} />
<Avatar      order={order} />
<PriceLabel  order={order} />
<MoreMenu    order={order} />
问题： StatusBadge 只用 order.status，但它能看见整个 order。 未来谁误用了 order.totalPrice，没人察觉。组件之间耦合到整个数据结构。
BAD · 后果

改 order 字段名，所有组件都炸

// 后端把 status 改名成 state
需要 检查:
  StatusBadge — 在用 order.status
  Avatar      — 没在用 status，
                但收了 order，要排查
  PriceLabel  — 同上
  MoreMenu    — 同上
  → 4 个组件都得开看
耦合代价： 一个不该关心 status 的组件也"知道" status。 字段重命名 / 字段删除时，不知道哪些组件真的依赖它 —— 只能逐个排查。
GOOD · 正例

每个子组件只收需要的字段

// OrderItem.tsx — 正例
<StatusBadge status={order.status} />
<Avatar      name={order.user.name} />
<PriceLabel  amount={order.totalPrice} />
<MoreMenu    orderId={order.id}
              onAction={handleAction} />
好处： 每个子组件的"知识范围"都收窄了。 StatusBadge 只知道 status；Avatar 只知道 name。 它们能在任何地方复用，不依赖 order 这个外壳。
GOOD · 收益

改字段，影响范围一目了然

// 后端把 status 改名成 state
影响:
  StatusBadge — 改 prop 名 status → state
  Avatar      — 不影响 ✓
  PriceLabel  — 不影响 ✓
  MoreMenu    — 不影响 ✓
  → 只改 1 处
设计师视角： 这跟 Figma 里 Component Properties 是一回事 —— 一个 Badge 组件应该有 label / variant 这种语义化属性， 而不是塞进去整个 "Order" 对象。
原则

最小知识原则 (Law of Demeter) —

简单说： 一个组件应该只知道它真正需要的，不要多给它一行代码用不到的信息。

给 Badge 组件传整个 order，等于在它身上挂了 10 个钩子 —— 任何字段改名都可能波及它，哪怕它根本没在用。

什么时候可以传整个对象？ 当组件确实是那个对象的"全权代表"时（比如 ProductCard 接收 product，因为它就是负责渲染整个 product 的）。叶子组件就别这么干了。

5 个案例的核心 take-away

01
商品卡 · 共同祖先原则

跨组件共享的数据，往上提到所有用它的组件的"共同父"。

02
状态归属 · Lifting state up

兄弟组件想共享 state，必须放在它们的最近共同祖先里。

03
主题嵌套 · 就近原则

多层 Context 嵌套时，离消费者最近的那一层赢。

04
覆盖 · 全局 → context → 局部

三层叠加，越具体越优先 —— 跟 Figma Variable Mode 一致。

05
OrderItem · 最小知识原则

叶子组件只接收它真正需要的字段，不要接收整个对象。
