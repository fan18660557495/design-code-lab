01 / 问题

这个 prop
是从 哪 来的？

看一段 React 代码，最难的不是语法，是数据流向 —— 这个值哪来的？被传到哪去？谁该改它？

不懂数据流的设计师

"我改了 Figma，前端说还要改 5 个文件"

你只改了一个商品名的字段，但代码里同样一段数据被一路传过 5 层组件。每层都得改 prop 名字，否则中间断掉。

这就是 数据沿组件树流动 的代价 —— 不懂这事，你跟前端讨论改动就靠猜。

懂数据流的设计师

"我看一眼就知道这个 prop 从哪来"

看到组件树，能指出这个值的来源、被谁拿走、谁该负责。讨论改动时一句话说清"这个值往上提到 ProductPage 就好"。

这一篇要建立的就是这套词汇表 —— 父子、props、state、context、覆盖。

04 / State

组件自己的
小账本。

State = 组件自管的内存（toggle 开关、loading、当前选中 tab 等）。 不传给父，也不传给子 —— 自己用。

ProductCard 1 · AddButton
state: { loading: false }
ProductCard 2 · AddButton
state: { loading: false }
ProductCard 3 · AddButton
state: { loading: false }

点任意一个按钮 → 只有那个组件的 state 翻页。
各管各的 —— 这就是组件 自管 state 的含义，跟其他组件无关。

05 / Drilling

中转 N 层
的 疲劳。

某个值从顶层一路传到第 8 层叶子，中间 7 层组件都不用它，只是"中转"。 这种状态叫 Props Drilling —— 是引出 Context 的痛点。

App user = { name: "范米花儿" }
Header 📦 user (中转，自己不用)
Nav 📦 user (中转，自己不用)
NavRight 📦 user (中转，自己不用)
Avatar ✓ user (终于用上了)
痛点：user 走了 4 层才到 Avatar。中间 3 层组件都得"承接 + 转发"，但自己根本不用。 哪天 user 字段改名（比如 user → currentUser），4 个文件都要改。

解药：Context（下一节）。
06 / Context

跨层级
的 广播。

Context = 在某个节点放一个 Provider，子树里所有节点都能直接读它， 不用一层层传 props。就近读取 —— 更近的 Provider 优先。

App 📡 UserContext.Provider value={user}
Header 无 props 传递 ✨
Nav 无 props 传递 ✨
NavRight 无 props 传递 ✨
Avatar 📡 useContext(UserContext) → user
Provider 一放 → 整棵子树都能"广播订阅"。
Avatar 直接 useContext 拿到 user，跳过中间 3 层。

Figma 的 Variables / Modes 就是这套思路 —— 在某个 frame 设一次，子层都能读。
07 / 覆盖

全局 → context →
组件特例。

Token / Theme 不是"一个值"，是分层的。全局给默认，context 给区域， 组件可以再加特例。就近读取，局部覆盖优先。

全局 theme: #16161d (黑)
营销区域 context: #ff4d2e (红)
组件特例: #1f6b3a (绿)
最终按钮颜色 →
绿 (组件特例)

点上面的 toggle 关闭某层 — 看叶子最终读到哪个值。
规则：就近读取 —— 局部 > context > 全局。

08 / 速查

几条
准则带走。

基础篇结束。下一篇（进阶）用真实场景练习这些概念。

速查表 · 6 条带走

  • Props — 父→子单向，不能反过来
  • State — 组件自管的小账本，不外传
  • Drilling — 中转 ≥ 3 层就考虑 Context
  • Context — 跨层共享 / 区域覆盖，子树就近订阅
  • 覆盖优先级 — 局部 > Context > 全局，最近的赢
  • 状态归属 — 兄弟共享时上提到共同父级
