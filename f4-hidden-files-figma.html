02 / 密码本

.env
是密码本。

所有"不能让人看到的字符串"都集中放在这里 —— API key、数据库密码、第三方服务的 token。这是 F 系列里最关键的一节。

它长什么样

纯文本文件，每行一个 KEY=VALUE。文件名前面有个点，所以默认是隐藏文件 —— Finder 默认看不见。

# 这是 .env —— 当成密码本看
OPENAI_API_KEY=sk-proj-aBc123XyZ...
AWS_ACCESS_KEY=AKIA1234...
AWS_SECRET=a8K2...
DATABASE_URL=postgresql://user:PASSWORD@host/db
STRIPE_SECRET=sk_live_...

里面的每一串都等于钱。OpenAI key 被人拿到 → 账单飙；AWS key 被人拿到 → 直接挖矿；Stripe live key 被人拿到 → 转走客户付的钱。

为什么 git 必须忽略

git 默认会把仓库里所有文件都 commit + push 上去。如果 .env 在仓库里，一个 git push 之后整个互联网都能看到。

更糟糕的是 —— GitHub 上有一群专门扫泄露 key 的爬虫机器人，分钟级就会扫到。一旦在公开仓库，视为永远泄露。

所以：.env 这个文件名，几乎所有项目的 .gitignore 第一行就写着它。这是行业默认。

一句话：.env 在你本地放着、用着 —— 但千万别让它进 git 历史。一进去就别想全删干净。

05 / 救命三连

不小心传了
怎么办。

发现 .env 已经在 GitHub 上了 —— 别慌，但速度第一。 三步救命，按顺序做。

⚠ 黄金原则：一旦 .env 上了公开仓库，里面的所有 key 立刻视作泄露。 哪怕你 5 秒后就 git rm 了 —— 假设有爬虫已经扫到。这条不动摇。
  • 01

    第一步：立刻 rotate（替换）密钥

    去 OpenAI / AWS / Stripe 的控制台，把那个 key 删掉 —— 重新生成一个新的。 原来那串 sk-proj... 从这一刻起作废，谁拿着也用不了。

    → 这一步比清 git 历史重要 100 倍。先做这个。
  • 02

    第二步：把 .env 从 git 里踢掉

    从仓库里删除这个文件，但本地保留（你还要用）。然后写进 .gitignore，下次 push 不会再带上去。

    $git rm --cached .env
    $echo ".env" >> .gitignore # 加进忽略清单
    $git commit -m "remove .env from tracking"
    $git push     → 复制给 AI 让它解释每条命令。--cached 是关键 —— 只删 git 里的，不删硬盘上的。
  • 03

    第三步：极端时清 git 历史

    就算 .env 现在不在 HEAD 里 —— git 历史快照里还能搜到。 如果是开源项目 / 已经被 fork 过，需要用 git filter-repo 或 BFG Repo-Cleaner 重写整个历史。然后强制 push。

    这步有破坏性，对所有协作者都有影响。私人小项目通常跳过，直接干掉旧仓库重建更省事。

    → 这一步写在最后是因为：就算清干净了历史，密钥还是要换。第 1 步永远跳不过。
设计师特供 如果不知道用了什么 key —— 看 .env 的内容、看代码里 process.env.XXX 的地方。 每个用过的服务都要去换一次。AI 可以帮你列清单。

commit 决策树 + .gitignore 必备清单 + 上传前自检 —— 想发项目到 GitHub 之前，过一遍这页。

· 5 类文件 commit 决策树

这是哪一类？
· 配置文件（package.json / tsconfig.json / vite.config.js）
→ ✅ 必须 commit。别人靠这份装项目。
· 源码 .tsx / .css / .js（src/ 下的所有手写代码）
→ ✅ 必须 commit。这就是项目本身。
· 资源（public/ 下的 logo、字体、图片）
→ ✅ 大部分要 commit。除非超过 50 MB —— 那要单独处理。
· 隐藏 .env / .DS_Store（密码 + 系统垃圾）
→ ❌ 绝对不 commit。.env 是密码本；.DS_Store 是 macOS 偷偷生成的没用文件。
· 生成产物（node_modules / dist / .next）
→ ❌ 不 commit。能从配置 + 源码自动重建，没意义传。
· 例外：.env.example（空模板）要 commit， .gitignore 自己也要 commit。

· .gitignore 必备 5 条

实际上 GitHub 给每种语言准备了官方模板 —— 创建仓库时勾一下 "Add .gitignore: Node"，自动生成 100 行的版本，能省事很多。

· 上传到 GitHub 前 · 自检 5 条

  • 确认 .gitignore 存在，并且第一行就是 node_modules/ 或 .env。
  • 用 git status 扫一眼 —— .env、密码、API key 这些字眼不能在"to be committed" 列表里。
  • 搜 .tsx / .js 文件，看有没有硬编码的 key（"sk-..." / "AKIA..." / "sk_live_..."）。有就改成读 process.env.XXX。
  • git log 看一眼历史，确认从来没有 commit 过 .env。万一有 —— 回 §05 救命。
  • 仓库设为 Private 还是 Public？不确定就先 Private。Public 之前再过一次以上 4 条。

最后一条最实用：去 GitHub 设置里开 Secret Scanning。它会自动扫你 push 的代码、发现 key 立刻邮件提醒。免费、强烈建议开。
