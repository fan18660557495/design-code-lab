01 / 误解

部署到底
是什么。

设计师听到"部署"，脑子里通常浮现三种画面 —— 全是错的。 先把这三个误解清掉，后面才好看流程。

YOU THINK · 你以为

"部署 = 把文件上传到服务器"

就像把一个 .fig 拖到 Drive，对方就看得见。

电脑里的代码 → 服务器
"复制粘贴一份就行了吧"
ACTUALLY · 实际上

"部署 = 让平台帮你 build，再分发到 CDN"

你给的是原料（源代码），平台先 build 成成品（HTML/CSS/JS），再把成品复制到全世界 50 个机房 —— 谁访问就从最近那个机房拿。

源码 → build → 成品 → 分发到 CDN
YOU THINK · 你以为

"我改了代码，网站就更新了"

就像 Figma —— 改完保存，链接打开就是新版。

改代码 → "网址自动新版"
ACTUALLY · 实际上

"改完要再 deploy 一次，才更新"

线上跑的是上次 build 出来的成品，跟你电脑上的源码已经断开关系。要让线上更新，必须重新触发部署 —— 通常是 git push，平台自动重 build。

改代码 → git push → 平台重 build → 新版上线
YOU THINK · 你以为

"部署完就有 fan.com 这种域名了"

类似 Notion 给的链接，发布完就直接是漂亮地址。

部署成功 → "fan.com 自动归我"
ACTUALLY · 实际上

"默认是 xxx.vercel.app，自定义域名要单独绑"

平台先给一个临时地址（比如 fan-portfolio-9j2k.vercel.app）—— 立刻能访问。漂亮的 fan.com 是另一件事：你得先买域名，再去注册商改 DNS 记录指向 Vercel。§04 会教。

部署成功 → xxx.vercel.app（默认）
fan.com → 自己买 + 自己绑 DNS
" 你不是在上传文件，你是在给平台一份配方，让它在云端做菜，再把成品端到全球桌上。 —— 这就是为什么本地能跑、线上挂的事会发生
02 / 全链路

git push 之后
发生了7 件事。

从你按下回车那一刻，到全球能访问，中间 7 步 —— 都是平台自动跑。 点 开始，看它一步步亮起来。

玩法点"开始演示" → 7 步会按时间顺序点亮，每步对应平台日志里的一段输出。
就绪 · idle
  1. 01
    YOU · 你

    你 git push 到 GitHub

    本地仓库的提交，被推上远程仓库。这一步跟部署本身没关系 —— 但它是触发器。

    $ git push origin main → github.com/you/repo
  2. 02
    GITHUB

    GitHub 通知 Vercel · webhook

    GitHub 收到新 push 后，给 Vercel 打个电话："你接管的那个项目又有新代码了"。这条电话叫 webhook，本质是一个 HTTP 请求。

    github → POST /api/v1/hooks → vercel
  3. 03
    VERCEL · 平台

    Vercel 拉代码 · git clone

    Vercel 启动一台干净的云端虚拟机，把你的代码完整 clone 一份到这台临时机器上。每次部署都重新拉 —— 不复用上次的环境。

    vm.new() → git clone github.com/you/repo
  4. 04
    VERCEL · 平台

    跑 npm install · 装零件

    读你 package.json，把所有依赖装到云端那台机器的 node_modules/。这一步可能要 30 秒到 2 分钟，看依赖多不多。

    $ npm install → 320 packages installed
  5. 05
    VERCEL · 平台

    跑 npm run build · 打包

    把源码 + 资源压缩、合并、生成静态文件 —— 输出到 .next/ 或 dist/。这是真正能上线的成品。build 失败，整个部署就停在这。

    $ npm run build → .next/ · 42 pages compiled
  6. 06
    CDN · 全球

    把成品分发到全球 CDN

    build 出来的文件被复制到 50+ 个 edge 节点（北京、东京、新加坡、法兰克福……）。用户访问时，从地理上最近的节点下载 —— 这就是为什么打开飞快。

    .next/ → sync to 50 regions · pdx · sin · hkg · cdg · ...
  7. 07
    DONE · 完成

    给你一个 URL

    平台生成一个临时域名（fan-portfolio-9j2k.vercel.app）—— 立刻能访问。同时还有一个固定生产域名（你绑过自定义域名的话）。整个过程从 push 到能访问，通常 15 秒到 2 分钟。

    https://fan-portfolio-9j2k.vercel.app · live
关键认知 — 1、2 是 GitHub 那边的事，3-6 全在 Vercel 的云端临时机器上跑（你看不见）。 所以"本地能跑、线上挂" = 你电脑跟那台云端机器环境不一样。最常见原因：版本号没锁、env 没设、文件名大小写。

04 / 自定义域名

把 fan.com
绑上去。

临时域名 xxx.vercel.app 不像样。 想要 fan.com？4 步。重点是 DNS。

STEP 01 · 在 Vercel

项目里 Add Domain

Vercel 项目页 → Settings → Domains → 输入 fan.com → Add。 这一步只是告诉 Vercel："以后访问 fan.com 的人，请响应给他们我这个项目"。

Vercel · Settings → Domains → + fan.com
STEP 02 · 在域名注册商

改 DNS 记录

去你买域名的地方（Cloudflare / 阿里云 / GoDaddy），找DNS 管理。 按 Vercel 提示加一条 A 或 CNAME 记录。

Cloudflare · DNS → + A 记录 → 76.76.21.21
STEP 03 · 等

DNS 全球生效

DNS 改完不是立刻全球都知道 —— 各地 DNS 缓存陆续刷新，需要 5 分钟到 24 小时。 可以在 whatsmydns.net 查进度。

刚改完 → 中国能访问 · 美国还没生效
耐心 · 大部分 5-30 分钟内全球生效
STEP 04 · Vercel 自动

自动配 HTTPS 证书

DNS 一生效，Vercel 立刻通过 Let's Encrypt 给你申请免费 SSL 证书 —— 你什么都不用做。 访问 https://fan.com 浏览器自动绿锁。

Let's Encrypt → SSL 自动签发 · https://fan.com ✓

Type Name Content TTL
A fan.com 185.199.108.153 Auto
MX fan.com mx1.gmail.com Auto
A fan.com 76.76.21.21 ← Vercel 给的 Auto

真实截图大致长这样 —— 上面两行是已有的（不要动），下面两行 = Vercel 提示你加的。橘色那条加好就行。
小心：A 记录冲突的话，把旧的删掉。同一个 host 不能有两条 A 指不同 IP。
" DNS 是电话簿：浏览器先查"fan.com 在哪"，电话簿告诉它"在 76.76.21.21"，浏览器才去那个地址。 改 DNS 就是改电话簿。 —— 你不是在挪文件，是在改"指路牌"
05 / 翻车现场

部署最常见
三种翻车。

每条都配诊断步骤 —— 按顺序排查，90% 能自己解决。 实在不行截图给 AI，把红字那一段带上。

  • 01

    Build 失败 —— 红色 ✕ 出现在日志里

    最常见。可能原因：依赖没装上、TypeScript 报错、文件名大小写不对（macOS 不区分，Linux 区分）。看日志最后红字那一段，第一行通常就是答案。

    诊断步骤 · 5 分钟
    1.打开 Vercel 项目页 → Deployments → 点失败那次。
    2.看 Build Logs —— 找第一行红字（不是中间，是最早出现的红字）。
    3.常见错：Module not found = 包没装；Type error = TS 报错；EISDIR = 文件路径错。
    4.本地试 npm run build —— 复现一下，本地能跑通才能上线跑通。
    → 复制最后 10 行日志 + 截图 → 喂给 AI，90% 直接给答案。
  • 02

    环境变量没设 —— 本地能跑、线上挂

    本地 .env 里有 OPENAI_API_KEY=sk-...，但 Vercel 上没设过 —— 线上跑的时候这个变量是 undefined，调 API 直接 500。.env 文件不会上传到 Vercel，必须在 Vercel 后台手填。

    诊断步骤 · 2 分钟
    1.本地翻一下 .env 或 .env.local，记下所有非空变量名。
    2.Vercel 项目 → Settings → Environment Variables → 一一加上。
    3.注意：加完要重新 Deploy 一次 —— 已部署的版本读不到新变量。
    4.区分 Production / Preview / Development —— 一般三个都勾上。
    → 关 2 就是练这个：env 没加 → build OK → 运行时挂。
  • 03

    域名解析没生效 —— DNS 改完打不开

    刚改完 DNS 立刻访问 fan.com 报错或转圈 —— 大概率是 DNS 缓存还没刷新。 你电脑、ISP、各地 DNS 服务器都缓存着旧记录，得等。

    诊断步骤 · 等 + 临时绕过
    1.先去 whatsmydns.net 查 fan.com —— 看绿勾覆盖度。
    2.如果大部分还是 ✕，等就完了。一般 5-30 分钟。极端情况 24 小时。
    3.急着自己测：本机改 hosts 文件 —— 加一行 76.76.21.21 fan.com，临时强制走 Vercel。
    4.测完记得删掉 hosts 那行，否则别人 DNS 已生效你还看老的。
    → 90% 情况是单纯没等够 —— 先泡杯咖啡再回来看。

· 3 个翻车 · 速诊

症状先查
build 失败 看日志第一行红字。 Module not found = 装包；Type error = TS。
本地能跑、线上挂 查 Environment Variables —— .env 没在 Vercel 配过。加完要 redeploy。
域名打不开 DNS 没生效。whatsmydns.net 查；急用先改本机 hosts。
改了没更新 浏览器缓存或 CDN 缓存。Cmd+Shift+R 强刷，或在 URL 后加 ?v=2。
部署很慢 大概率 npm install 装了一堆 —— 看是不是 dependencies 写多了。
90% 的部署问题就这 5 类。卡住先来这看一眼。

· 部署关键术语

词是什么
build 把源码 → 成品的过程。压缩、合并、转 TS、生成 HTML。
deploy 把成品送上线的过程。包含 build + 分发。
CDN 遍布全球的缓存机房网。让访问者从最近的机房拿文件。
DNS 域名 → IP 的电话簿。改 DNS = 改 fan.com 指向哪台机器。
env .env。装密钥/配置的文件。不上传 git，要在 Vercel 单独配。
webhook 一种"事件回调"。GitHub 用它告诉 Vercel"代码变了"。
AI 用任何一个术语，回来这查一眼 —— 不再卡词。
