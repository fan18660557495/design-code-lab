03 / 拆开

4 件套
各自的
设计原则。

每件套有 3 条经验法则。点开看代码示例 + "破坏这条会怎样"。 违反一条 = 组件难复用。

Props · 形态开关
0/3 已看
1 命名要业务化——描述意图，不描述实现 › 
// 描述实现 — 把样式细节绑死
<Button
  bgColor="blue"
  paddingX="20"
  borderRadius="100" />
// 描述意图 — 实现可换
<Button
  variant="primary"
  size="md"
  shape="pill" />
破坏这条会怎样 改个色，所有 bgColor="blue" 都要扫一遍替换。API 把实现细节固化了——换皮肤就是 breaking change。
2 默认值要安全——什么都不传也能跑 › 
// 必填一堆 — 上手成本高
<Modal
  isOpen={true}
  disabled={false}
  loading={false}
  closable={true}
  backdrop={true} />
// 一行就能跑 — 默认值兜底
<Modal isOpen>...</Modal>

// disabled / loading 默认 false
// closable / backdrop 默认 true
破坏这条会怎样 每次用都要写一长串 prop。消费方抄上一份"标准配置"开始改——文档读完才会用。
3 避免布尔 + 字符串混用——统一 enum › 
// 类型混乱 — true 是哪个值？
size: 'sm' | true | 'lg'

<Input size />
// ??? 这等于 'md' 还是 'lg'
// 联合字面量 — 一眼明白
size: 'sm' | 'md' | 'lg'

<Input size="md" />
破坏这条会怎样 <Input size /> 等同于 size=true——布尔 + 字符串混用时，简写歧义无解。
Events · 消息通道
0/3 已看
1 on 前缀——是惯例，别发明新词 › 
// 自创命名 — 消费方要去查
handleClick
emitChange
whenClose
clickHandler
// 行业惯例 — 看名字就懂
onClick
onChange
onClose
onSubmit
破坏这条会怎样 IDE 补全、文档目录、跨组件搜索全部失效——认知一致性是组件库的隐形规范。
2 载荷要完整——别让消费方从 event 里挖 › 
// 只给 event — 消费方自己挖值
onChange: (e) => void

// 用法：
onChange={e => setValue(e.target.value)}
// 给业务值 + event 兜底
onChange: (
  newValue: string,
  e?: Event
) => void

onChange={v => setValue(v)}
破坏这条会怎样 每个消费方都写一遍 e.target.value。换底层实现（比如 input → contenteditable），所有消费方都炸。
3 同步 vs 异步——Promise 支持挂住默认行为 › 
// 只能同步 — 异步校验无路
onConfirm: () => void

// 关闭立刻发生
// 服务端校验失败也已关了
// Promise 挂住关闭
onConfirm: () =>
  void | Promise<boolean>

// 返回 false / reject → 不关
// resolve(true) → 才关
破坏这条会怎样 消费方做异步校验时 Modal 已经关了——结果是手动 setState 重新打开 + 闪烁。
Slots · 定制点
0/3 已看
1 命名要语义——header / body / footer，不是 slot1 › 
// 编号 slot — 看不出位置
<Modal>
  <template slot="slot1">
    标题
  </template>
  <template slot="slot3">
    按钮
  </template>
</Modal>
// 语义 slot — 一眼就懂
<Modal>
  <template slot="title">
    确认删除
  </template>
  <template slot="footer">
    <Button>OK</Button>
  </template>
</Modal>
破坏这条会怎样 消费方每次回查文档"slot2 是 body 还是 footer 来着"。重排序时所有用法都要改。
2 默认 fallback——空时显示什么 › 
// 空 slot 出现空白区
<Card>
  <// 不填 title 槽>
  <template slot="body">...</>
</Card>
// → 顶部留个空白 header
// 空 slot → 整块不渲染
<Card>
  <template slot="body">...</>
</Card>
// title 没传 → 没有 header 区
// footer 没传 → 默认按钮
破坏这条会怎样 每个消费方都要处理"空态"占位。或者画稿子时设计师永远要画一个"空 slot 会怎样"。
3 不要过度 slot——slot 太多 = 没有形态 › 
// 全是 slot — 等于啥都没规范
<Modal>
  <template slot="closeBtn"/>
  <template slot="backdrop"/>
  <template slot="shell"/>
  <template slot="icon"/>
  ...
</Modal>
// 只 slot 真正需要定制的
<Modal>
  <template slot="title"/>
  <template slot="children"/>
  <template slot="footer"/>
</Modal>
// close 按钮、蒙层、a11y 都由组件统一
破坏这条会怎样 所有消费方都要自己画 close 按钮——位置、样式、a11y 全部分裂。组件失去了"统一"的价值。
State · 谁管什么
0/3 已看
1 受控 vs 非受控——选一个不要混 › 
// 两个都给 — 优先级模糊
<Input
  value={v}
  defaultValue="hi"
  onChange={...} />
// 谁赢？value？default？
// 切换时 React 会警告
// 受控：value + onChange
<Input value={v} onChange={set}/>

// 或非受控：defaultValue
<Input defaultValue="hi"/>

// 不要两个都给
破坏这条会怎样 组件初始化后切换 controlled/uncontrolled = 内部状态丢失。React 会警告，结果是莫名其妙的 bug。
2 内部状态不能 leak——animation / focus / scroll › 
// 暴露实现细节 — 消费方乱用
animationPhase: 'idle' | 'enter'
                | 'leave'
focusedIndex: number
scrollTop: number
// → 消费方写代码控制动画态
// → 改实现就是 breaking change
// 内部状态 — 不暴露
// 只暴露语义事件：
onOpenComplete: () => void
onFocus: (item) => void

// 实现想怎么变都行
破坏这条会怎样 消费方依赖 animationPhase=== 'enter' 写逻辑。改用 CSS animation 时——所有依赖都炸。
3 画线在"在意/不在意"分界处 › 
// 没分清 — 全暴露
isOpen, scrollTop,
focusedItem, animPhase,
menuPortalNode, ariaId,
hoverIndex, searchQuery...
// 30 个 prop。消费方迷茫
// 分清后 — 暴露 5 个
// 受控：
value, isOpen, onChange,
onOpenChange, onSearch

// 内部（不暴露）：
// scroll, focus, anim, hover...
破坏这条会怎样 API 表面积越大，组件越难升级。每一个对外暴露的 state 都是一个"消费方可能依赖的钩子"。
4 件套 × 3 法则 = 12 条经验。
每条都是无数项目踩出来的——你点开几条，就把别人的坑读完了。

完整的 Modal 契约——
看人家是怎么填的。

把 4 道门全部预填好——重点看 ⚠ 标的几条，是设计师最容易踩的状态边界坑。

Modal 组件 · 完整契约

这就是 Section 01 那个 Modal 的展开版——每一道门都附"为什么"。

PROPS 从外面进来 5 条
isOpen : boolean 必填 / 受控——Modal 自己不存"开 / 关"状态，由父级用 isOpen 驱动。这是受控组件的核心。
title ?: string 标题文字——简单场景用 string prop。复杂场景（要传 JSX）改用 header slot（见下方）。
size : 'sm' | 'md' | 'lg' | 'full' 4 档枚举——决定 max-width。不传 width: number——把布局责任揽给组件就回不去。
closeOnEsc ?: boolean 默认 true。按 Esc 触发 onClose。"删除确认"这种关键操作可以设 false，防止误关。
closeOnOverlay ?: boolean 默认 true。点蒙层触发 onClose。同上——关键操作可以设 false。
EVENTS 向外发出 3 条
onClose : (reason) => void 必填——任何关闭动作（Esc / 蒙层 / × 按钮 / 取消按钮）都喊它，父级根据 reason 决定下一步。reason: 'esc' | 'overlay' | 'button' | 'cancel'。
onConfirm ?: () => void | Promise 主操作——确认按钮触发。返回 Promise 时 Modal 自动转 loading 等异步完成。
onCancel ?: () => void 取消按钮专用——和 onClose 是两件事。onCancel = "用户主动放弃"，onClose = "Modal 关上了"。
SLOTS 留口给父级填 3 条
header ?: ReactNode 可选——传了就覆盖默认 title 渲染。父级想加 icon / badge / 副标题，都从这道口塞进去。
children : ReactNode 必填 / 主内容——Modal 不规定中间放啥。表单、富文本、列表，父级想塞啥塞啥。
footer ?: ReactNode 可选——不传时回退到默认的"取消 + 确认"按钮。传了就完全替换（比如要加"保留草稿"按钮）。
STATE 边界 谁管哪部分 3 条
isOpen → 受控 父级管——Modal 不存自己的"开 / 关"。这条不商量——所有 Modal 都该是受控的。
中间输入框 → 父级决定 Modal 是容器——里面的输入框是 controlled 还是 uncontrolled，由调用方决定。Modal 不应预设。
focusLock / scrollLock / animationPhase → 内部 焦点锁、滚动锁、动画阶段——实现细节。永远不 leak 给父级，父级也不该在意。
⚠ 用 defaultOpen 让 Modal 自己管开关 稿子写"Modal 默认关着，点按钮打开"——很直觉。但父级丢失了控制权：路由切换时 Modal 不会自动关 / 表单成功后没法编程关闭。永远用 isOpen + onClose，开关存父级。
⚠ 把 content 写成 string prop 稿子写 content: "删除后无法恢复"——只能传纯文本。下次需要"加粗 + 红字 + 链接"就要重做 prop。用 children: ReactNode slot，富文本直接塞 JSX。
⚠ 把 footer 锁成 prop 稿子写 confirmText / cancelText 两个 string prop——99% 场景能用，但1% 的"保留草稿"按钮就要返工。footer 留 slot，默认 fallback 取消 / 确认，灵活性 + 一致性两头都有。
⚠ onClose 和 onCancel 合并成一个 看起来都是"关掉"——其实语义不同。Esc / 蒙层 / × 是 onClose（"组件关了"），取消按钮是 onCancel（"用户主动放弃"）。合并 = 父级再也分不清来源，埋日志的时候哭。
05 / 把这件事变成习惯

做组件前，
先写完 4 件套。

从这一篇开始，每次画一个新组件，先在脑子里把 4 件套填一遍—— 填完再开始动设计稿。

i

props 跟着意图走

不跟着实现细节走。variant 不是 bgColor、size 不是 paddingX。

命名问一句"换皮肤这名字还活吗"。活——好命名。死——重命。

ii

slot 是设计决策

不是工程偷懒。多给一个 slot = 把那部分内容主权交给消费方。

要明确这块是定制点还是统一规范——别为了"灵活"开口子，开多了组件就没了。

iii

受控早期决定

一开始想清楚：value + onChange 还是 defaultValue。

写完上线再改 API = breaking change——所有消费方都要回头改。早 10 分钟想，省后面 10 小时。

划重点 · TL;DR

组件契约不是"一份 props 列表"，是4 张分工清晰的表——props（能改啥）、events（会通知啥）、slots（能塞啥）、state（谁管啥）。

做组件前先写 4 件套——不写完不能开始画。不然画到一半，会发现 footer 该是 slot 还是 prop 没想清，整个返工。
