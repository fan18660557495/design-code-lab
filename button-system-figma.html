01 / 问题

一个组件
怎么悄悄变成了
30 个组件？

每加一种新需求 —— 红色的、大号的、带图标的 —— 就复制粘贴一个新组件。 三个月之后，"Button" 在你的库里有 30 个变体。改个圆角，要改 30 个地方。

BAD · 散装组件

30 个 Button，每个只用一次

每来一个新需求复制一份。改个全局圆角，找半天找不全；改完一个忘了改另一个，UI 立马走样。

ButtonRed ButtonRedLarge ButtonBlue ButtonBlueLarge ButtonBlueDisabled ButtonGhost ButtonGhostSm ButtonIconLeft ButtonIconRight ButtonLoading ButtonDanger ButtonDangerSm SubmitBtn CancelBtn …
GOOD · 一个 API

一个 Button，多个变体维度

一个组件，靠 variant / size / icon 这些维度组合出所有形态。改一个地方，全站统一。

<Button variant="primary" size="md" icon={<Plus/>}> 新建项目 </Button> // 同一组件、不同形态 ↓ <Button variant="ghost" size="sm" disabled>取消</Button> <Button variant="danger" loading>删除</Button>
02 / 概念

把"变化"扔进
4 个桶 里

组件库里所有"长得不一样"的东西，本质上只有四种来源 —— 谁决定的、什么时候变、变在哪一层。先认清桶，再做 API 决策。

prop

调用方决定

props · 由父组件传入的固定参数

页面用这个组件的时候，明确告诉它"我要这种样子"。组件自己不会变，调用方说什么就是什么。

variant="primary" size="md" disabled type="submit"
state

组件自管

internal state · 组件自己持有、自己改的状态

用户和组件交互的瞬时反馈：鼠标悬停、按下、获得焦点、loading 中。调用方一般不需要知道。

:hover :focus :active isLoading
slot

放进去任意东西

children / slot · 给调用方留的"插槽"

组件不规定里面放什么，调用方自由组合。文字、图标、甚至嵌套整个组件都行。"洞"一定要留得对。

children leftIcon rightIcon label
theme

全局变量

design tokens · 整个产品共用的设计变量

颜色、字号、圆角、间距 —— 这些不该写死在组件里。它们是全局规则，整个产品的视觉根，由 token 系统统一供给。

--color-primary --radius-md --font-sm --space-2

任意内容

04 / 演化

从 v1 到 v6，
看 API 怎么 长大

真实的组件不会一开始就完美。它从最简单的 default 开始， 每次需求来了往上加 prop / state / slot —— 直到某一刻必须拆开。下面是一个 Button 的演化轨迹。

1
DEFAULT ONLY

最朴素的 Button

只接收一段文字。一个写死的样式。够用 —— 直到产品经理说"这个按钮换成蓝色的"。

children
Button.tsx · v1
function Button({ children }) { return ( <button className="btn"> {children} </button> ); }
2
+ VARIANT

加 variant 区分主次

蓝色不是新组件，而是同一个组件的"另一种身份"。把"主操作 / 次操作 / 危险" 这种语义也带进来。

+ variant --color-*
Button.tsx · v2
function Button({ children, variant = 'primary' }) { return ( <button className={`btn btn-${variant}`}> {children} </button> ); }
3
+ SIZE

加 size 应对不同密度

表单里的按钮要小，关键 CTA 要大。同样是有限枚举值 —— 一个 prop 解决所有。

+ size --space-* --font-*
Button.tsx · v3
function Button({ children, variant = 'primary', size = 'md' }) { return ( <button className={`btn btn-${variant} btn-${size}`} > {children} </button> ); }
4
+ ICON SLOT

留出 icon 插槽

icon 不该是个 prop（要传 icon 名 + 渲染函数 + 大小？太重）。直接留 slot：调用方把任意 React 节点塞进来。

+ leftIcon + rightIcon
Button.tsx · v4
function Button({ children, variant, size, leftIcon, rightIcon }) { return ( <button className={...}> {leftIcon} <span>{children}</span> {rightIcon} </button> ); }
5
+ LOADING STATE

loading 是 prop 还是 state？

这里有取舍：loading 经常由"父组件知道接口请求中"决定，所以做成 prop 由调用方传。但 loading 时按钮自己要 disable + 切换 spinner —— 内部还是有 state 行为。

+ loading (输入) 内部 disabled+spinner
Button.tsx · v5
function Button({ children, variant, size, leftIcon, rightIcon, loading, disabled }) { const isDisabled = disabled || loading; return ( <button disabled={isDisabled}> {loading ? <Spinner/> : leftIcon} <span>{children}</span> {rightIcon} </button> ); }
6
SPLIT

到点了 —— 拆出子组件

"图标按钮没有文字"、"链接按钮带 href" —— 这些已经不是 Button 的另一面，而是不同的东西。 强行用 prop 撑会让 API 越来越复杂。该拆。

Button (主) IconButton (子) LinkButton (子)
LinkButton →
button/index.ts · v6
// 同根三件套 export { Button } from './Button'; export { IconButton } from './IconButton'; export { LinkButton } from './LinkButton'; // 共享 token / 样式 / 内部 useButton hook export { useButton } from './useButton';
05 / 边界

什么时候该 停手？
什么时候该 拆？

组件 API 不是越多越好。prop 5 - 6 个是上限，再多说明这个组件其实承担了多个职责 —— 它该被拆开。判定标准很朴素：列出来一目了然。

BAD · 一个组件硬扛

20 个 prop 的 Button

"加一个就好了" 加到第 20 个，调用方自己都不知道哪些 prop 能组合、哪些不能。文档比代码还长。

prop count
20 / 5
<Button variant="primary" size="md" href="/foo" // link? iconOnly // icon button? iconName="plus" iconPosition="left" tooltip="新建" badge={3} dropdown={[...]} confirm // 弹确认框? async // 包装 promise? cooldown={3000} loading disabled analyticsId="btn-x" onClick={...} />
GOOD · 拆成同根多组件

Button + 几个表亲

核心 Button 留 5 个 prop。需要"图标按钮"、"链接按钮"、"按钮组" 这些有自己结构的，独立成兄弟组件，共享底层 hook 和 token。

prop count
5 / 5
<Button /> 主按钮 · variant / size / loading / disabled / icon
<IconButton /> 纯图标按钮 · 自带 a11y label
<LinkButton /> 看着像按钮的链接 · href + 路由
<ButtonGroup /> 按钮组 · 共享尺寸、连边
<ConfirmButton /> 业务包装 · 弹确认 / 防双击
5

经验阈值：核心组件的 prop 在 5 个左右。

超过这个数，先看：是不是有几个 prop 总是"一起出现"？是不是某个 prop 只在某个 variant 下才有意义？是的话 —— 拆。

06 / 速查

决策树 +
命名 速查

遇到一个新需求 —— 比如"按钮要支持加载中" —— 走这棵树就知道该做成什么。 命名上也有几条简单规则，不背也能避免常见坑。

决策树 · 这个变化属于哪个桶？

DECISION TREE · prop / state / slot / token
这件事的"值" 整个产品共享吗？（颜色、字号、圆角、间距、阴影...）
→ theme token
这件事是"放什么内容进来"，而不是"怎么变样子"吗？（图标、文字、副标题...）
→ slot / children
这件事是调用方每次明确指定的、有限枚举值或开关吗？（variant、size、disabled...）
→ prop
这件事在组件运行时由它自己改，调用方一般不需要关心吗？（hover、focus、open/close...）
→ state

variant 是 prop，不是 N 个布尔

PROP NAMING

"主 / 次 / 危险" 这种"互斥的多选一"，用一个 enum prop —— 不要做成多个布尔，互斥关系会丢。

× isPrimary / isSecondary / isDanger
✓ variant: 'primary' | 'secondary' | 'danger'

布尔 prop 用 is / has / disabled

PROP NAMING

布尔 prop 一眼能看出"是 / 不是 / 有 / 没有"。避免起两个意思相反的布尔（loading vs notLoading）。

× notDisabled / loadingFlag
✓ disabled / loading / hasIcon

事件用 on + 动词

EVENT NAMING

事件回调统一用 on 开头，后面跟动词或动作。命名出现两种风格会让用的人猜半天。

× clickHandler / triggerSubmit
✓ onClick / onSubmit / onClose

token 走 --color-primary 不走 --red-500

TOKEN NAMING

token 表达"语义"而不是"具体颜色"。明天主色改成蓝，业务侧不用动。

× --red-500 / --color-FF4D2E
✓ --color-primary / --color-danger
