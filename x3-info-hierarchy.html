02.5 / 想清楚

选主对象 =
固定一个上下文。

山上的庙里，老和尚给小和尚讲故事——这一句话里藏着 4 个不同的主对象。每换一个根，业务语义就换一套。

根 · ROOT 1
主对象 = 老和尚
老和尚什么时候都讲故事。
app.com/monks/old-master
上下文：无

详情页讲的是老和尚这个人，故事是他的属性。听众、地点、天气都跟主题无关——它们最多是过滤条件，不影响"故事存不存在"。

根 · ROOT 2
主对象 = 讲故事这一次
需要小和尚在场，老和尚才讲。
app.com/sessions/s-2026-04-19
上下文：听众

详情页讲的是一次互动——老和尚 + 小和尚 + 这一次。没有听众这次"讲"就不成立。一对一的事件型对象，跟通话记录、订单类似。

根 · ROOT 3
主对象 = 庙
他只在庙里讲。
app.com/temples/t-123/talks
上下文：地点

详情页讲的是这座庙，老和尚和故事都是它的"内容"。同一个老和尚换庙就是另一份记录——多租户、场地预定、园区管理常见这种切法。

根 · ROOT 4
主对象 = 山上的庙
他只在山上的庙讲，
河边的庙不讲。
app.com/mountains/wutai/temples/t-123
上下文：嵌套地点

详情页要嵌套上下文——"山"是庙的容器。换成河边的庙，业务语义完全不同。集团-公司-部门、组织-项目-任务都是这种嵌套层级。

四个根 = 四套业务语义 = 四个完全不同的对象。
选主对象 = 选你愿意被绑死的那个上下文。"一张大页面塞所有东西"危险的真实原因——你拒绝绑定上下文，所以业务语义全是糊的。

03 / 拆开

主对象决定
5 件事。

把这 5 件事拉出来，单独看每一件如何从"主对象"派生。点开看 3 种主对象下的差异，以及搞错了会发生什么。

i
URL 模式 /[主对象]/[ID]?attached=[附属]
+
客户为主
/customers/123
?view=activities
商机为主
/opportunities/2401
?customer=123
合同为主
/contracts/c-2401
?customer=123
搞错了会发生什么：分享给同事的链接打开是错对象。本来想发"商机 #2401"——同事点开却是"客户 #123"的页面，找不到要讨论的那条商机。
ii
面包屑 链回主对象的列表/父对象，不显示附属
+
客户为主
客户 / 张明
商机为主
商机 / O-2401
/ 星河续约
合同为主
合同 / C-2401
搞错了会发生什么：用户回不去。把附属对象塞进面包屑——点客户名跳走，再点客户面包屑，回到客户列表而不是回到原来的商机。
iii
页面标题 / Tab title H1 = 主对象.name（或 .title）
+
客户为主
"张明 · 客户"
商机为主
"O-2401 续约 · 商机"
合同为主
"C-2401 · 合同"
搞错了会发生什么：浏览器多 tab 全叫同一个名。开 5 个商机 tab——5 个都显示"张明"，用户切 tab 凭运气。SEO 同样错。
iv
back / 返回行为 回主对象的列表页 · 用 Link/replace 不依赖浏览器历史
+
客户为主
← 客户列表
商机为主
← 商机列表
合同为主
← 合同列表
搞错了会发生什么：用户一直在客户和商机之间打转。点附属里的客户跳走 → 在客户页 back → 回到商机？回到客户列表？规则不清，用户就在两个对象间反复横跳。
v
焦点起点（a11y）tab 第一次按 → 主对象的主字段/主操作
+
客户为主
焦点 → 客户.name
→ "联系客户" 按钮
商机为主
焦点 → 商机.title
→ "推进到下一阶段"
合同为主
焦点 → 合同.contract_id
→ "下载合同 PDF"
搞错了会发生什么：键盘用户跳不到关键按钮。打开一个商机详情，焦点在客户卡上——按 Tab 7 次才能到达"推进商机"，这页面对键盘用户基本不可用。

04 / 看更多

3 个真实需求，
看人家怎么选主对象。

每个需求都有 2-3 个"看上去都对"的候选主对象。 下面直接给你 哪个对、为什么对——还有 选错会掉的坑。

场景 01

"看张明客户的本月活动"

"做一个页面，能看到张明这个客户本月所有跟进活动——电话、邮件、会议。"
✓ 正解
crm.com/customers/123?view=activities
客户/张明
← 回 客户列表
张明 客户
主对象 =Customer (张明)

用户问的是 "某客户的活动"——这是详情页的子视图，不是"全部活动 + 筛选"的列表。

① URL
/customers/123?view=activities
② 面包屑
客户 / 张明
③ 标题
张明
④ back
客户列表
⑤ 焦点
客户.name / 第一条活动
⚠ 陷阱：选 Activity 当主 → URL 变 /activities?customer=:id——那是"全部活动 + 按客户筛选"的列表页。同一句需求，换主对象 = 换产品概念。
场景 02

"审批一条加班申请"

"做一个页面，HR 主管收到通知后能查看、审批一条加班申请。"
✓ 正解
hr.com/overtime/R-1042
加班申请/R-1042
← 回 待审批列表
R-1042 申请单
主对象 =OvertimeRequest (申请单)

操作的对象是 "申请单"，不是人。申请人 (User) 是申请单上的附属字段。

① URL
/overtime/R-1042
② 面包屑
加班申请 / R-1042
③ 标题
R-1042 [申请单]
④ back
待审批列表
⑤ 焦点
"通过 / 驳回" 主操作按钮
⚠ 陷阱：选 User 当主 → URL 变 /users/:id/overtime——那是"看某个人所有加班"的报表视角。审批关注的是这一条申请，不是这个人。
场景 03

"在合同 C-2401 上加一条备注"

"打开合同 C-2401，在右边的备注区写一条新备注，保存。"
✓ 正解 · 主不变
crm.com/contracts/C-2401#new-note
合同/C-2401
← 回 合同列表
C-2401 合同
主对象 =Contract (C-2401) · 不变

备注是合同上的子操作——不换页。URL 加 #new-note 或开浮层，主对象保持是 Contract。

① URL
/contracts/C-2401#new-note
② 面包屑
合同 / C-2401 (不变)
③ 标题
C-2401 (不变)
④ back
合同列表 (不变)
⑤ 焦点
备注输入框 textarea
⚠ 陷阱：跳到 /notes/new 把 Note 当主 → 用户丢失合同上下文——写完不知道回哪里。子操作不该换 URL 主体——这是"主对象不变"的常见反例。
05 / 把这件事变成习惯

下次接需求，
先问"主对象是谁"。

这 4 个字一旦出现在你的脑子里，整张稿子的 URL/back/标题/SEO 就有了锚点。

i

第一句话先问"主对象是谁"

接到需求，画稿子之前问一句。整张页面的 URL/back/焦点/标题/SEO 全围着这个答案转。

没决定主对象前不动笔。

ii

附属 ≠ 不重要，只是"挂载"

附属对象不该决定 URL，也不该让用户"以为这是它的页面"。它的 link 通常打开它自己的详情页。

附属是只读摘要 + 跳转入口。

iii

"我设计了一张大页面"是危险句

一张页面就是一个主对象。两个对象的"详情都在同一页"意味着至少一个被错降级了。

要么改主，要么把附属换成摘要+跳转。

划重点 · TL;DR

下次设计画面前，把"主对象"写在稿子角落。这 4 个字在场，你就不会再做出"信息层级模糊"的稿子。

URL、面包屑、标题、back、焦点——5 件事一旦从"谁是主"派生，工程师再也不用追问。稿子第一眼就有结构。
