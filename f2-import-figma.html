02 / 三种路径

同事 / 楼下
/ 花名册。

所有 import 路径都属于这三种之一。 看完这页，再看 AI 写的代码，就能一眼分辨这是哪种打电话方式。

./ 同部门 / 同事

找同文件夹的人

./Button = "我这层楼里，叫 Button 的那个工位"。不能跨楼层，只能在自己这层喊一声。

../ 楼下 / 隔壁部门

先出门，再去找

../ = "先回到上一层"。每多一个 ../ 就再上一层。然后再说要去哪。

04 / @/ 别名

@/ 不是魔法
是项目花名册。

看到 @/components/Button 别慌 —— @ 不是什么特殊字符，是项目的配置文件里写的别名。 AI 给的 alias 偶尔不对会 build 失败 —— 知道在哪改。

● tsconfig.json 项目根目录 
{
  "compilerOptions": {
    "target": "ES2020",
    "module": "esnext",
    "jsx": "preserve",
    "baseUrl": ".",
    "paths": {
      "@/*": ["src/*"]
    }
  }
}

这 4 行是花名册本人

"paths" 里写的就是所有别名。 上面这段告诉 TypeScript（和后续的打包器）： "看到任何以 @/ 开头的 import，都翻译成 src/ 开头"。

@/components/Button
⤳
src/components/Button
@/lib/theme
⤳
src/lib/theme
@/pages/Home
⤳
src/pages/Home

Vite 项目同样的事是写在 vite.config.js 的 resolve.alias 里。 Next 自动认 tsconfig.json 这一段，不用再配。 不同模板可能用 ~/ 或 #/ 代替 @/ —— 看配置文件就知道。

注意 · GOTCHA AI 偶尔会写出 @/components/Button，但项目里根本没配这个 alias —— 这时跑起来会 build 失败。看 tsconfig.json 有没有 paths，没有就改回 ./ / ../。
实战 看到陌生项目，先扫一眼根目录的 tsconfig.json 或 vite.config.js —— 有没有 paths / alias。 有：可以放心用 @/。没有：老老实实 ./ / ../。
05 / 三种报错

90% 报错
都是这三种。

改完 import，跑起来红屏一片别慌。 看终端最后红色那段，对照下面三种 —— 修法基本都在 30 秒内。

  • 01
    Error: Cannot find module '@/components/Button' or its corresponding type declarations.

    "Cannot find module" —— 找不到那个文件

    翻译成人话："我按你给的路径走过去了，那地方没人"。 三种可能：(1) 路径打错了（最常见）；(2) 文件还没建；(3) 用了 @/ 但项目没配 alias。

    怎么修 先把那个路径真的拿出来对一遍 —— 在 VS Code / Cursor 里点开侧边栏，照着文件夹一层一层走。八九不离十是 ../ 数错了一层，或者大小写错了（Button ≠ button）。
  • 02
    Error: Module not found: Can't resolve '../lib/theme' in /src/pages

    "Can't resolve" —— 同一种病另一种说法

    这个是打包器（Vite / Webpack / Next）自己的报错口吻 —— 含义和上一条一模一样，只是说话方式不同。 留意它会告诉你 in /src/pages —— 这就是"出发地"。从这里开始数 ../。

    怎么修 抓住报错里的"in 哪个文件夹"信息。从那里出发，对着实际文件树重新数一遍 ../。如果 ../lib/theme 不对，多半要改成 ../../lib/theme —— 多一层。
  • 03
    Error: Cannot find module './button'. Did you mean './Button'?

    "Did you mean ./X" —— 大小写 / 拼写错

    这是最贴心的报错。系统已经猜到你想要的是哪个，但给的字符串不完全一样。 90% 是 大小写（button vs Button）；剩下是少打了或多打了一个字符。

    怎么修 照报错里的 Did you mean 后面那个字符串原样改回去。Mac 本地不区分大小写所以你看着像对，但 Linux 服务器上线就崩 —— 所以这种错一定要修。
通用方法 报错里引号内的字符串就是那条 import。报错里的 in 路径就是出发地。 把这两个圈出来，问题立刻具体。

路径三步法则

看到 import 先问这三个问题 · ASK 3 QUESTIONS
1

我在哪？

先确认 出发地。这条 import 写在哪个文件里？这个文件在哪个文件夹？—— 路径永远是从这里开始算。

2

目标在哪？

那个要 import 的文件，真实位置是什么？打开侧边栏看一眼。别凭印象。

3

几楼几号？

同层 → ./；上一层 → ../；上两层 → ../../；跨好几层 → 用 @/（前提是项目配了）。

· 符号速查

写法含义
./ 同文件夹。./Button = 同事，紧挨着。
../ 上一层文件夹。每多一个 ../ 多上一层。
../../ 上两层。三个就上三层 —— 看到 4 个以上，建议改 @/。
@/ 项目 src 根（最常见配置）。绝对位置，不管你在哪都一样。
react / lodash 没有任何前缀 = 来自 node_modules/，是装好的零件。
看 import 字符串第一个字符就够：. = 相对路径；@ = 别名；字母 = npm 包。

· 三个最容易翻车的点

陷阱怎么躲
数错 ../ 最常见。多一层 / 少一层。修法：手指头点屏幕一层一层数。
大小写 Mac 本地不区分，Linux 服务器区分。./button ≠ ./Button。
用了未配的 @/ 先看 tsconfig.json 有没有 paths。没有就别用。
忘了文件后缀 大多数项目可以省 .tsx / .ts / .js，但带上更稳。
移动文件忘改 把文件拖到别的文件夹后，所有 import 它的路径都得改。VS Code 会问你"要不要自动更新？"选是。
最稳的工作流：让 VS Code / Cursor 自动补全 —— 别手敲长路径。打 "./ 它会列出当前文件夹里所有文件。

· import 整体语法长这样

形式含义 / 例子
import X from './Y' 默认导出（default）。 import Button from './Button' —— 那个文件 export default 出来的东西，给它起个名叫 Button。
import { X } from './Y' 命名导出（named）。 import { useState } from 'react' —— 从 react 里挑出名字叫 useState 的那个东西。
import { X, Y, Z } from './m' 一次挑多个。大括号里的名字必须和源文件 export 的名字完全一致。
import * as X from './Y' 把那个文件里所有 export 当成一个对象挂在 X 上。少见，但偶尔会看到。
import './styles.css' 没有 from。表示"只是把这个文件加载进来" —— CSS 文件、副作用脚本常这样写。
默认 vs 命名：看源文件有没有 export default。AI 给的写法不对，会报 has no default export，把大括号加 / 去掉就好。
