01 / 问题

为什么稿子画完，
线上还是一堆空白和报错？

因为你画的是有数据 · 网好 · 用户有权限的那张稿。 真实世界里，用户大部分时间都不在这条 happy path 上。

一张稿 · HAPPY PATH

"稿子画了一张任务列表页"

有 10 条任务、3 个筛选、操作按钮。看起来完整。

工程师追问：

  • 首次加载长啥样？
  • 用户一个任务都没有呢？
  • 接口挂了呢？
  • 这个用户没权限看呢？
  • 数据有 1 周没刷新了呢？

你：呃……要再画 4 张？

5 张稿 · STATES

"任务列表 · 5 态完整"

(1) loading · skeleton 占位
(2) empty · "去创建第一个" CTA
(3) error · 错误 + 重试
(4) stale · 上次缓存 + 红点
(5) no-perm · "你没权限" + 申请入口

工程师收到：5 张稿 + 状态切换条件。能直接落代码。

结果：线上空白和报错 → 0 个。

幕二 · 拆开
03 / 拆开

5 状态的
设计骨架。

每个状态要回答 4 件事——版式骨架 / 主文案 / CTA / 二选行动。 点开任一卡片，看它的骨架和"不画会怎样"。

⏳
Loading · 加载中
数据在路上 · 给用户视觉反馈
+
版式骨架
skeleton 复刻最终布局——不是居中 spinner。让用户立刻"看见"内容大概的位置。
主文案
通常不需要——skeleton 本身就解释了"在加载"。文字反而画蛇添足。
主 CTA
无。用户唯一能做的事是等。
二选行动
加载超 3s 显示"还在加载……"。超 10s 给"取消"或"重试"。
⚠ 如果设计稿不画
工程师默认用 spinner → 用户看不到内容轮廓 → 感觉"什么都没有" → 误以为页面坏了，主动刷新 → 服务器多扛一轮。
⊘
Empty · 空
数据来了 · 但是空的
+
版式骨架
居中 大字 + 插图 + 引导操作。把"空"变成"系统的潜力"。
主文案
"你还没有 X"——主语是"你"。不是"暂无数据"——主语缺失。
主 CTA
主要操作。例如 创建第一个 X。不是"刷新页面"——你刷一万次也还是空。
二选行动
引导文案（怎么用）、看示例、跳模板库。给犹豫的用户一个软入口。
⚠ 如果设计稿不画
工程师写 "暂无数据" 4 个字 + 居中。用户站在"空"前面 → 不知道下一步该干嘛 → 离开。新用户留存就这么没了。
✗
Error · 报错
系统坏了 · 但用户还得用
+
版式骨架
居中错误图标 + 简短解释 + 操作。保持页面骨架（导航 / 标题），别整个跳错误页。
主文案
用户能理解的语言。"加载失败"，不是 "500 Internal Server Error"。错误码放小字。
主 CTA
重试（主要）。让用户先有一个不需要思考的动作。
二选行动
刷新页面 / 联系支持 / 报告问题。给"重试还是不行"的用户一个出路。
⚠ 如果设计稿不画
工程师直接把后端错误码贴上去——"NullPointerException at line 247"。用户骂；客服热线被打爆；其实问题可能 30 秒后自动恢复。
⏰
Stale · 过期 / 离线
数据是真的 · 但不是最新
+
版式骨架
内容 + 顶部 banner。内容稍灰 / 降饱和——视觉上提示"不新鲜"。
主文案
"数据为离线 · 上次更新 X 分钟前"。要带时间——用户判断要不要相信。
主 CTA
重新连接。把"刷新"做成显眼的动作，不是埋在菜单里。
二选行动
强制刷新 / 继续看离线版。给"知道是旧的但就想看看"的用户留路。
⚠ 如果设计稿不画
工程师不知道要不要标记"过期"——可能用户以为是实时数据，结果按 5 分钟前的库存下了单。线上事故就是这么来的。
🔒
No-permission · 无权限
你不能看 · 但要告诉你为什么
+
版式骨架
锁图标 + 解释 + 行动。可以保留页面外壳（顶部导航、面包屑），让用户知道"是这页不让看，不是整个应用挂了"。
主文案
"你的角色无法访问此页面" + 当前角色 + 需要的角色。把信息说全。
主 CTA
申请访问。一键发邮件给管理员，附上当前 URL 和原因。
二选行动
联系管理员 / 返回首页 / 切换账号。
⚠ 如果设计稿不画
工程师直接跳 403 错误页 → 用户不知道为啥进不去 → 以为是 bug → 找客服 → 客服查半天才发现是权限。
5 状态 × 4 维度 = 20 个设计决策。
漏一个 → 工程师猜 → 用户买单。

幕二.5 · 拆开
03.5 / 拆开

不同页面，
不一样的状态集。

上面 5 态是从"列表页"视角讲的。换成详情 / 创建 / 编辑——状态集就变了。 状态 ≠ 页面的属性，是 (页面类型 × 数据生命周期) 的笛卡尔积。

下面 4 种页面类型——每行是它"会真实进入的状态"。空 (empty) 在列表页是"没记录"，到详情页就变成 404 找不到； 编辑页 还多出"未保存 / 冲突 / 已保存"这种表单生命周期的状态。画稿前先看自己在哪一行。

列表页 · table
/customers
读多条记录
loading empty error stale no-perm
这就是上面讲的 5 态，"列表的标准模板"。empty 是 "0 条记录"——配引导 CTA。
详情页 · detail
/customers/:id
读单条记录
loading 404 not-found error stale no-perm
empty 在详情页不存在——记录要么有要么没有。"没有"是 404，UI 完全不同：不是"创建第一个"，是"找不到，或已被删除"。
创建页 · new
/customers/new
写一条新记录
blank draft 草稿恢复 submitting error 提交失败 success 跳转
完全不一样的一组——以"表单生命周期"为轴。没有 empty 和 stale，但有 draft（之前没填完）、submitting（正在提交）、success（提交完跳哪）。
编辑页 · edit
/customers/:id/edit
改一条已有记录
loading 初始 dirty 未保存 saving saved conflict 版本冲突 error no-perm
最复杂的一组——读 + 写都有。dirty（"未保存离开会丢"）和 conflict（"别人也改了"）是编辑页独有，但稿子最常漏画的两态。
"每张稿都画 5 态"是入门，"先看自己在哪一行"是进阶。
创建页画 empty 是错的（它本来就是空的），详情页画 empty 也是错的（应该画 404）。编辑页只画 5 态会漏 dirty / saving / conflict——线上事故经常从这里来。
下次画稿前，第一个问题："这是哪一种页面？" ——然后再从这一行抄状态集。

04 / 再做

轮到你——
给「设置中心」
配 5 张稿。

下面 5 个状态。每个有 3 个候选（版式 + 文案 + CTA 各 1 选）。 选完按"提交打分"——错一个会解释为什么。

设置中心 · 5 状态选择题

已作答 0 / 5
1 · Empty
新用户首次进入设置中心

这位用户刚注册，第一次打开"设置中心"。他还没改过任何东西。你要给他看什么？

✓ 正解 B——设置中心没有"真正的空"。系统总有一份默认配置；第一次进就该看到这些默认值。 选 A 是把"empty 模板"乱套——不是所有页面都需要 empty。 选 C 是把"loading"和"empty"混了——loading 是数据还没到，empty 是数据到了但是空。
原则：问"这页真的会空吗"——可能根本没有 empty 态。
2 · Loading
网络慢 · 2 秒拉取设置

网络慢，后端要 2 秒才返回当前用户的设置。这 2 秒里页面长啥样？

✓ B 或 C 都对——但代价不同。
B（skeleton）简单，但 2 秒后数据替换会有"闪烁"。
C（乐观渲染）体验最好，但要处理"用户已经改了，后端又把它覆盖了"的并发问题。
A 最差——大 spinner = 把"loading"做成了"什么都看不到"。
原则：loading 不是"挡住用户"，是"让用户看见即将到来的内容"。
3 · Error
接口 500 · 后端挂了

后端 500。前端没收到任何设置数据。怎么向用户解释？

✓ 正解 B——错误应保持页面骨架（导航 / 标题），用户知道是这页加载失败，不是整个应用挂了。
给 2 种行动：重试（也许只是临时网络抖动）+ 还原默认（实在不行用默认值先用着）。
A 不行——toast 一闪没了，用户没看清就消失，页面还是空白。
C 也不行——一处 API 挂了不该让用户感觉整个应用都挂了。
原则：错误要"局部化"——出错的是数据，不是页面。
4 · Stale
用户切断网了一会儿

用户在地铁里，网断了 1 分钟。他现在看到的设置是从本地缓存读的——不是实时的。怎么处理？

✓ 正解 B——告诉用户"你处于离线模式" + 限制能做什么（不能保存）+ 仍可浏览（不打断阅读）。
A 危险——用户改了设置以为生效了，网恢复后发现没保存，骂。
C 太暴力——网断 1 分钟，整个应用瘫痪？地铁里能用的 app 就这么没了。
原则：stale 的核心是"诚实但不阻断"。
5 · No-permission
普通员工进入"管理员设置"

普通员工点开了"管理员设置"链接（可能是别人发的）。他没权限。怎么处理？

✓ B 或 C 都对——看产品偏好。
B：清晰反馈 + 行动入口。适合开放型组织——让用户知道有这权限可以申请。
C：隐藏式权限。适合敏感场景——比如"财务报表"这种连存在都不希望普通员工知道的页面。
A 最差——通用 403 页，用户不知道哪个权限不够 / 找谁申请 / 该做什么。
原则：no-perm 不是"踢出去"，是"告诉为什么 + 给出路"。
05 / 把这件事变成习惯

下次接需求，
第一句问"这页有几态"。

"页面"和"状态集合"的差距，会出现在你每一次需求评审里。 让自己从"画一张稿"切换到"画一组稿"。

i

每张稿都画 5 态

至少 loading + empty + error。
stale 和 no-perm 看页面性质——只读页面通常不需要 stale；公开页面不需要 no-perm。

漏一个就交付一张猜的版本。

ii

空态不等于"暂无数据"

空态是"系统的潜力"——给用户一个清晰的下一步动作。"暂无数据"是放弃。

把空态当作产品 onboarding 的延伸：用户站在空前面，你说什么决定他留不留。

iii

错误不等于"出错了"

错误是"我帮不了你这次"——但要让用户能继续行动：重试 / 联系 / 报告 / 跳走。

错误码 / 堆栈写在小字里给排查用；主文案永远是人话。

划重点 · TL;DR

页面 = 1 张 happy path × 5+ 个状态稿。每个状态都是一张完整的稿——版式 / 文案 / CTA / 二选行动，全都不一样。

下次需求评审，第一句问"这页有几态？loading / empty / error 都画了吗？"。整个会议的语言就变了——从"差不多就行"到"5 张都要"。
