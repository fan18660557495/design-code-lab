01 / 问题

为什么半年后的需求，
会让团队推翻重做？

因为第一版稿子里——角色、字段、流程全是写死的。 加一个新角色，就要改 50 个页面、200 处代码。设计师以为"做完一版"是终点，其实每一版都要回答"未来怎么改"。

固化设计 · HARDCODED

"先把现在的需求做完再说"

稿子里硬编码 5 个角色：admin / sales / finance / cs / customer。所有 if-else 写在页面里。每个角色一张稿、每个字段一个 Input、每条流程一条路径——清清楚楚、稳稳当当。

半年后产品要加"审计员"角色：改 50 个页面、改 200 处代码、改 14 个权限判断。开发说"做不动了"，设计师重画整套。

真相：你不是在设计产品——你是在替自己挖坑。

注册表设计 · REGISTRY

"先把'未来会变的'识别出来"

稿子里画一张"角色 × 权限矩阵"——角色是数据，不是代码。一张表 + 一个角色管理后台，新增角色 = 加一行配置。

半年后加"审计员"：业务方自己在后台加一行；设计师不需要再画一稿；开发不需要再发布一次。

结果：新角色上线 → 0 工程师介入。

03 / 拆开

4 个扩展点。

设计稿要预留这 4 个扩展点——加新内容时不动布局。 点开每张卡片，看它在设计稿上长什么样，以及"反例"是怎么把未来钉死的。

01 FIELD REGISTRY

字段注册表

— Every field is a row of config, not a hardcoded input.

是什么：每个字段一条配置 {name, type, label, validation, visible_to: [roles]}。

设计稿表现：表单 / 详情区显示"字段渲染区"——里面字段从配置渲染，不是手画 Input。

反例：硬编码 12 个 <Input> → 加字段要改组件、改样式、改校验、改权限——4 处联动。

02 ROLE MATRIX

角色矩阵

— Roles × actions, two-dimensional. Add a row, not a page.

是什么：角色 × 字段 / 角色 × 操作的二维表。角色是行，能力是列。

设计稿表现：UI 元素带 role-required 标记，runtime 隐藏 / 显示——不是为每个角色画一张稿。

反例：每个角色一个独立的页面 / 路由 → 5 角色 = 5 倍工作；加 1 角色 = 重写一遍。

需求评审：报销审批功能

已回答 0 / 3 题
需求 · BRIEF
做一个"报销审批"功能。当前要 3 类报销（差旅、餐饮、办公）、3 级审批（直属 → 部门 → 财务）。 产品方还说："未来还会加 5 种报销类型 + 不同分公司可能审批级数不同。"
A
全硬编码

三段路径写死，看起来最稳。

  • 3 个独立表单页面（差旅 / 餐饮 / 办公）
  • 3 级审批路径在代码里写死
  • 审批人按职级查表硬绑定
预设：稳定、可控、可读性高
B
部分扩展

关键变量做枚举，其他写死。

  • 1 个表单 + 报销类型 dropdown
  • 类型决定显示哪些字段（字段注册表）
  • 3 级审批做成 enum，可改顺序
  • 审批人按部门 + 职级查表
预设：当下 + 未来 1 年均衡
C
全配置

业务方在后台自由组装。

  • 1 个表单 + "报销类型"配置
  • 审批流做成可视化编辑器
  • 字段 / 流程 / 规则全在后台配
  • 分公司可独立配置
预设：前期投入大，后期自由度高
Q1 / 03

哪个设计加"医疗报销"最便宜？

产品新需求：除了差旅 / 餐饮 / 办公，再加一种"医疗报销"。字段不同、审批级数不同。

Q2 / 03

哪个加"4 级审批"最便宜？

某分公司要求"直属 → 部门 → 分管副总 → 财务"四级审批。其他公司还是 3 级。

当前需求适合哪个？

回到需求评审现场——开始就该选 A / B / C 哪个？这题没有标准答案，看产品语境和团队成熟度。任何选择都有反馈。

05 / 把这件事变成习惯

下次评审，
多问一句：
"2 年后这块怎么变？"

扩展性不是"过度设计"——是把"会变的"和"不会变的"提前分开。 分开成本低，混在一起回头改成本巨高。

i

稿子要标"扩展点"

哪些地方将来会加内容——明示出来。注上 "// extension point"，工程师就知道这块要做成配置而不是硬编码。

不标注，工程师就照着画稿做——做完了才发现"加一行要重写"。

ii

过度配置也是坑

不是所有都该配。"3 年内不会变"的可以硬编码——比如登录页结构、品牌名。

"半年内可能变"的该配——比如角色、字段、流程、文案、阈值。识别清楚再下手。

iii

算清楚 break-even point

配置前期贵后期省，编码反之。算清楚多久回本——3 个月内不会改的，先编码；3 个月内会改 ≥3 次的，做配置。

不是凭感觉——是算账。

划重点 · TL;DR

扩展性 = 把"变化的部分"和"不变的部分"分开，把变化的部分做成数据 / 配置 / 注册表。设计稿要标出 4 个扩展点：字段注册表、角色矩阵、流程模板、主题 token。

需求评审最后一问："未来 2 年这块会怎么变？"答案决定哪里要做扩展点。答得出 → 设计稿带配置；答不出 → 再问一遍产品经理。
