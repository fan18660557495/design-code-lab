03 / 拆开

4 个扩展点。

设计稿要预留这 4 个扩展点——加新内容时不动布局。 点开每张卡片，看它在设计稿上长什么样，以及"反例"是怎么把未来钉死的。

01 FIELD REGISTRY

字段注册表

— Every field is a row of config, not a hardcoded input.

是什么：每个字段一条配置 {name, type, label, validation, visible_to: [roles]}。

设计稿表现：表单 / 详情区显示"字段渲染区"——里面字段从配置渲染，不是手画 Input。

反例：硬编码 12 个 <Input> → 加字段要改组件、改样式、改校验、改权限——4 处联动。

02 ROLE MATRIX

角色矩阵

— Roles × actions, two-dimensional. Add a row, not a page.

是什么：角色 × 字段 / 角色 × 操作的二维表。角色是行，能力是列。

设计稿表现：UI 元素带 role-required 标记，runtime 隐藏 / 显示——不是为每个角色画一张稿。

反例：每个角色一个独立的页面 / 路由 → 5 角色 = 5 倍工作；加 1 角色 = 重写一遍。

04 / 看更多

3 类扩展——
看人家怎么留位置。

加字段 / 加角色 / 加流程——B 端三大常见扩展。 每个案例预填了有扩展点 vs 没扩展点两套稿，加一条"为什么"，加一条⚠ 陷阱。

01

新增「VIP 等级」字段

FIELD · 加字段
✗ 没留扩展点
把现有 6 个字段挤变形
姓名 / 公司 / 手机 / 跟进人 / 状态 / 创建时间
+ ⚠ VIP 等级（硬塞在"公司"后面）

稿子上 6 个字段排得很满 —— 加第 7 个只能缩间距、改对齐、改栅格。每加一个都要重画。

✓ 留了扩展点
分区 + 自定义字段区域
[基础信息] 姓名 / 公司 / 手机
[业务信息] 跟进人 / 状态
[自定义字段] ← VIP 等级落这里
[系统信息] 创建时间

稿子上明确画一块"自定义字段区域" —— 加字段 = 配 JSON，布局不动。

为什么有效：字段被分到"按语义分区"的容器里，每个区是可伸缩的列表。加第 7 / 第 30 个字段都不影响其它区。
⚠ 陷阱扩展点放错地方：把"自定义字段"放在"基础信息"和"业务信息"之间 —— 用户找不到分区边界，看到的是"中间一段奇怪的字段"。扩展点要在固定字段的尾部，不是夹在中间。
02

新增「审计员」角色

ROLE · 加角色
✗ 没留扩展点
每页用 if-admin 散落判断
if (role === 'admin') showExport()
if (role === 'sales') hideFinance()
if (role === 'cs') maskPhone()
... 50 处散落判断

加"审计员"意味着改 50 处页面、找全所有 if-admin。设计师要重画 28 屏。

✓ 留了扩展点
角色 × 权限矩阵
查看 编辑 审批 导出
admin ✓ ✓ ✓ ✓
sales ✓ ✓ — —
customer ✓ — — —
auditor ✓ — — ✓ ← 加一行

稿子上有一张明确的"角色 × 操作"矩阵 —— 加角色 = 加一行，UI 不动。

为什么有效：权限是数据不是代码。角色矩阵让"谁能做什么"变成一张可视化、可配置的二维表，业务方自己能维护。
⚠ 陷阱矩阵粒度过细 —— 把"按钮可见"和"字段可见"混在一张大矩阵里。结果是几百格的稿子没人能读。按层拆：操作权限一张、字段权限一张、数据范围一张，三张小矩阵比一张大矩阵好用 10 倍。
03

新增「返厂维修」流程分支

FLOW · 加流程
✗ 没留扩展点
塞一个 hack 状态
已发货 → 已签收 → 已完成
↓
⚠ "返厂维修"
（硬塞个状态，不知道下一步去哪）

状态机里没预留分支点 —— 加"返厂"只能在某个状态旁边硬塞一个，没有明确的进出规则。日志、报表、统计全乱。

✓ 留了扩展点
状态机预留分支节点
已发货 → 已签收 → 【售后判定】 → 已完成
↓
返厂维修 → 重新发货
↓
退款关闭

状态机里明确有一个"售后判定"分支节点 —— 加"返厂维修"是在已知分支下加一个枝，状态机结构不变。

为什么有效：流程稿不是画"理想路径" —— 是画可能分叉的位置。在每个"可能多结果"的状态后画一个判定节点，未来的所有变种都挂在这个节点下。
⚠ 陷阱分支节点画太多，每个状态后都加"判定" —— 状态机变成蜘蛛网。只在"业务真有分叉"的位置加判定：付款后、发货后、签收后这三个是高频分叉点；"创建成功 → 待付款"这种线性步骤不需要加判定。
05 / 把这件事变成习惯

下次评审，
多问一句：
"2 年后这块怎么变？"

扩展性不是"过度设计"——是把"会变的"和"不会变的"提前分开。 分开成本低，混在一起回头改成本巨高。

i

稿子要标"扩展点"

哪些地方将来会加内容——明示出来。注上 "// extension point"，工程师就知道这块要做成配置而不是硬编码。

不标注，工程师就照着画稿做——做完了才发现"加一行要重写"。

ii

过度配置也是坑

不是所有都该配。"3 年内不会变"的可以硬编码——比如登录页结构、品牌名。

"半年内可能变"的该配——比如角色、字段、流程、文案、阈值。识别清楚再下手。

iii

算清楚 break-even point

配置前期贵后期省，编码反之。算清楚多久回本——3 个月内不会改的，先编码；3 个月内会改 ≥3 次的，做配置。

不是凭感觉——是算账。

划重点 · TL;DR

扩展性 = 把"变化的部分"和"不变的部分"分开，把变化的部分做成数据 / 配置 / 注册表。设计稿要标出 4 个扩展点：字段注册表、角色矩阵、流程模板、主题 token。

需求评审最后一问："未来 2 年这块会怎么变？"答案决定哪里要做扩展点。答得出 → 设计稿带配置；答不出 → 再问一遍产品经理。
