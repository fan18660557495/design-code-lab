01 / 解剖

一个 PR
长什么样

PR = Pull Request，"申请合并"。本质是一个网页，把这条分支改了什么展示给同事看，让同事点同意。 别看英文唬人，结构就 5 个部分。

点点看下面是一个真 PR 的复刻。左边卡片点一下，右边的"小窗"会告诉你那一块是干嘛的。
#247 open

fix: Login button hover state · 修登录按钮的 hover 颜色

YOU 你想把分支 fix/login-btn 合进 main · 改了 3 个文件 · +12 −5
Part 01 · 标题 + 描述

说清楚改了什么 / 为什么

第一行就告诉同事："这是个 fix，修登录按钮。" 描述里再写一两句"为啥要修"。标题写得越好，review 越快。

Part 02 · 文件改动清单

这次动了哪几个文件

右上那个 3 files changed —— 鼠标点进去，能看到具体哪几个文件被改了、每个文件加了几行删了几行。

Part 03 · diff 视图

红绿对比 · 每一行的改动

红色 = 删掉的，绿色 = 新加的。diff 是 PR 的核心 —— §02 会专门讲怎么读。

Part 04 · reviewer 区

谁要审、审完是什么状态

右边一栏列着"被点名审 PR 的人"。每人审完会留一个标记 —— approved / changes requested / commented。

Part 05 · 状态徽章

这个 PR 现在什么状态

顶上那个绿色小药丸：open 表示"还在审，没合"。下面有 4 种状态的全集。

PART 01 · 标题 + 描述
fix: Login button hover state
↑ 这一行就是标题。前缀 fix: 表示
  "这是个修 bug"。其他常见前缀：
  feat:  新功能
  chore: 杂活、不影响行为
  docs:  改文档

### 描述
登录按钮 hover 时颜色和设计稿不一致：
- 设计稿：#ff5e1a
- 代码里：#ff8c5e（差了一个色号）
- 这次把 token 改对。

// 描述里讲清"为什么"，
// reviewer 一看就知道你在干嘛。
open
还在审

PR 刚提交、有人审了但还没批准。大多数 PR 的常态。

changes requested
改了再合

reviewer 说"先改一下"。不是骂你，是常态 —— 改完再 push。

approved
同意了

至少一个 reviewer 点了"同意"。这时可以点 merge。

merged
合进去了

PR 完事了 —— 你的改动正式进 main。本次任务完成。

关键PR 不是"考试"，是对话。多数 PR 都会经历 open → changes → 你修改 → approved → merged 的循环 —— 被打回来很正常，反复改两轮才合并是常态。
02 / 红与绿

怎么读
diff

diff 就是"两版代码的对比"。规则只有三条 —— 红、绿、白。看懂这三种行，diff 就读完一半了。

+  绿色行
新增

这一行是这个 PR 加的。

−  红色行
删除

这一行是被删掉的。

    白色行
上下文

这一行没改，只是给你看周围。

04 / 礼貌

review 不是
挑刺大赛

同样一个意思，怎么说，决定了对方愿不愿意改。 review comment 的核心是提建议，不是下判决。

原则把"你写错了"翻译成"我看了一下，会不会 X 更好"。意思一样，对方愿意改的概率差三倍。
✕反例 · 直接判刑

"这个间距错的，不对。"

没说哪里错、没说应该是多少，只留下一个对方不知道怎么改的尴尬感。

✓正例 · 提建议

"这里 padding 是 14，
设计稿里应该是 16，
能不能改一下？"

具体 + 有依据 + 软语气。三秒能改完。

✕反例

"颜色不对啊。"

所谓"对"是按谁的标准？是 token 错？设计稿改了？信息量为零。

✓正例

"按钮用了 #ff8c5e，
但 token 是 --accent: #ff5e1a，
建议改成 token，免得以后不一致。"

指出问题 + 给出对的版本 + 解释为什么。

✕反例

"hover 也没做？"

问号挂在脸上，对方接收到的是责备，不是问题。

✓正例

"我看了一下，
这个按钮没有 hover 状态 ——
设计稿里有，要不要顺手加上？"

"我看了一下"是软化语气，"要不要顺手加上"给对方留余地。

三个万能句式

不知道怎么开口的时候，套这三个句式 —— 工程师同事用的也是它们。

PHRASE · 01

能不能 X？

能不能 [改成 / 加上 / 拆开]

提请求时最稳的句式 —— 把"指令"变成"问句"，对方更容易接住。

"能不能把这个 padding 改成 16，跟设计稿对齐？"

PHRASE · 02

考虑 Y？

考虑 [拆组件 / 加 a11y / 用 token]

建议结构性改动时用 —— "考虑"两个字，把"改"变成"想一下"。

"考虑把这两个按钮包一层 .btn-group？现在挨太紧。"

PHRASE · 03

我看了 Z 觉得 W

我看了 [设计稿 / token / 文档] 觉得

有依据时用 —— 不是"你错了"，是"我跟你信源对不上"。

"我看了 Figma 里的最新稿，觉得这里应该是 #ff5e1a，是不是 token 没拉？"

补充 实在拿不准是不是问题 —— 加个 "nit:"（nitpick · 鸡蛋里挑骨头）前缀。意思是"不重要，可以忽略"。 如：nit: 这里多了个空格。这是工程师圈子的标准做法。
05 / 视角

设计师能在
review 里贡献什么

不会写代码 ≠ 不能 review。有些东西工程师反而看不见 —— 颜色、间距、状态、文案、可访问性，这五件事是设计师的本职。

心态不要纠结"我不懂代码"。你审的不是代码 —— 你审的是用户最终会看见什么。 工程师懂代码，但不一定懂"这个橙色对不对"。
ANGLE · 01

●视觉

颜色对不对、字号对不对、字重对不对。这些不需要懂代码 —— 你看一眼就知道。和设计稿对一遍。

YOU你 · 在 Button.tsx 第 12 行

这里写了 #ff8c5e，跟设计稿里的橙色 #ff5e1a 不一样 —— 是不是 token 没拉到最新？

ANGLE · 02

↔间距

按钮 padding 是不是 16、卡片之间的 gap 是不是 24。设计师的眼睛对间距特别敏感 —— 工程师常常凑差不多的数。

YOU你 · 在 login.css 第 2 行

padding 是 10px 14px，但我们的间距栅格是 4 / 8 / 12 / 16 / 24 —— 改成 12px 16px 行不行？

ANGLE · 03

◐状态

hover / focus / disabled / loading —— 设计稿里画了，但代码里经常忘做。这是 review 里设计师 80% 的有效输出。

YOU你 · 在 login.css 第 6 行

看见你写了 /* TODO: hover state */ —— 设计稿里 hover 是颜色加深 10%，要不要顺手加上？这关合并前补一下吧。

ANGLE · 04

"文案

按钮上写的字、错误提示、空状态文案 —— 工程师常常自己编一个临时词。你过一遍能挡掉一堆奇奇怪怪的"系统出错请稍后重试"。

YOU你 · 在 LoginPage.tsx 第 23 行

label 写的是 "Login"，我们的产品文案是中文 —— 改成 "登录" 比较一致。注册按钮同理。

ANGLE · 05

◯可访问性

aria-label 加了吗、对比度够吗、能用键盘 tab 到吗。这是设计师最容易"会聊"但工程师容易"忘做"的一块。

YOU你 · 在 Button.tsx 第 12 行

这个按钮里只有 onClick，没有 aria-label。如果按钮里只有图标的话，屏幕阅读器读不出来 —— 加一个？

ANGLE · 06

∗边界情况

文字超长怎么办、加载时长按钮要不要禁用、网络断了显示啥。设计师做产品比工程师久 —— 这种"边界"经常是你最先想到。

YOU你 · 在 LoginPage.tsx

登录中要不要加个 loading 态？现在点击之后没反馈，用户会以为按钮坏了 —— 网络慢的时候特别明显。

底气遇到代码逻辑看不懂的部分 ——那一段交给工程师 reviewer，不是你的责任。 你只对"最终用户能看见的样子"负责，这就够了。
