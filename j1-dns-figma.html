01 / 别名

浏览器不知道
fan.com 是什么。

域名是给人看的别名。电脑只认数字 IP。 fan.com 必须先翻译成 104.21.5.32 才能继续走。

FOR HUMANS · 给人看

fan.com — 一串好记的字母

你买的域名。本质上是一个别名 —— 没有实际的"位置"含义，电脑也不知道它指向谁。

人脑fan.com → "我的网站"
电脑fan.com → ??? 不知道
FOR MACHINES · 给电脑认

104.21.5.32 — 一组数字坐标

互联网上每台服务器都有一个IP 地址。这是真正的"门牌号"，电脑就靠它找路。

电脑104.21.5.32 → "知道往哪寄"
人脑104.21.5.32 → ??? 记不住

· 类比：你说"打给小明"，但手机只能拨号

小明→138-XXXX-1234
小红→139-XXXX-5678
fan.com→104.21.5.32
vercel.app→76.76.21.21
通讯录 = DNS。每次你按"小明"的名字拨号，手机先去通讯录查号 → 再拨数字。 浏览器访问 fan.com 也是 —— 先去 DNS 查 IP，再连那个 IP。
" 域名 = 便条上的名字，IP = 真实门牌号。 DNS 就是中间那本"名字 → 地址"的翻译手册。 —— 整篇文章的总开关
02 / DNS 查询

查一个 IP，
要问 4 次人。

浏览器拿到 fan.com 后，DNS 查询不是一步到位 —— 它像问路一样， 一层层往上问，直到找到那个知道答案的人。

玩法点右下"下一步"逐帧看 DNS 怎么从你的电脑出发，最后拿回 IP。每一步配一个城市/邮局类比。
  1. ①
    本机缓存 · browser cache
    浏览器先翻自己的小本本 → 没找过 fan.com
  2. ②
    本地 DNS · ISP / 路由器
    问运营商 → 运营商也没缓存，要去外面问
  3. ③
    根 DNS · root server
    问"全球总台" → .com 那条线归 X 服务器管
  4. ④
    .com TLD · 顶级域名服务器
    问 ".com 部门" → fan.com 归名为 NS1.fan.com 的服务器管
  5. ⑤
    NS · 权威服务器
    问 fan.com 自己的解析服务器 → A 记录 = 104.21.5.32
  6. ⑥
    回程缓存 · cache & return
    把 IP 沿路缓存回来 → 浏览器拿到 104.21.5.32 ✓
  7. ★
    完成 · 资料就位
    从此浏览器知道 fan.com → 104.21.5.32（缓存几小时～几天）
NARRATOR · 旁白
STEP 1 / 7

浏览器先翻自己的小本本

最快的办法 —— 看自己最近有没有访问过 fan.com。 访问过就有缓存，直接拿出 IP，跳过后面所有步骤。

类比 / Analogy 像你翻通讯录 —— 先在自己手机里搜"小明"，找到了直接拨号，没找到再问别人。
" 整套流程 10–200 ms 完成 —— 你按回车的瞬间，浏览器已经问完了 4 个人。 所以你察觉不到它的存在 —— 直到出错。 —— DNS 是隐形基础设施

01 · PROTOCOL

协议 · 用什么"语言"对话

http 是基础版，https 加了一层"防偷听"加密。 2026 年所有正经网站都必须 https —— 没有的话浏览器会贴红色警告。

设计师视角 装上 https = 浏览器地址栏出现小锁🔒。Vercel / Netlify 默认就给你装好了，不用自己折腾证书。
速记 协议 / 子域名 / 主域名 / 端口 / 路径 —— 5 段。每一段都可以独立配置、独立出错。
05 / 5 个坑

设计师常踩的
5 个坑。

买了域名、绑了 Vercel，按下回车依然报错 —— 90% 是这 5 个坑里的一个。 每条怎么诊断 + 怎么修 一句话讲清。

  • 01

    "我都买好域名了 —— 怎么打开还是显示买家？"

    买域名 ≠ 解析。买是说"这个名字归我"，解析才是说"这个名字指向哪台服务器"。 没配 A 记录 或 CNAME —— 域名就是个空壳。

    修：登录域名商（阿里云/Cloudflare/GoDaddy）→ DNS 设置 → 加一条 A 记录指向 Vercel/服务器的 IP。或者贴 Vercel 给的 CNAME。
  • 02

    "我配了，但现在还是没用？"

    DNS 改完不是立刻生效。全球各地的缓存（你电脑、ISP、运营商）要陆续过期 —— 这个延迟叫 TTL，通常 10 分钟到 24 小时。

    诊断：用 dnschecker.org 输入 fan.com，看全球各地是不是已经看到新 IP 了。修：等。换手机数据 / 用 cmd /flushdns 强清本地缓存。
  • 03

    "浏览器红警告：'不安全'"

    证书没装 / 装错了。HTTPS 不是默认，要给域名签一张"合法身份证"（SSL 证书），浏览器才认。

    修：用 Vercel/Netlify/Cloudflare 一类的现代平台 —— 它们自动装证书，你什么都不用做。自己买 VPS 的话，用 Let's Encrypt 免费证书，配 Caddy 一句话搞定。
  • 04

    "fan.com 能打开，www.fan.com 报错？"

    它们是两个不同的域名，要各自配解析。"裸域名"和"www 子域名"指向同一个 IP 不会自动同步。

    修：DNS 里加两条记录 —— 一条 @ 指向 IP，一条 www 指向 IP（或 CNAME 到主域名）。Vercel/Cloudflare 后台一般有"添加 www"按钮一键搞定。
  • 05

    "昨天还能打开，今天网站消失了？"

    域名过期没续费。域名是租的不是买断的 —— 大多数 1 年起，到期不续费会先警告几天，然后被回收。

    修：登录域名商续费。建议开自动续费。重要域名续 5 年起，不省那点钱。续费邮件务必看，丢了域名再抢回来非常贵。

· DNS 查询 · 7 步

步骤问谁
1 / 本机浏览器自己的缓存。最近访问过就直接出结果。
2 / 系统操作系统的 hosts 文件 + DNS 缓存。开发常用 hosts 改向。
3 / 本地运营商或 8.8.8.8 / 1.1.1.1 这种公共 DNS。
4 / 根全球 13 组根服务器，告诉你 .com 归谁管。
5 / TLD顶级域名服务器，比如 .com / .cn 各管一摊。
6 / NS权威服务器，存着 fan.com 的真实 A 记录。
7 / 缓存回程一路缓存，下次再访问就直接命中。
实际生活中 3 / 7 步常见，4–6 步只在大量缓存失效时才一路问到底。

· 域名 5 部分

段是什么
https://协议。带不带 s 决定要不要加密、有没有小锁。
www.子域名。可省略，可换成 blog. / app. / docs. 等等。
fan.com主域名。你买的那一段。.com / .cn / .io 是后缀。
:443端口。https 默认 443，http 默认 80，写不写都行。
/about路径。指向网站内的具体页面。可以无限叠加。
看到 ?key=value 别慌 —— 是查询参数，分享带的 utm 来源就是它。
