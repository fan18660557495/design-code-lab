03 / 拆开

3 个角色
怎么用这份 spec。

Spec 不是写给设计师看的——是写给工程 / QA / PM 看的。 点开每张卡片，看每个角色具体看哪几节、缺哪节会卡住、抱怨"spec 不全"时具体指什么。

ENG
视角 01 · 工程师

工程师怎么用

把 spec 当作"实现说明书"——从视觉到 schema 到状态机，每一步要落代码的东西，都能直接对位到 spec 里的一节。

▶展开看 6 项使用
开工时先翻这几节
  • 布局 / UI 组件实现2.1 + 5.1 + 5.2
  • 数据模型 / schema 落库3.1 + 3.2
  • 路由 / 页面状态机4.1 + 4.2
  • 权限 / RBAC 实现4.3
  • 错误 / 边界处理4.4 + 6.2
  • API 调用 / mock3.3 + 3.4
⚠ 工程师抱怨 "spec 不全" = 3.2 schema、3.4 API 契约、或 4.4 异常处理 任一节空白——他没办法独立开始写代码。
QA
视角 02 · 测试

QA 怎么用

把 spec 当作"测试大纲"——每一条 Given/When/Then 就是一条 case，每一个边界场景就是一条 negative case，每一项性能指标就是一道验收门。

▶展开看 5 项使用
开工时先翻这几节
  • 正向测试用例6.1
  • 负向 / 边界用例6.2 + 4.4
  • 自动化 mock 数据3.3 + 3.4
  • 性能 / 响应式验收5.4 + 6.3
  • 权限 / 角色覆盖4.3
⚠ QA 抱怨 "测不了" = 6.1 用户场景 或 6.2 边界场景 没写——只能去问产品 / 设计补，case 写不全。
PM
视角 03 · 产品

PM 怎么用

把 spec 当作"工期 / 取舍说明书"——组件复用度决定 UI 多少天、状态机决定逻辑多少天、扩展点决定能不能砍范围保 milestone。

▶展开看 5 项使用
开工时先翻这几节
  • 工期估算 · UI5.1 + 5.2
  • 工期估算 · 逻辑4.2 + 4.4
  • 范围取舍 · MVP vs 完整6.1 + 7
  • 沟通对齐 · 干系人1.1 + 1.2
  • 上线后跟进 · 指标1.3
⚠ PM 抱怨 "看不懂" = 1.1 业务目标 或 1.2 核心用户 没写——他无法对外讲"这玩意为啥要做"，也估不出 ROI。

⚠ 5 个常见陷阱（17 项之外的"暗坑"）

  1. 只发 Figma 链接——觉得"视觉清楚就够"。工程问 1 次 = 你回 3 句话 + 改 1 次稿 + 等 2 小时——还不如一次写清。
  2. empty 态让前端写——稿里只画了"有数据"那张，empty / loading / error 三态没画 → 工程师自己写"暂无数据" → 不一致。
  3. 权限"我口头说"——评审会上"客服看不到金额"——会议室出门就忘。3.1 角色矩阵必须写在文档里。
  4. 焦点 / tab 顺序不写——工程默认按 DOM 顺序，跳来跳去用户骂街。表单 spec 加 5.1 是 1 小时的事，省下来用户每次操作多 3 秒。
  5. 跳转不写回程——"+ 新建商机" 跳走了，保存后呢？回详情页 tab=opp 还是回列表页？不写 = 工程让用户回首页。

05 / 收尾

22 篇下来，
你长出了
一套语言。

高阶篇到此收尾。但语言一旦换了，团队跟着换——下一份真实需求，你就能感到差别。

i

从 "设计师" 长成 "产品架构师"

完成 22 篇后，你已经能从 5 个阶段（对象 / 流程 / 状态 / 组件 / 演进）+ 案例 3 部曲来看产品。每一篇都有可识别的 "理解时刻"——回去翻你的 5 个月前的稿子，会觉得不像自己画的。

ii

评审第一句话变了

下次需求评审，从 "我打算这么画" 变成 "这是个什么对象，它的角色矩阵 / 状态机 / 任务流 / 组件树各是怎样"——整套语言换了，整个团队跟着换。工程师听到这种讨论会眼睛亮。

iii

稿子从图变成了 schema

工程师不再追问，QA 能直接测试，PM 能直接排期。设计这个角色，从"装饰"晋升到"骨架"——你定义产品的结构，不只是它的样子。
