01 / 问题

"做完自己那块"
不等于系统一致。

设计师以为"做完自己那块就 OK"。其实系统级一致性 = 视觉 token / 共享组件 / 共享 formatter / 共享 validator—— 这 4 层都需要主动建立 + 维护，否则 6 个月后系统就裂开。

各做各的 · DRIFT

"我团队这块做完了"

A 团队的"客户状态"用 enum 红/绿/灰；B 团队的同一字段用 chip 蓝/橙/紫；C 团队的用图标。用户在系统里看 3 套语言。

6 个月后：同一份业务概念在产品里被画成 5 种、写成 5 种、算成 5 种。

用户感觉：这不是 1 个产品，是 5 个长得有点像的产品。

单一来源 · SOURCE

"客户状态"是一个原子

"客户状态"作为一个原子定义在 design system 里——所有页面用同一个 component + 同一个 token + 同一份文案。

漂移 = bug。任何页面画得不一样 = PR 被打回。

结果：1 个原子 1 处定义，N 个页面 1 套体验。

幕二 · 拆开
03 / 拆开

4 层一致性。

把"客户状态"统一，需要 4 层都对齐——只对齐颜色不够。 点每张卡片展开看每一层做了什么 + 不做会怎么裂。

1

设计 token颜色 / spacing / 圆角 / 字号

原子值——颜色、间距、字号、圆角这些"视觉常量"。统一这层，视觉漂移才能止住。

展开看反例 ›
是什么 定义一组原子值。例：color.status.active = #1F6B3A，color.status.dormant = #8A8A92。
落地 所有团队 import 同一个 token 文件——Figma 变量 + 代码 CSS 变量双绑定。
反例 颜色硬编码——A 用 #1F6B3A，B 用 #20A040。设计师视觉上认为一样，用户在两个页面切换时能感觉出"绿不一样"。
破窗效应：第一个团队硬编码颜色 → 第二个看到也觉得"我也可以" → 6 个月后 token 名存实亡。
2

共享组件StatusBadge / EmptyState / Pagination

把"客户状态显示"做成一个 React 组件，所有团队 import 同一个——不让每个团队各自画 chip。

展开看反例 ›
是什么 封装为 <StatusBadge value="active"/>，组件内部决定颜色 / 文案 / icon。
落地 在 design system 仓库发布；所有 app import 同一个版本；改动通过 PR + design review。
反例 每个团队自己用 <span> 画一遍 → 视觉漂、可访问性漂、交互态（hover / disabled）也漂。
破窗效应：组件不存在 → 大家各自画 → 即便后来发布了组件，老页面也不会迁移 → 系统始终有 2 套实现。
3

共享 formatter数字 / 日期 / 货币 / 状态文案

把"原始值 → 显示字符串"的映射放在一个 utils 包里。统一这层，文案漂移才能止住。

展开看反例 ›
是什么 全局唯一函数：formatStatus('active') = '活跃'，formatMoney(480000) = '¥ 48 万'。
落地 放在 @app/utils/format 包；i18n 文案 key 唯一；任何页面要展示状态都从这里读。
反例 每个页面写 if (status==='active') return '活跃' → 一个写"活跃"，另一个写"已激活"，第三个写"active"。
破窗效应：没有共享 formatter → 翻译团队改"活跃"→"已激活"时只能逐文件改 → 永远改不干净。
4

共享 validator业务规则 / 阈值

"什么算活跃"这条业务规则要全局唯一。统一这层，数据漂移才能止住。

展开看反例 ›
是什么 全局唯一函数：isActive(customer) = customer.lastContactAt > now - 30d。所有页面调它。
落地 放在 @app/domain/customer；后端 + 前端共用同一份；改阈值只改一处。
反例 A 团队定义"活跃 = < 7 天"，B 团队"< 30 天"，C 团队"< 90 天"。后台同一个客户，3 个页面里 3 种状态。
破窗效应：规则散落 → 运营问"为啥这个客户在 A 页是流失、在 B 页是活跃" → 没人答得上来。
4 层都对齐 = 系统稳。
任何一层裂 = 漂移连锁。

05 / 把这件事变成习惯

下次设计评审，
第一句先问 token / 组件。

从这一篇开始，"一致性"会变成一种工程师朋友会感谢你的设计语言。

i

一致性是基础设施不是装饰

token / 共享组件 / formatter / validator 是产品的脊柱。这 4 层不立，每个团队各自做加法 = 系统裂。

先建基础设施，再做新功能。

ii

设计稿要标"用了哪个 token / 哪个组件"

稿子里不标 = 工程师自己画一遍 = 漂移。每个元素都该有"来源"——这是 token 还是新原子？这是 StatusBadge 还是新组件？

不标 = 漂移的起点。

iii

允许"有意识的不一致"

通知里的 emoji 可能是产品决策。重点是"主动选择"——记录下来"这里我们故意和别处不同"，而不是"随手画"。

选择 ≠ 错误。漂移才是。

划重点 · TL;DR

一致性不是某个人的洁癖——是所有团队都需要的基础设施。视觉 token / 共享组件 / 共享 formatter / 共享 validator 这 4 层任何一层裂开，漂移就会从那里开始扩散。

下次设计评审，第一句问"这块用了 design system 的哪个组件 / token？"——只要这一句变成会议默认问题，一致性就开始自动维护了。
