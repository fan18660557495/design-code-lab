03 / 拆开

4 种异常的
诊断书。

每种异常都要回答 4 个问题：什么触发？谁负责？UI 怎样？怎么恢复？ 点卡片展开看 4 个槽位 + "不设计会怎样"。

失败 Failure

服务端不可用、网络断开——请求根本没有完成。设计的不是"出错文案"，是"用户的下一步"。

触发 · TRIGGER服务端 500 / 网络断 / API 不可达 / DNS 失败
负责 · OWNER工程 +/或运维（不应让用户承担责任）· 设计承接体验
UI · UI 兜底错误 toast / 对话框 + 保留用户输入 + 重试按钮 + 如果可降级则降级（缓存数据 / 离线模式）
恢复 · RECOVERY自动重试 1-2 次（指数退避）→ 仍失败则让用户决定（重试 / 取消 / 联系客服）
不设计这条 → 用户的订单数据全没了，再填一次再点提交，重蹈覆辙。
超时 Timeout

请求还在路上，但用户已经等不动了。"还在转吗"是设计要回答的最常见疑问。

触发 · TRIGGER接口慢于 3 秒（具体阈值看产品 · 移动网络更松）
负责 · OWNER工程优化（缩短）+ 设计承接（讲清楚状态、提供取消）
UI · UI 兜底> 1s 显示加载动画；> 3s 显示"还在加载……"文案 + 进度提示；> 10s 给"取消"按钮
恢复 · RECOVERY超时后自动取消请求 + 让用户重试 · 切忌静默挂起
不设计这条 → 用户狂点 / 关页面 / 投诉"卡死"，并触发"重复"异常。
冲突 Conflict

两个人同时改同一份资源——谁后保存谁覆盖。但"后改"未必是"对的"。

触发 · TRIGGER两人同时改同一资源，后改的会覆盖前改的（无锁 / 无版本号）
负责 · OWNER工程（乐观锁 / 悲观锁 / version 字段）+ 设计（冲突 UI 提示）
UI · UI 兜底保存时检测到"你不是最新版"——提示"小明 2 分钟前改过"，给用户三选一：看 diff / 强制覆盖 / 取消
恢复 · RECOVERY基于 diff 让用户合并；或先取消让用户重新载入再决定
不设计这条 → 客户电话一直打不通，业务部门和销售互相甩锅。
重复 Duplicate

用户多次点击 + 后端无幂等保护 = 数据被多次创建。客户端 + 后端要"两道防"。

触发 · TRIGGER用户多次点击（焦躁 / 网络慢 / 误触）+ 后端无幂等键
负责 · OWNER工程（幂等 idempotency-key）+ 设计（按钮禁用 + 视觉反馈 + 去抖）
UI · UI 兜底按钮 disabled / Loading 状态、视觉反馈（"提交中……"）、客户端去抖（debounce 500ms）
恢复 · RECOVERY后端检测到相同 key → 返回同一结果而非创建新单 · 前端提示"已有相同请求，请稍候"
不设计这条 → 一次点击产生 3 单 · 退款流程比下单流程长 10 倍 · 用户从此不信任系统。
4 种异常 × 4 个槽 = 16 个决策。
设计稿少 1 个，用户就少 1 份信任。

03.5 / 拆开

为什么错误态长得
都一样？

答案不在设计稿——在工程师怎么处理异常。 异常被吞 → 你只能画"出错了请重试"。异常被显式表达 → 你能画 7 张精准的错误态。

隐式 · IMPLICIT

throw / try-catch / 500 一兜了之

异常被吞，UI 拿不到原因
try { await submitOrder(data); } catch (e) { // 啥都不分，500 抛给前端 toast("出错了，请重试"); }

失败的真实原因丢失了——库存不足？支付被风控？地址超区？前端只知道"有错"。所以错误态长得都一样：一句"出错了"+ 一个重试按钮。

设计可做的 只能画 1 张通用错误态。看到 try-catch 把所有失败都 catch 成一句话——你的素材已经丢了。
显式 · EXPLICIT

typed Result<Ok, Err>

每种失败都有名字，调用方必须分支处理
type SubmitErr = | "OUT_OF_STOCK" | "PAYMENT_BLOCKED" | "ADDRESS_UNSUPPORTED" | "PROMO_EXPIRED"; const r = await submitOrder(data); if (r.ok) ... else switch (r.err) { ... }

每个失败有 typed reason。调用方拿到 `OUT_OF_STOCK` 还是 `PROMO_EXPIRED` 是两条完全不同的代码路径——也是两张完全不同的错误态。这是 result-ts / effect-ts 这类库的思路。

设计可做的 可以画 N 张错误态——每个 typed reason 对应一种 UI、一种"用户的下一步"。异常变成了交互的素材。
嵌套异常 · NESTED EXCEPTIONS

异常会沿着链路传递——传到 UI 时，
展开 or 折叠？

B 端流程里一个动作背后是一串调用。中间任何一步失败，最外层都看得到——但 UI 应该说哪一层的话？

失败 · 风控拦截
UI 应该说 "提交订单失败"（最外层）—— 还是 "支付被拦截"（最内层）—— 还是 "订单已暂存，支付未通过 · 请联系风控"（展开链路）？
三种说法对应三种用户下一步。设计要主动告诉工程师："这个错给我嵌套结构，别压平"——否则你拿到的永远是最外层那句空话。
交互上我们普遍缺"某件事是否能做 / 是否做成功 / 为什么没成"的表达。
背后是异常被弄丢了——设计要主动跟工程师讨"typed reason + 嵌套结构"。

04 / 看更多

「提交订单」的 4 种异常——
看人家怎么承接。

每种异常给你 触发 → UI 兜底 → 恢复策略 三栏，每一栏都讲明白为什么。

场景 01 · 失败

网络断，用户点提交订单

触发

用户填好订单 ¥1,280，点提交——家里网络断了 5 秒，请求没发出去。

UI 兜底

红条 toast："网络问题，已为你保留"。表单输入不清空，按钮恢复可点，重试 + 稍后再说 两个去向。

恢复策略

系统不要自动重试——用户在场要让他决定。自动重试可能在他切了 wifi 后误提交。用户主动重试最稳。

场景 02 · 超时

接口卡 8 秒

触发

用户点提交后，后端已经开始处理，但响应慢——8 秒还没回。用户开始坐立不安。

UI 兜底

分级提示：> 3s 显示"还在加载…" + 进度条。> 10s 给"取消"按钮。状态在变化，让用户知道系统活着。

恢复策略

超时后取消前端请求（释放 UI），但不能粗暴报错——后端可能成功了。提示："请求已超时，查一下我的订单"让用户自己核对。

场景 03 · 冲突

两个销售同时改电话

触发

销售 A 14:02 改了客户电话 → 销售 B 14:03 不知道也改了——版本号已经变。

UI 兜底

B 保存时弹"小明 2 分钟前改过" + diff 对照。三选一：合并 / 强制覆盖 / 取消。不要静默覆盖。

恢复策略

合并 = 让用户基于 diff 做决定。"强制覆盖" 必须留审计日志——A 销售周后能看到 B 覆盖了他的修改，才不至于背锅。

场景 04 · 重复

用户连点 3 次提交

触发

用户焦躁 / 网慢 / 误触，0.5 秒内点了 3 次提交按钮。

UI 兜底

第 1 次点 → 按钮立即 disabled + Loading + 文案变"提交中…"。第 2、3 次点击视觉无响应（系统识别但不发请求）。

恢复策略

客户端 + 后端双拦。客户端用 debounce + button disabled；后端用幂等键（idempotency-key）——同一个请求重复发，后端只生效一次。

05 / 把这件事变成习惯

下次画稿，
先问 3 个问题。

从这一篇开始，"我假设这不会发生"会从你的设计评审里逐渐消失。 异常 = 路径，不是 bug。

i

稿子至少画 4 张"异常态"

失败、超时、冲突、重复，每个一张——哪怕只是文案。

没画过 = 工程师靠猜 = 用户骂街。16 个状态都画清，工程师 0 追问。

ii

错误信息不是给用户看的——是让用户能行动

"网络异常" 不够 → "请检查网络后重试" 才行动；

"提交失败" 不够 → 保留输入 + 重试按钮 才行动。

iii

"我假设这不会发生"是危险句

网络会断、接口会慢、用户会连点、两个人会同时改。

规模够大时，每一种每天都是高频——千分之一也是上百次。

划重点 · TL;DR

异常 ≠ 错位，异常是路径。把失败 / 超时 / 冲突 / 重复 这 4 种当任务流的同等公民去画——稿子的鲁棒性就出来了。

更深一层——异常要 typed reason + 嵌套结构。看到工程师用 try-catch 一句"出错了"兜底，立刻喊停——那是你的素材在丢。

下次需求评审，试一次——"happy path 我画了，4 种异常各自的承接是这样，每种 reason 一个画面……"。工程师看你的眼神就变了。
