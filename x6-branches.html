任务流

01 / 问题

为什么你的流程图，
工程师总要追问"那个怎么办"？

因为你只画了"一切顺利"那条线。 真实任务有 4 种反向 + 重启的分支——任何一条没画，工程师就得猜。

直线思维 · LINEAR

"用户填表 → 提交 → 审批 → 完成"

稿子里画了 4 步直线，每步一张界面。看起来流程闭环了。

工程师追问：被驳回呢？用户改完重提是新单还是同一单？撤回是删数据还是改状态？审批人没回怎么办？提交完发现填错了能改吗？

你：呃……我没画这些。

树状思维 · BRANCH

"主流 4 步 + 4 个分支"

稿子里有 主流时间线，加上 4 个分支：驳回（回退到填表）、撤回（用户主动取消未审批的）、重提（被驳回后重新提交）、提前终止（用户或审批撤销整单）。每个分支带触发条件 + 状态迁移 + UI 入口。

工程师收到：完整流程图——直线主流 + 4 条支线，每条都有出口。

结果：追问 → 0 次。

03 / 拆开

4 个分支的
结构对称性。

每个分支都有 3 件事：触发方 + 状态迁移 + 重启入口。 把这张表填完，分支就跑通了。点任一行展开"如果稿子不画这条分支"的反例。

分支 触发方 主对象状态迁移 重启入口 ·
驳回 审批人 submitted → rejected → drafting 从原单"再次编辑"按钮 ▸
如果设计稿不画"驳回"这条分支—— 工程师只能猜：是新建一单？还是改原单？被驳的原因显示在哪？ 3 种猜法 3 种产品。
(a) 走"撤回"造假把驳回当撤回处理 → 申请历史污染。
(b) 让用户新建一单原单的审批记录断了 → 审计追不回。
(c) 干脆不允许重提申请人只能"放弃" → 体验崩塌。
撤回 申请人 submitted → withdrawn → drafting 从原单"撤回再编辑" ▸
如果设计稿不画"撤回"这条分支—— 工程师可能：
(a) 走"驳回"流程造假让申请人自驳？审批记录变假。
(b) 让用户"删除"单据污染审计 / 编号断层。
(c) 干脆禁止撤回用户提交后只能等审批人驳 → 反人性。
3 种走法 = 3 种产品——不画稿，3 选 1 就在工程那里发生了。
重提 申请人 rejected / drafting → submitted "重新提交"按钮 ▸
如果设计稿不画"重提"这条分支—— 被驳回的单据停在 rejected 状态没出口。用户被迫新建一单。
编号断裂同一件事变成两个 ID，审计/统计错乱。
审批历史丢失新单看不到上次为什么被驳——申请人会再犯同样的错。
审批人困惑"这个人怎么一周提了 3 次假？"
重提 = 驳回的反向出口。少画就死循环。
提前终止 双方 any → cancelled 不可重启（单据归档） ▸
如果设计稿不画"提前终止"这条分支—— 已通过的单据被现实推翻时（人调岗、时间冲突、决策反悔），系统里那条记录还在"生效"。
HR 系统不一致请假已批准但人已离职 → 工时统计错。
申请人手动改把原单时间改成历史日期"假装没请" → 审计造假。
开发临时加按钮"删除"按钮强删 → 没有归档证据。
cancelled 是一个独立的终态。不是 "rejected"，也不是 "completed"——它是"曾经成立但被撤销"。
分支 vs 状态 · 区分点

分支是流程图的边——从某个状态走另一条边到另一个状态。 状态迁移是节点之间的方向。 分支决定有几条路，状态迁移决定每条路怎么走。两者一起画，流程才闭合。

4 个分支 = 4 条新边 = 4 个新设计稿。
少画一条就少一种工程实现。

05 / 把这件事变成习惯

下次评审，
第一句问"被驳回会怎样"。

从这一篇开始，"直线思维"和"树状思维"的差距会出现在你每一次流程评审的第一句话里。

i

"被驳回怎么样"是评审第一句

任何提交流程评审，第一句问"被驳回会怎样、用户能不能撤回、撤回算谁的责任"。

少 1 句话 = 多 1 周返工。

ii

重启入口是分支的隐形要求

驳回/撤回后必须有"重新进入"的路径，否则单据死在某个状态里——你的产品突然就有了"幽灵单"。

每个分支都要问一句："用户怎么重启？"

iii

不是所有"转向"都是分支

"部分通过"是状态分裂、"暂存"是未提交、"草稿"是初始态。

分支是"从某状态走另一条边到另一状态"——其它都是其它结构。

划重点 · TL;DR

需求评审第一句问"被驳回会怎样"，第二句问"有几条分支"。直线 + 分支一起画的稿子，才叫任务流稿。

记得：分支不是 if-else 几句话。是流程图本身的结构——是边，是新设计稿，是新状态迁移。
