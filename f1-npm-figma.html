01 / 误解

你以为 vs
实际上

npm 是这套体系里最容易被误会的角色。先把三个最常见的误解摆出来 —— 看完这页，你不会再卡在这里。

YOU THINK · 你以为

"npm 就是一个网站，跟 GitHub 差不多"

看 AI 跟你说 "npm install xxx"，你可能下意识当它是某种 GitHub 命令。

npm ≈ "GitHub 的另一种说法"
node_modules → "看着像 bug，要不要删？"
install → "装完一次就好了吧"
ACTUALLY · 实际上

"npm 是一整套零件市场 + 配方 + 仓库"

三件东西要分开看：npm 是公共仓库（零件市场）；package.json 是你写的购物清单（配方）；node_modules 是已经买回家的零件柜。

npm = 全世界共用的零件市场
package.json = 你的购物清单 / 配方
node_modules = 真实的零件柜（1.2 GB 是它）
" 项目源代码常常只有 2 MB，但 node_modules 动不动就 1 GB。 不是文件失控，是这个生态本来就这么运转的。 —— 看完这页就懂了
02 / 配方表

package.json
是配方表。

每个项目根目录都有一份。它不写代码，写的是"这个项目叫什么、想要什么、怎么跑"。 相当于做菜前先列的清单。

互动左边是真实的 package.json，右边是讲解。点 / hover 任何一行，看它在干什么。
package.json 
{
  "name": "my-design-portfolio",
  "version": "0.1.0",
  "private": true,
  "scripts": {
    "dev": "next dev",
    "build": "next build",
    "start": "next start"
  },
  "dependencies": {
    "next": "15.0.2",
    "react": "19.0.0",
    "react-dom": "19.0.0"
  },
  "devDependencies": {
    "tailwindcss": "4.0.0",
    "typescript": "5.6.0"
  }
}
↑ POINT TO A LINE

name项目身份证

项目的英文名 —— 不光给你看，也是发包到 npm 仓库时的 ID。私人项目随便起，准备发布的话得是全网唯一。

类比 / Designer 视角

像 Figma 文件名 —— 你和别人都靠它指代这份东西。

注意

一行写错 —— 整个 npm install 直接报错。这文件是结构化的，不是注释。

三段重点 scripts = 项目能跑的命令； dependencies = 上线后必需的零件； devDependencies = 只在你开发时用、上线不带走（比如打包工具、TS 编译器）。

04 / 三条命令

最常用的三条
真实含义。

AI 给你的指令里 90% 都是这三条。 知道它们各自在你电脑上做了什么，沟通和排错都顺。

$ npm install

去市场取零件

读 package.json → 把所有零件下载到 node_modules/ → 同时拍一张进货单（package-lock.json）。

什么时候跑：第一次拿到代码 / 改了 package.json / 删过 node_modules 之后。
$ npm run dev

启动一台开发服务器

在你电脑上跑起一个本地小网站（通常 localhost:3000）。改代码自动刷新。别人看不到 —— 只有你自己能访问。

dev server · running
→ localhost:3000
什么时候跑：每次开始写代码 / 改设计的时候。关掉终端就停。
$ npm run build

打包成上线版本

把所有源码 + 图片 + 样式压缩、合并、优化 → 输出到 dist/ 或 .next/。这个文件夹才是真正能丢服务器的东西。

→
什么时候跑：准备上线前。本地开发用不到。
关键 install 是取零件、 dev 是试装样品、 build 是正式发货。 —— 跟买家具同一套逻辑。
05 / AI 协作

AI 让你 install
怎么看懂。

Claude / Cursor / GPT 给的安装指令，看着像咒语。 4 个最常碰到的问题 —— 各配一句话答案。

  • 01

    "npm install framer-motion" —— 这一句是什么意思？

    "去市场取一个新零件叫 framer-motion，加进我的清单"。 它会做两件事：node_modules/ 多一个新文件夹；package.json 的 dependencies 多一行。

    → 看完终端没报错就 OK，可以继续写代码。
  • 02

    "npm install -D @types/node" —— 那个 -D 是什么？

    -D = --save-dev，意思是装到 devDependencies 里 —— 只在开发时用，上线版本不带走。 常见的 -D 包：TypeScript 类型定义、ESLint、构建工具。

    → 看 AI 在哪一段叫你装。装组件库通常没 -D；装编译工具通常有 -D。直接照抄就行。
  • 03

    报错了 —— 第一步先看哪？

    不要先看 node_modules（看不出名堂）。先看 package.json —— AI 是不是把版本写错了？包名打错？ 然后再看终端最后红色那一段（不是中间，是最后），把它复制回去问 AI。

    → 90% 的报错都在终端最后 5 行。截屏给 AI。
  • 04

    "npm install 没用，要 rm -rf node_modules" —— 为什么？

    有时 node_modules 里某个零件状态损坏了 —— 半装一半、版本对不上、缓存出问题。这时候 install 不会自己修。 rm -rf node_modules = "把整个零件柜清空"，再 npm install = 重新进一遍货。

    → 这是终极重启大法。代价：要等几分钟重新下载。

· 术语对照表

术语是什么
package 一个零件。被打包好可以分享的代码。比如 react、tailwindcss。
module 一个 .js 文件。一个 package 通常含很多 module。
dependency 项目需要的零件。dependencies 字段里写的就是它。
devDep 只开发时用的零件 —— 上线版本不带它。
lockfile 进货单。锁死所有零件的精确版本，确保你和别人装出一样的东西。
看到 peerDependencies 别慌 —— 是"我要求你这边也得有 X"，进阶概念。先记前 5 个。

· 包管理器三兄弟

名字区别
npm 官方默认，跟 Node.js 一起装的。最常见。本文讲的就是它。
yarn 早期更快的替代品，命令几乎一样。yarn add = npm install。
pnpm 现在最流行 —— 共享零件柜，10 个项目只占 1 份硬盘。命令也类似。
bun 新选手，主打极快。Vibe coding 圈里越来越多人用。
区别 三者互通 —— 都读同一份 package.json。看到不同命令，对应换一下就行。
AI 用哪个就跟哪个走。不要混用（同一个项目里别一会儿 npm 一会儿 pnpm）。

· 你会在项目根目录看到的关键文件

文件名含义
package.json 配方表。手写。所有清单都在这。
package-lock.json 进货单。npm 自动生成。不要手改，但要提交到 git —— 别人才能装出一样的版本。
node_modules/ 零件柜。所有装好的零件都在这。不提交 git（太大，并且能从配方+进货单重建）。
.gitignore 里面通常第一行就写着 node_modules —— 告诉 git "这文件夹别管"。
yarn.lock / pnpm-lock.yaml 如果项目用的是 yarn 或 pnpm，进货单文件名就不一样。作用一样。
看到一个项目，先扫一眼 package.json 里的 dependencies 就大概知道这是个什么项目了 —— React、Vue、Next、还是后端。
