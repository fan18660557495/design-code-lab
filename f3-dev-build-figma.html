01 / 一份源码 · 两副面孔

同一份代码
跑出两份产物。

你硬盘里只有一份源码 —— 但跑出来的页面有两个版本。 一份给你写代码用，一份给真用户用。两边长得不一样、跑得不一样、大小也不一样。

关键左边是你每天看见的"工地"；右边是上线给用户的"成品"。从源码到产物中间，差着 build 这一步。
DEV · 开发模式

带辅助轮的"工地版"

你写代码时看见的版本 —— 啥都没压缩、啥都好读、出错时一目了然。

~ 500 KB
单个 page bundle · 含 source map
  • 文件原样 · 不压缩、不混淆，方便排错
  • 带 console.log · 你写的所有日志都在
  • 报错带行号 · 直接指到源文件第几行
  • 类名可读 · HomePage_button__abc12
  • 改一行就刷新 · 热更新（HMR）
BUILD · 上线版

压扁打包的"成品"

真正丢到服务器、给用户访问的版本 —— 体积小、跑得快、看不见过程。

~ 80 KB
同一个页面 · gzip 压缩后 ~ 30 KB
  • 压缩混淆 · 变量名变 a, b, c，去掉换行
  • console.log 全删 · 用户控制台干净
  • 报错只剩位置 hash · 看不出是源文件哪行
  • 类名 hash 化 · .a3f 这种 3 个字母
  • 改完要重新 build · 没法热更新
" 你看见的 localhost:3000 不是那个网站 —— 它只是同一份源码，被打开了"开发模式"的开关。 用户最终看见的是另一份。 —— 这一篇就是讲这条链
02 / dev 在做什么

改一行
页面秒级刷新。

npm run dev 启动了一台只在你电脑上跑的小服务器。 它一直盯着你的源码 —— 改一个字，对应那块 UI 立刻就变。这叫 HMR（Hot Module Replacement，热更新）。

玩法左边是代码，点任意一颗色块改主色 —— 右边浏览器会立刻跟着变。注意顶部 status 和右上角的 toast。
app/page.tsx 
// 你的源码
export default function HomePage() {
  return (
    <div>
      <h1>欢迎来到我的设计站</h1>
      <button style={{
        background: "#ff4d2e",
        color: "white",
        padding: "8px 14px"
      }}>
        立刻预约
      </button>
    </div>
  )
}

03 / 模式 swap 游戏

同一段代码
两副面孔。

左边的源码一字没动。 切顶部那个开关 —— 右边的渲染、控制台、文件大小、CSS 类名，全都跟着换一套人格。 这就是 dev 和 build 的区别。

玩法点上面的大开关切换 DEV / BUILD。看右下方 5 个属性的橙色脉冲 —— 每一项都被 build 改了。
app/HomePage.tsx · 源码不变 
// 这段代码两边完全一样 —— 变的是产物
import { useState } from 'react';
import styles from './HomePage.module.css';

export default function HomePage() {
  const [count, setCount] = useState(0);

  // 你随手写的调试日志
  console.log('HomePage 渲染了', count);

  if (count > 10) {
    console.warn('count 太大了');
  }

  return (
    <div className={styles.button}>
      <button onClick={() => setCount(count + 1)}>
        点击次数: {count}
      </button>
    </div>
  );
}

04 / build 三件事

build 不是"打包"
是三件事。

中文里"打包"两个字概括得太糙了。 npm run build 至少做三件不同性质的事 —— 各自能单独失败、各自影响结果。

步骤 01 · TRANSPILE / 转译

把写法翻成浏览器能懂的

你写的 .tsx（TypeScript + JSX）浏览器根本看不懂。 build 第一步是把它翻成普通的 .js。Tailwind 类名也是这步处理。

.tsx → .js
失败常见：TS 类型错、import 路径错。dev 时也会报，但 build 更严格。
步骤 02 · BUNDLE / 打包

几百个文件合成几个 chunk

你的项目可能有 500 个 .js 文件，但浏览器一个一个下太慢。 build 把相关的合并到几个大块（chunk），按页面切。

→
结果：访问首页只下载首页要的那个 chunk，不浪费用户流量。
步骤 03 · OPTIMIZE / 压扁

把每个 chunk挤干水分

压缩（minify）+ 摇树（tree-shake，删没用到的代码）。 一个 500 KB 的 dev bundle 经常能挤到 80 KB。

→
build · 84 KB
顺手做的：删 console.log、删注释、变量名换字母。前面提过的"两副面孔"就是这步干的。
关键 build 输出到 .next/ 或 dist/ 文件夹 —— 这个文件夹才是丢服务器的东西。 源码文件夹（app/、src/）服务器根本不需要。
05 / 部署后白屏

3 个最常见的
翻车现场。

本地一切正常，部署后白屏 / 报错 / 样式没了。 90% 的情况是这 3 条。每条都附诊断步骤，照着 1, 2, 3 走。

  • 01

    你只跑了 dev，上传了源码

    最常见的误会 —— 把 app/ 或 src/ 文件夹直接打了 zip 上传，以为这就是网站。 不是。服务器不会自己跑 npm run dev，它要的是 build 的产物。

    • 1 本地先跑 npm run build，等它真的跑完没报错。
    • 2 找输出文件夹 —— Next 是 .next/，Vite 是 dist/，CRA 是 build/。
    • 3 上传这个文件夹的内容（或用 Vercel/Netlify 这类自动 build 的平台，更省心）。
  • 02

    .env 没在线上，环境变量丢了

    本地有个 .env 文件存着 API key、数据库连接。 它不在 git 里（.gitignore 默认忽略），也就是说线上服务器看不到它 —— 代码跑起来一查没值，直接挂。

    • 1 打开本地 .env，把每一行都记下来（key + 值）。
    • 2 去部署平台（Vercel / Netlify / 你的服务器）后台，找 Environment Variables 设置面板。
    • 3 一行一行加进去，名字大小写要一致，加完重新 deploy 一次才生效。
  • 03

    代码里写死了 localhost:3000

    联调时图省事写了 fetch('http://localhost:3000/api/...')。 本地跑没事 —— 本地确实有 localhost:3000。线上跑用户的浏览器去访问用户自己的 localhost，谁家会跑你的服务？

    • 1 全文搜 localhost、127.0.0.1、:3000 —— 找出每一处。
    • 2 能用相对路径就改成 /api/...（同站请求自动用当前域名）。
    • 3 必须用绝对地址的，放进 .env，按 §02 的方法两边都配上 —— 本地填本地 URL，线上填线上 URL。
" 诊断顺序固定：先看是不是没 build，再看环境变量，最后查 localhost 残留。 照这三步走，半小时能把 80% 的白屏修掉。 —— 部署的 80/20 法则

· 4 条命令真实含义

命令做什么 / 什么时候用
npm run dev 开开发服务器。本地 watch 源码 + HMR + 友好报错。 → 写代码时
npm run build 生成上线产物。转译 + 打包 + 压缩，输出到 .next/ 或 dist/。 → 准备上线 / 测真实大小
npm run start 跑刚 build 出来的产物。在本地模拟一下"线上"的样子。必须先 build。 → 上线前最后一次自测
npm run preview Vite 项目里 start 的别名。作用一样。 → 看你的 package.json 写哪个
dev 和 start跑出来都是 localhost:3000，但跑的是不同东西 —— dev 是源码 + 辅助轮，start 是 build 产物的本地预览。

· 上线前自检 3 步（点一下打勾）

  • 本地跑 npm run build —— 全程没红字才算过 报警告（warning）通常不致命，但红色 error 必须先修。
  • 再跑 npm run start 在本地访问 —— 比 dev 更接近线上 这一步能抓出"dev 时正常、build 后挂"的 90% 问题。
  • 全文搜过 localhost、127.0.0.1 —— 没硬编码地址 每一处都要么改相对路径，要么用环境变量。
