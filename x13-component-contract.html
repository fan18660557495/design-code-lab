03 / 拆开

契约 4 件套。

Props（接入数据）/ Events（发出反馈）/ Slots（可插内容）/ State boundary（谁管哪部分状态）。 点开任意一张卡片看深度解释和典型反例。

P

Props接入数据 / 组件的输入端

+

组件从外面接受的输入。决定它长什么样、处于什么模式。

常见例子

variant（外观风格）/ size（尺寸）/ disabled（禁用与否）/ value（受控值）/ placeholder（占位文案）

反例 · 模糊的设计稿

稿子写"button has hover state"——但没说 hover 是prop 触发还是CSS 触发。工程师可能写一个无意义的 hovered prop（永远没人传），也可能写 CSS（外部代码无法编程触发）。

工程上看这件事
// 好的 props 就像参数列表 type ButtonProps = { variant: 'primary' | 'secondary'; disabled?: boolean; // 由外部传，组件不自管 };
E

Events发出反馈 / 组件的输出端

+

组件向外发出的通知。用户做了什么动作，组件喊一声让外面知道。

常见例子

onClick(e, value) / onChange(newValue) / onFocus() / onClose() / onSelect(item)。命名规则：on* 前缀。

反例 · 副作用归属不清

设计稿："按了按钮之后弹出 modal"——但没说是按钮内部直接打开 modal（耦合），还是按钮发 onClick、外部决定要不要弹（解耦）。工程师只能猜——猜错就埋了一个耦合 bug。

工程上看这件事
// 组件只负责"喊一声" onClick: (e: MouseEvent) => void; onClose: () => void; // closable 时必填
S

Slots可插内容 / 组件留的"洞"

+

组件留给消费方的"洞"——这里要塞什么内容，组件不规定。

常见例子

children（主内容）/ leftIcon、rightIcon（前后图标）/ trailing（行尾区）/ header、footer（页头页脚）

反例 · prop / slot 没分清

设计稿画了一张 Card：里头有标题、副标题、图标、操作按钮。工程师懵——哪些是prop（结构固定，只能传文字/值）哪些是slot（自由，能塞 ReactNode）？只画图不能区分。

工程上看这件事
// 给 slot 等于"这块内容我不管" title: string; // prop, 固定为文字 trailing?: ReactNode; // slot, 你想塞啥都行
B

State boundary状态边界 / 谁管哪部分

+

组件内部管什么、外部管什么。受控 vs 非受控的根本问题。

两种模式

受控：value + onChange——值住在外面，组件只是渲染。
非受控：defaultValue——值住在组件里，外部不管。

反例 · 默认值没说清归属

设计稿画"Toggle 默认是关闭"——没说是受控（每次都要消费方告诉它当前是开还是关）还是非受控（组件自己记着状态）。工程师猜——猜错就出现"为什么我设的值不生效"这种 bug。

工程上看这件事
// 受控 value: boolean; onChange: (v: boolean) => void; // 非受控 defaultValue?: boolean; // 只初始化一次

04 / 看更多

完整的 Button 契约——
看人家是怎么填的。

把 props / events / slots 全部预填好——重点看 ⚠ 标的几条，是设计师最容易踩的坑。

Button 组件 · 完整契约

这就是 Section 01 那张合同的展开版——每一条都附"为什么"。

PROPS 能从外面接什么 7 条
variant : 'primary' | 'secondary' | 'danger' 颜色风格——决定主次。由意图驱动，不是 bgColor。
size : 'sm' | 'md' | 'lg' 3 档枚举——padding / font-size 都跟着 size 推导，不要单独传字号。
disabled ?: boolean 由外部传——组件不自管"启用 / 禁用"。
loading ?: boolean 异步态——传 true 自动转圈 + 阻止点击。设计稿里"灰着 + 转圈"那张稿就是这一条。
leftIcon ?: ReactNode 前置图标——slot 不是 prop，外部塞任意 React 节点。
rightIcon ?: ReactNode 后置图标——通常是 "→" 或 caret。
type ?: 'button' | 'submit' | 'reset' 原生 HTML 类型——表单里要用 submit。默认 button，避免意外 submit。
EVENTS 会向外发什么 3 条
onClick : (e: MouseEvent) => void 必填——按钮的灵魂。命名固定 onClick——不叫 onPress / handleClick / submit。
onFocus ?: (e) => void 可选——只有需要追踪键盘导航时才传。
onBlur ?: (e) => void 可选——失焦回调，少用。
SLOTS 中间留口给消费方 1 条
children : ReactNode 按钮里的文字——不是 label: string prop。给 slot 等于"内容你管，我只管壳"。
不该是 PROPS 这些放进去，组件就坏 反向清单
fontSize : number 字号应该由 size 推导，不该独立传。否则一份组件可以画出 100 种字号的奇怪 Button。
bgColor : string 颜色由 variant 决定。直接传色值 = 把设计系统拆穿了——再也回不去统一了。
paddingX / paddingY : number 间距由 size 推导。任何能从其他 prop 推出来的值都不该独立做 prop。
hovered : boolean hover 是 CSS :hover 自动触发的——不是 prop。设计稿里画的 hover 态是给 CSS 看，不是给 props 看。
⚠ 把 onClick 改名 很多设计稿写"点击 → handleClick" / "onPress" / "submit"——命名不一致工程师就要在每个组件查文档。on* 是行业惯例，onClick / onClose / onChange 永远叫这个。
⚠ 把 children 写成 label 稿子标注 "label: 提交"——工程师只能传字符串，传不进 <Icon /> + 文字。改成 children: ReactNode，立刻能塞图标 + 文字 + emoji。
⚠ 把 variant 拆成 isDanger / isPrimary 看起来直观——但两个 boolean 能勾出 4 种组合（包含"既 primary 又 danger"这种不合法态）。enum 强制单选，永远只有 3 种合法态。
⚠ 把 disabled 和 loading 混用 它们是两件事——loading 是"按钮自己在干活"（异步），disabled 是"外部说你不能点"（权限 / 校验）。UI 表现一样但语义不同，两个 prop 都要留。
05 / 把这件事变成习惯

画组件之前，
先写合同。

从这一篇开始，做组件评审时，第一句话从"我打算这么画"变成"这组件的契约是什么"。整个会议的语言就变了。

i

画组件之前先写 props / events / slots

一句话写不出契约的组件，证明你还没想清楚它要做什么。先把这三件套口头描述一遍——卡住的那一项，就是含糊的地方。

ii

slot 是"信任"的设计

给一个 slot 等于把这部分内容主权交给消费方。给不给 slot，是关键设计决策——不是技术细节。

iii

受控 vs 非受控不是工程细节

它决定了组件能不能被外部驱动。设计稿要明说——表单组件几乎都要受控，展示组件几乎都无状态。中间地带最容易出 bug。

划重点 · TL;DR

组件交付不是图，是合同。下次评审，第一句问"这组件的契约是什么"——能讲清 4 件套的，是组件设计师；只能讲清图层的，是图层堆叠师。

Figma 里的视觉差异 → 都能映射回一条契约。一旦你开始用契约描述组件，工程师追问 0 次，复用率反着涨。
