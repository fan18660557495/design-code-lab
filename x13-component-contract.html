组件不是
图层——
是一份契约。

Figma 里一个组件 = 一组图层 + variants。

01 / 问题

为什么"看起来一样"的组件，
到了代码里不能复用？

因为设计师在描述视觉单元（图层 + 变体）， 工程师在写一份合同。可复用与好不好看无关——只看契约稳不稳。

图层思维 · LAYERS

"我做了一个 Button 组件"

5 个 variant、3 个 size、有 hover 态。看起来完整——可工程师还在追问。

工程师追问：禁用怎么写？loading 态有吗？图标 + 文字怎么传？点击事件叫什么？是不是受控？icon 自由插还是固定 prop？

你：呃……我画了视觉，
没写"它能接什么"。

"我交付了一份 Button 契约"

视觉只是合同附录。真正交付的是一份 props / events / slots / state 清单，工程师直接落 TypeScript interface。

幕二 · 拆开
03 / 拆开

契约 4 件套。

Props（接入数据）/ Events（发出反馈）/ Slots（可插内容）/ State boundary（谁管哪部分状态）。 点开任意一张卡片看深度解释和典型反例。

P

Props接入数据 / 组件的输入端

+

组件从外面接受的输入。决定它长什么样、处于什么模式。

常见例子

variant（外观风格）/ size（尺寸）/ disabled（禁用与否）/ value（受控值）/ placeholder（占位文案）

反例 · 模糊的设计稿

稿子写"button has hover state"——但没说 hover 是prop 触发还是CSS 触发。工程师可能写一个无意义的 hovered prop（永远没人传），也可能写 CSS（外部代码无法编程触发）。

工程上看这件事
// 好的 props 就像参数列表 type ButtonProps = { variant: 'primary' | 'secondary'; disabled?: boolean; // 由外部传，组件不自管 };
E

Events发出反馈 / 组件的输出端

+

组件向外发出的通知。用户做了什么动作，组件喊一声让外面知道。

常见例子

onClick(e, value) / onChange(newValue) / onFocus() / onClose() / onSelect(item)。命名规则：on* 前缀。

反例 · 副作用归属不清

设计稿："按了按钮之后弹出 modal"——但没说是按钮内部直接打开 modal（耦合），还是按钮发 onClick、外部决定要不要弹（解耦）。工程师只能猜——猜错就埋了一个耦合 bug。

工程上看这件事
// 组件只负责"喊一声" onClick: (e: MouseEvent) => void; onClose: () => void; // closable 时必填
S

Slots可插内容 / 组件留的"洞"

+

组件留给消费方的"洞"——这里要塞什么内容，组件不规定。

常见例子

children（主内容）/ leftIcon、rightIcon（前后图标）/ trailing（行尾区）/ header、footer（页头页脚）

反例 · prop / slot 没分清

设计稿画了一张 Card：里头有标题、副标题、图标、操作按钮。工程师懵——哪些是prop（结构固定，只能传文字/值）哪些是slot（自由，能塞 ReactNode）？只画图不能区分。

工程上看这件事
// 给 slot 等于"这块内容我不管" title: string; // prop, 固定为文字 trailing?: ReactNode; // slot, 你想塞啥都行
B

State boundary状态边界 / 谁管哪部分

+

组件内部管什么、外部管什么。受控 vs 非受控的根本问题。

两种模式

受控：value + onChange——值住在外面，组件只是渲染。
非受控：defaultValue——值住在组件里，外部不管。

反例 · 默认值没说清归属

设计稿画"Toggle 默认是关闭"——没说是受控（每次都要消费方告诉它当前是开还是关）还是非受控（组件自己记着状态）。工程师猜——猜错就出现"为什么我设的值不生效"这种 bug。

工程上看这件事
// 受控 value: boolean; onChange: (v: boolean) => void; // 非受控 defaultValue?: boolean; // 只初始化一次

05 / 把这件事变成习惯

画组件之前，
先写合同。

从这一篇开始，做组件评审时，第一句话从"我打算这么画"变成"这组件的契约是什么"。整个会议的语言就变了。

i

画组件之前先写 props / events / slots

一句话写不出契约的组件，证明你还没想清楚它要做什么。先把这三件套口头描述一遍——卡住的那一项，就是含糊的地方。

ii

slot 是"信任"的设计

给一个 slot 等于把这部分内容主权交给消费方。给不给 slot，是关键设计决策——不是技术细节。

iii

受控 vs 非受控不是工程细节

它决定了组件能不能被外部驱动。设计稿要明说——表单组件几乎都要受控，展示组件几乎都无状态。中间地带最容易出 bug。

划重点 · TL;DR

组件交付不是图，是合同。下次评审，第一句问"这组件的契约是什么"——能讲清 4 件套的，是组件设计师；只能讲清图层的，是图层堆叠师。

Figma 里的视觉差异 → 都能映射回一条契约。一旦你开始用契约描述组件，工程师追问 0 次，复用率反着涨。
