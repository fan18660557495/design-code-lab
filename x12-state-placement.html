01 / 问题

设计稿里"状态都标了"，
但没人知道它住在哪。

设计师以为状态就是"按钮高亮/不高亮"。其实状态有 4 种住址——住址决定它在刷新、分享、跨设备、关 tab 时怎么表现。 住址不写清楚 = 行为靠工程师猜。

把状态当装饰 · BAD

"筛选 / 弹窗 / tab 都标了高亮"

稿子里这三件状态都画了——筛选条选中态、弹窗打开态、当前 tab 高亮态。看起来完整。

工程师追问：刷新页面，筛选会保留吗？把当前 URL 发给同事，他打开能看到一样的 tab 吗？关掉弹窗下次再开，是新的还是上次那个？

你：呃……我没想。

状态有住址 · GOOD

"筛选住 URL，弹窗住组件，tab 住本页"

稿子里每个状态都标了住址——筛选住 URL（可分享 + 刷新保留）/ 弹窗住组件（关掉就重置）/ tab 住本页（页内共享，刷新还原）。

工程师收到：三件状态各有明确"住址 + 行为"，直接落 router + state schema。

结果：追问 → 0 次。

每种住址都有 4 个特征——刷新行为 / 分享行为 / 跨设备 / 持久度，加上典型用例。 点每一行展开看技术承载、设计反例、选择信号。

住址 刷新 分享 跨设备 持久 典型用例
组件内+ ✗ ✗ ✗ 关闭即灭 弹窗开 / 关、tooltip、hover、菜单展开
技术承载 · How
useState · ref · 本地实例变量

状态住在组件实例里。组件 unmount 就消失。
React 里就是 useState / useReducer；Vue 里是 ref / reactive。

设计反例 · Bad
把"当前筛选"塞进组件里

同事打开你分享的列表 URL，看到的是没筛选过的全表。因为筛选状态死在你的组件实例里，URL 里没带——根本传不出去。

选择信号 · When
瞬态 + 私有

只有当前组件用 + 关掉就该丢 → 选这个。Hover 高亮、dropdown 开合、表单字段聚焦——典型组件内状态。
本页+ local: ✓ / mem: ✗ ✗ ✗ 设备本地 侧栏折叠、主题、字体大小、列表密度
技术承载 · How
localStorage / Context / Memory

跨组件、跨页面共享，但只活在当前设备 / 浏览器。
localStorage（刷新保留）/ Context Provider（仅内存，刷新清空）。

设计反例 · Bad
主题设置只住本页

用户换台电脑/手机登录，主题、字号、密度全部重置。如果"用户偏好要跟人走"，就该住服务端，不是 localStorage。

选择信号 · When
本设备私有偏好

跨页面共享 + 但不想跨设备 + 不想分享。布局密度、最近打开过的项、本机草稿这种"这台设备上我喜欢这样"——选这个。
URL+ ✓ ✓ ✓ (链接) URL 有效期 筛选、排序、当前 tab、当前页码、搜索词
技术承载 · How
URLSearchParams / path / hash

状态序列化进 URL。三种位置：?key=val（query）/ /path/x（path）/ #anchor（hash）。
路由库 (React Router / Next router) 提供读写 API。

设计反例 · Bad
把模态框开 / 关塞进 URL

用户后退 → 模态自动重开；再前进 → 又关掉。浏览器历史里一堆"打开 / 关闭"的脏数据。模态框是瞬态 UI，不该出现在历史里。

选择信号 · When
想分享 / 刷新 / 后退能回来

用户希望：把当前视图发给同事、刷新不丢、后退退回上一次的视图。一句话——"如果丢了用户会骂"的状态，URL 是默认住址。
服务端+ ✓ ✓ ✓ 数据库永久 草稿、未读、用户偏好、订阅、协作状态
技术承载 · How
DB / API / Sync 协议

状态写进数据库，前端只是"展示 + 同步"。
典型：autosave 草稿 / 用户 settings 表 / WebSocket 实时同步。

设计反例 · Bad
未登录用户的主题塞服务端

用户没账号 → 没地方存。强制登录才让选主题——劝退体验。先在 localStorage 起，登录后 sync 到服务端，才是渐进路径。

选择信号 · When
必须跨设备 / 协作

"换设备能续上"、"团队里别人能看到"、"不能丢一个字"——三个里有一个，就该住服务端。代价是接口 / 同步 / 冲突，不是 default，是 last resort。
反例 01
筛选住组件 → 链接传不出去

用户分享当前列表链接给同事，同事打开看不到一样的筛选——因为筛选状态死在你组件里，URL 干干净净。

反例 02
模态框住 URL → 后退诡异

退回去 → 模态自动重开；再前进 → 又关掉。浏览器历史变成"开关弹窗"的日志。模态是瞬态 UI，不进历史。

反例 03
主题住服务端 → 未登录无解

新用户没账号 → 没法存偏好 → 必须先注册才能换暗色模式。渐进式应该是 localStorage 起 → 登录后 sync。

4 种住址 = 4 种产品行为。
设计住址 = 设计行为。状态写在哪 = 它能不能被分享、能不能跨设备、能不能后退。

05 / 把这件事变成习惯

下次评审，
每个状态先问"住哪"。

住址不是技术细节——它就是用户行为。能不能分享、能不能后退、能不能跨设备——全由住址决定。

i

先问"住哪"再问"长啥样"

设计稿里每个有状态的元素，标上住址——@url / @component。

工程师拿到稿就能落 schema，不用追问。

ii

URL 是设计语言

凡是"用户希望分享 / 刷新保留 / 后退能回来"的状态——URL 是默认住址。

筛选、排序、tab、页码、搜索词……能进 URL 的尽量进 URL。

iii

服务端不是 default

服务端持久代价高——接口 / 同步 / 冲突 / 未登录怎么办。

只有"必须跨设备 / 协作 / 不能丢"的状态才该住服务端。其余先从本页或 URL 起。

划重点 · TL;DR

状态有 4 种住址——组件 / 本页 / URL / 服务端。住址决定它在刷新、分享、跨设备、关 tab 时的行为。

下次评审，问每个状态"住哪"——4 个字省 10 次追问。设计稿里写"@url / @component"，比"高亮态 / 普通态"信息量大 10 倍。
