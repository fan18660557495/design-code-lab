01 / 一次请求长啥样

浏览器和服务器，
一来一回。

每一次"页面上出现新东西" —— 用户名、商品、头像 —— 背后都是这两个动作：浏览器问、服务器答。点中间的箭头看每一段是什么。

互动点中间两个长条："去"是请求，"回"是响应。每点一个，下面对应的小卡会高亮。这是整篇文章的基础模型。
CLIENT · 浏览器

你的电脑

页面在这里。设计师每天打开 F12 那个 Chrome / Safari，就是它。

localhost:5173 / your-app.com

① 请求 · REQUEST
GET/api/user/cart
Cookie · Authorization · User-Agent
"我是 Fan，能不能给我看看我购物车有啥？"
② 响应 · RESPONSE
200 OKapplication/json
{"data":{"cart":[...]}}
"我是后端，给你 → 这是你的购物车"
SERVER · 后端

服务器

不在你电脑上。在某个机房 / AWS / 你公司的后端工程师那里。它有数据库。

api.your-app.com

A · 方法 / METHOD
GET / POST

GET 是"取" —— 去拿数据。POST 是"送" —— 提交表单 / 创建订单。还有 PUT（改）、DELETE（删）。

B · 地址 / URL
/api/...

问的是哪扇门。比如 /api/user/cart 就是"用户的购物车那扇门"，/api/login 是"登录那扇门"。

C · 头 / HEADERS
Cookie · Auth

夹在请求里的"身份证"。常见的有 Cookie（你登录后浏览器存的小纸条）、Authorization（token），告诉服务器"我是谁"。

D · 响应体 / BODY
JSON 数据

服务器回的实质内容。99% 是一坨 JSON：{ key: value } 嵌套。页面上看得见的内容（用户名、价格…）就是从这里读出来的。

心智每次页面"出新数据" —— 不是页面"自动长出来"，而是有一次完整的"问 → 答"。少了任何一步，页面就空 / 错。Network 面板，就是把这一来一回每次都摊在桌面上。
02 / 状态码

数字背后
是三种心情

服务器回的每条响应，都带一个 3 位数。叫 status code。看头一位就知道这次顺没顺：2 顺、4 是你错、5 是它挂。

2xx · GREEN

2xx 顺

成功 · 拿到了

服务器说："收到了，给你"。看到 2 开头你就放心 —— 这条请求本身没问题。

200
OK万事顺利。最常见的成功码。
201
Created新东西创建成功。注册新账号、发了一条新评论 → 通常回 201。
204
No Content成功了，但没有数据要返给你。删除操作常见。
304
Not Modified3 开头是"重定向 / 缓存"。304 = "你之前缓存过，没变，直接用本地的"。看到不慌。
4xx · ORANGE

4xx 你错了

前端 · 请求不对

问题在你（或调用接口的代码）这边。要么没登录、要么地址写错、要么没权限。需要你 / 工程师改前端。

400
Bad Request"你发的东西我看不懂" —— 通常是参数缺了或格式错了。
401
Unauthorized"你是谁？没登录" —— token 过期 / 没传。设计师最常撞这一个。
403
Forbidden"知道你是谁，但你没权限" —— 比如普通用户访问管理员接口。
404
Not Found"这个地址我不认识" —— URL 拼错 / 接口名变了 / 路径变了。
5xx · RED

5xx 它挂了

后端 · 服务器自己出问题

问题在服务器。请求是对的，但它内部出毛病了 —— 数据库挂了、代码崩了、超时了。找后端工程师，跟你没关系。

500
Internal Server Error"我（服务器）自己出 bug 了" —— 一般后端代码报错。最常见的 5xx。
502
Bad Gateway"中间环节坏了" —— 比如负载均衡 / 网关在跟真的服务器之间断了。
503
Service Unavailable"我现在挤不动了" —— 服务器在维护或被打爆了。
504
Gateway Timeout"等太久了" —— 后端处理超时。可能是数据库慢爆。
一句话记法
看到红字 / 橙字 → 先看头一位数字。2 顺 不用管；4 你错（前端 / 401 没登录最常见）；5 它挂（后端的事）。
设计师只要做一件事：把这个数字截图给工程师。其他他们一看就知道。

04 / Response 怎么读

服务器吐回来的
那坨 JSON

点开任何一条 API 请求 → 切到 Response 标签，你会看到一坨像下面这样的东西。它叫 JSON，是后端跟前端说话的"标准格式"。把鼠标 hover 到任何一个 key 上，看右边的解释。

互动下面是 /api/user/me 的真实响应。移动鼠标到不同的字段上（colored 的 key 都可以点）—— 右边会换上对应的解释。注意那个橙色的 [] —— 这就是设计师追的"购物车为啥是空"的元凶。
200 OK application/json
242 B · 84 ms
{ "status": "success", "message": "User profile loaded", "data": { "user": { "id": 1042, "name": "Fan", "email": "fan@example.com", "avatar": "https://cdn.../u1042.jpg", "verified": true }, "cart": [], "orders": [ { "id": "o-9981", "total": 128.00 }, { "id": "o-9994", "total": 59.50 } ], "lastSeen": null } }
— 等你 hover

把鼠标移到任何一个字段上

JSON 的每一层 key 都有意义。这边会告诉你：这个字段是什么、设计师能从它读出什么、什么时候它"看上去对，其实是 bug"。

提示：那个橙色的 [] 特别值得点一下 —— 90% 的"页面空白" bug 都长这样。

顶层 · TOP-LEVEL

"status" — 这次响应整体顺不顺

注意：这跟 HTTP 状态码不是一回事。HTTP 200 是"网络层成功"；这里的 "status" 是业务层成功 —— 后端工程师约定的字段，告诉前端"逻辑也通了"。常见值："success" / "error" / "fail"。

坑有些后端会回 HTTP 200，但 status: "error"。一看 200 就以为没事是错的 —— 还要看这个字段。
顶层 · TOP-LEVEL

"message" — 给人看的话

一句人话，多用于报错。"Invalid email" / "User not found" / "操作成功"。设计师可以直接把这句话当 toast 提示文案。

顶层 · TOP-LEVEL

"data" — 真正的数据在这里

大多数后端会把"实际数据"包在一层 data 里 —— 这样 status 和 data 不会撞名。设计师 80% 的时间要看的，都是这一坨。

嵌套对象 · OBJECT

"user" — 用户对象

大括号 {} 表示一个对象。里面是这个用户的全部字段。
页面上"小头像 + 名字 + 蓝色对勾"那块 UI，就是从这里读出来的。

字段 · FIELD

"id" — 用户唯一编号

每个用户在数据库里都有一个 id。设计师一般不显示它，但"删除 / 编辑 / 跳到详情"这些操作背后都靠这个。看到 1042 就是数字编号。

字段 · FIELD

"name" — 显示名

页面右上角那个名字。设计师能问的：是不是 null？是不是空字符串？是不是用户没填过的默认值（"User_xxx"）？ 这三种情况 UI 应该长不一样。

字段 · FIELD

"email" — 邮箱

注意敏感性。前端有时候会显示打码版（f***@example.com），那是前端处理出来的，不是后端给的。后端会给完整的，前端要负责掩码。

字段 · FIELD

"avatar" — 头像 URL

是个网址字符串。头像不出来的常见原因：① 这个 URL 是 null / 空 → 页面要 fallback 到默认头像；② URL 有但加载 404（去 Img 列单独看）。

字段 · FIELD · BOOLEAN

"verified" — 真 / 假

true / false 是布尔值。两种状态。设计师最常用它来决定：显不显示那个蓝勾对号、能不能进某个页面。

嵌套数组 · ARRAY

"cart" — 购物车列表

方括号 [] 表示一个数组（list）。里面是 0 个、1 个、N 个对象。页面上"购物车里有 3 件商品"那块 UI，就是 cart 数组的长度 + 每一项渲染。

★ 关键 · BUG 候选

空数组 [] —— 是空的

设计师追的"购物车永远空" —— 大概率长这样。

它不是 bug（请求成功了），但页面看上去就像 bug。可能性：
① 这个用户真的购物车空 → 设计要做 empty state；
② 后端没把购物车关联上 → 找后端；
③ token 错了，拿到的是别人的（空）账户 → 找 401 类问题。

区分[] = 数组但没东西。null = 完全没有这个字段。两个跟设计页面要做的逻辑完全不一样。
嵌套数组 · ARRAY

"orders" — 订单列表

这次有 2 条。每条又是一个对象（{ id, total }）。数组里嵌对象是最常见的结构 —— 列表页 / 表格页基本都长这样。

字段 · FIELD

"id" — 订单号

跟 user.id 一样的概念，每个订单的唯一编号。这次是字符串（"o-9981"），跟 user.id 是数字不一样。id 不一定是数字，有时候后端用字符串。

字段 · FIELD · NUMBER

"total" — 订单金额

注意单位。这里是 128.00 —— 是 128 块，还是 128 分？这要看后端文档。常见的坑：人民币用分存（12800），但前端不知道，直接显示 12800 元。

特殊值 · NULL

null — 完全没有这个值

null 不是字符串 "null"，是"这里啥都没有"的特殊值。比如这个用户从来没上线过，所以 lastSeen 是 null。设计要决定：null 时显示啥？—— 通常是占位符 "-" / "暂无" / 隐藏整行。

设计师能读出的① 字段是不是缺了？ ② 是不是 null / 空数组 / 0？ ③ 数字单位对不对？ ④ "status": "error" 时 message 写的什么。这 4 件事不需要懂代码，凭眼睛读 JSON 就能发现 80% 的"页面不对劲"。
05 / 5 个技能

设计师最该
掌握的五件事

Network 面板按钮一堆。设计师不需要全会。会下面 5 件事，已经能自己定位 80% 的"页面没数据"问题。

1

打开 Network 面板

F12 → 切到 Network 标签

右键页面 → "检查"，或者直接 F12（Mac 是 ⌥⌘I）。打开后顶上一排 tab —— Elements / Console / Network / Sources / Application。点 Network。

⌥ ⌘ I Mac
F12 Win
Elements Console Network Sources
2

刷新一次，看所有请求

Cmd / Ctrl + R

Network 面板要在"录制"状态下才能抓请求 —— 看左上角那个 红色小圆点，亮着就是在录。然后刷新页面，所有请求都会蹦出来排成表。

REC ⊘ clear preserve log
⌘ R → 所有请求出现
3

过滤 Fetch / XHR

只看 API 调用，屏蔽图片 CSS

面板顶部一排 type 过滤按钮：All / Fetch/XHR / JS / CSS / Img / Font。设计师 90% 只关心 Fetch/XHR —— 这就是"代码在问后端要数据"那批请求。点一下，其他全藏。

All Fetch/XHR JS CSS Img Font
↑ 只看 API · 其他全藏
4

点开一条请求 · 看 Headers / Payload / Response

右侧抽屉 · 5 个 tab

点表格里任何一行 → 右边会展开一个面板，里面 5 个 tab：Headers（请求 / 响应头）、Payload（你提交了啥）、Preview（漂亮版 JSON）、Response（原始响应）、Timing。最常用的是 Preview。

Headers Payload Preview Response
▾ data: {
  user: {"Fan"}
  cart: []
}
5

复制 cURL 给工程师

右键 → Copy → Copy as cURL

想让工程师 1 秒复现你这个请求？右键那一行 → Copy → Copy as cURL。粘贴给他，他在终端一跑就能完整重放你这次请求。不用截图、不用描述，他什么都看得见。

cURLcurl 'https://api.app.com/user/me' \
  -H 'Cookie: session=xxx' \
  -H 'Authorization: Bearer ...'
→ 工程师粘贴到终端就能跑
套路这 5 件事按顺序串起来，就是 "页面有问题 → 自查 → 找工程师" 的标准流程：① 打开 Network → ② 刷新看请求 → ③ 过滤 Fetch/XHR → ④ 点开可疑那条看 Response → ⑤ 看不懂就 Copy as cURL 给工程师。每一步都不需要会写代码。
06 / 速查卡

四张卡片
塞进口袋

全篇精华浓缩成四张卡。打开 Network 面板感觉懵 —— 按这个顺序读，几乎不会卡。

PART 01 · 状态码

三位数 · 三种心情

看头一位
2xx · 顺
200 OK · 201 Created · 204 没内容。不用管。
4xx · 你错
401 没登录 · 403 没权限 · 404 地址错 · 400 参数错。前端 / 你的事。
5xx · 它挂
500 后端挂 · 502 网关错 · 503 维护中 · 504 超时。后端的事。
PART 02 · 表头六列

Network 表格

columns explained
Name
请求的地址 / 文件名。/api/user / logo.png。
Status
HTTP 状态码。看头一位即可。
Type
请求类型。fetch/xhr = API 调用；document/script/stylesheet/png = 资源。
Size
响应有多大。0 B / "(failed)" 通常意味着挂了。
Time
请求花了多久。> 1 s 就该警觉，> 5 s 几乎一定有问题。
Waterfall
视觉化时间条。哪条最长就是哪条最慢。

PART 04 · 诊断 4 步

页面没数据时

debug recipe
STEP 1 · 有没有这条请求
Network 里完全找不到 → 前端代码没发出去。是个前端 bug。
STEP 2 · status 是什么
看头一位。4xx = 前端的事；5xx = 后端的事；2xx 还出错 → 看下一步。
STEP 3 · Response 里的字段
2xx 但页面空 → 90% 是 data: [] / data: null。请求成功，但内容是空的。
STEP 4 · 截图给工程师
看不懂就把 整张 Network 截图 + cURL 发给工程师。比"我这边不行"有用 100 倍。
