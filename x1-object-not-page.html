04 / 看更多

3 个新场景，
看人家选哪几个字段。

客户还要露脸——通知、搜索、邮件签名。字段数被限死，每个场景只能挑几个。 下面直接给你答案 + 为什么这几个赢、为什么那几个被砍。

场景 A · 限 2 字段

移动端通知 toast

"手机锁屏上一闪而过——只能放 2 个字段。"
✓ 正解 · NOTIFICATION
张 张明
给你发了消息
2 分钟前
字段订阅 = avatar + name

通知是认出来谁而已——头像 + 名字是"我能不能 0.5 秒内识别这是谁"的最低标准。其他全砍。

✗ company / title — 没空间
✗ phone / email — 通知里 没用
✗ status — 用户不关心
⚠ 陷阱：把 company 塞进去 → 通知变成"张明 · 星河科技 给你发了消息" → 锁屏一闪而过，名字反而看不清。字段越多 = 识别越慢。
场景 B · 限 4 字段

搜索下拉条目

"用户在搜框里输入时，能一眼认出'是不是这个人'。"
✓ 正解 · SEARCH
张 张明
CTO · 星河科技
字段订阅 = avatar + name + title + company

同名"张明"可能有 5 个——身份信息 (title + company) 是区分的关键。头像和名字仍然是认出"这个人"的入口。

✗ phone / email — 搜索结果里 太长干扰，点进详情看
✗ deal_value / status — 跟"识别"无关
⚠ 陷阱：少 company 一个字段 → 5 个"张明"长得一样 → 用户得鼠标 hover 才能知道选哪个。搜索是识别场景，不是预览场景。
场景 C · 限 3 字段

邮件签名引用

"邮件里'和 XX 的合同'自动展开成职业身份。"
✓ 正解 · EMAIL CITATION
张明
CTO，星河科技
zhang@xinghe.cn
字段订阅 = name + title + company

邮件是正式场合——职业身份 (title + company) 比头像更重要。"和谁"的语义在邮件正文里，引用块只补充身份。

✗ avatar — 邮件是文本场景，放图反而像广告
✗ phone — 已在邮件签名里，重复
⚠ 陷阱：不分场景，把 avatar 在所有场景都塞进去 → 邮件签名变图文混排 → 用户邮件客户端可能直接屏蔽外链图片，整个引用挂掉。字段订阅是上下文敏感的。
3 个场景，3 种字段订阅——同一个客户对象，每个表面只露需要的几个字段。下次画稿子，先列"这个表面要订哪几个"，再画。工程师就能直接落 query + view 配置。
05 / 把这件事变成习惯

下次接需求，
先问 3 个问题。

从这一篇开始，"页面思维"和"对象思维"的差距会出现在你每一次需求评审的发言里。

i

这是什么对象？

需求里出现"客户详情页"——脑子里要立刻冒出：客户是一个对象，它有 N 个出现的地方，当前需求只是其中之一。

先问对象，再画页面。

ii

它在别的地方什么样？

设计任何一个出现位置之前，先列出这个对象已经出现在哪些表面。

新加的表面要和老的保持一致——头像规则、状态色、空态文案不能各画各的。

iii

这次用哪几个字段？

每个表面交付时带一份"字段订阅清单"。工程师不用追问，直接落 schema。

设计稿不再是图，是规则。

划重点 · TL;DR

页面是 对象 × 一个使用场景 的产物。从对象出发，页面是结果；从页面出发，对象就被肢解成 N 份不一致的副本。

下次需求评审，试一次——第一句不是"我打算这么画"，而是"这是个什么对象，它已经在哪些地方出现了"。整个会议的语言就变了。
