01 / 问题

为什么你的稿子，
工程师总要来回追问？

因为你交付的是一个页面。但工程师在搭建的是一个数据模型。 语言对不上，每个字段、状态、权限规则都要来回 ping 你。

页面思维 · PAGE

"我设计了一个客户详情页"

稿子里画了详情页布局——头像、姓名、公司、字段表、跟进时间线。看起来完整。

工程师追问：列表里这个客户显示什么？@mention 时呢？通知里呢？搜索结果里呢？卡片里呢？仪表盘上呢？……

你：呃……要再画 6 张稿子？

对象思维 · OBJECT

"我设计了客户这个对象"

稿子里有字段清单（含格式、权限、空态规则），加上一份投影清单——客户在 9 个地方出现时分别用哪几个字段。

工程师收到：对象模型 + 9 个 view 的字段订阅清单。能直接落 schema。

结果：追问 → 0 次。

05 / 把这件事变成习惯

下次接需求，
先问 3 个问题。

从这一篇开始，"页面思维"和"对象思维"的差距会出现在你每一次需求评审的发言里。

i

这是什么对象？

需求里出现"客户详情页"——脑子里要立刻冒出：客户是一个对象，它有 N 个出现的地方，当前需求只是其中之一。

先问对象，再画页面。

ii

它在别的地方什么样？

设计任何一个出现位置之前，先列出这个对象已经出现在哪些表面。

新加的表面要和老的保持一致——头像规则、状态色、空态文案不能各画各的。

iii

这次用哪几个字段？

每个表面交付时带一份"字段订阅清单"。工程师不用追问，直接落 schema。

设计稿不再是图，是规则。

划重点 · TL;DR

页面是 对象 × 一个使用场景 的产物。从对象出发，页面是结果；从页面出发，对象就被肢解成 N 份不一致的副本。

下次需求评审，试一次——第一句不是"我打算这么画"，而是"这是个什么对象，它已经在哪些地方出现了"。整个会议的语言就变了。
