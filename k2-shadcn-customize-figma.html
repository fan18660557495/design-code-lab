02 / 三层地基

shadcn 改造的
三层。

改造不是一拍脑袋全改,有三层。 每一层改起来的难度和影响不一样。 点左边任意一层,看右边解释。

玩法点左边三层中任意一层 —— 右边显示这层改什么、影响哪里、Figma 类比。
L3Token 层 / 设计变量
最上层 · 改起来最轻 · 影响全站
CONSUMES ↓消费下层
L2Base 组件 / 原子件
中间层 · 改起来中等 · 影响所有用它的地方
PROVIDES ↑提供给上层
L1复合组件 / 区块
业务层 · 改起来最重 · 只影响一个场景
L3 · TOKEN

改 token —— 一改全改

token = 一堆"全局变量":主色、文本色、圆角、字号、间距。 所有组件都读这套变量,所以改 token = 改完所有组件。 90% 的"换皮"都在这一层完成 —— 也是性价比最高的改造。

Figma 类比

改 Variables 的值 —— 所有引用这个 variable 的组件 instance 一起变。

改了之后会影响
所有按钮 所有卡片 所有输入框 所有徽章 所有 Modal 复合组件也跟着变

04 / 实操

改造 4 步。

知道了原理之后,具体怎么操作? 按这四步走,从浅到深,每一步都可以停。

01
改 Token · 90% 工作

改一组变量:主色、辅色、文本色、圆角、字号、间距、字体。 这一步搞定就解决了大部分换皮需求。

// tokens.css (改这里) --primary: #0a0a0a → #1677ff; --radius: 6px → 2px; --font: Geist → PingFang;
FIGMA 类比改 Variables 的值
02
调 Base 组件 · 细节

token 不够时,改 base 组件源码:按钮再高一点、Input 加个图标插槽、Card 默认带头部。

// button.tsx (打开这个文件) height: 36px → 40px; + 加一个 loading 状态 + 加一个 icon 插槽
FIGMA 类比进主组件改图层
03
删 / 加组件 · 取舍

不用的删掉(shadcn 给的可能 80% 不需要);没有的自己加(比如复杂表格、特殊上传)。

− 删: drawer (用不到) − 删: command (用不到) + 加: file-upload (业务需要) + 加: complex-table
FIGMA 类比清 library 的主组件
04
拼复合组件 · 沉淀

基于 base 组件拼区块:页头、侧栏、数据卡片、空状态…… 让 AI 直接调用,而不是每次现拼。

// page-header.tsx (新建) import Button from './ui/button' import Badge from './ui/badge' // 拼成: 标题 + 面包屑 + 操作区
FIGMA 类比拼嵌套主组件
提示 不必每步都做 —— 90% 的项目改完 Step 1 就够用了。 Step 4 是规模化(团队 5+ 人)才需要的"预制菜沉淀"。
05 / 升级 + 协作

改完之后,
会有什么坑?

改造完不是结束 —— 还有两个长期问题: shadcn 自己升级了你跟不跟?5-10 个人怎么用同一份?

↑

升级问题

UPGRADE

shadcn 自己升级了 ——
我改过的还在吗?

token 不丢(因为是单独文件); base 组件改过的部分可能要手动 merge(因为你的改动跟新版可能冲突)。

① 跟还是不跟 —— 大部分团队选不跟,锁在当前版本
② 真要升级,用 diff-merge —— AI 帮你看哪里冲突
③ 蓝色丸子:核心组件改得越深,升级越疼
类比 像 detach 了 Figma kit 之后,原 kit 升级了。 你已经改过的图层,要不要同步,人工决定。
⋯

协作问题

COLLAB

5-10 个人都在改 ——
怎么保证大家用同一份?

三个保险:私有 NPM 包统一发布、Git 共享 token 文件、给 AI 写 rules 强制约束。

① 私有 NPM 包:打包发到公司内部 registry,大家 install 用
② Git monorepo:把改造后的 ui 库放在主仓库,大家直接引用
③ AI rules 文档:写一份"必须用 @company/ui",AI 不会乱用 shadcn 原版
类比 像 公司 publish 一个 team library, 大家都 enable —— 自动保持一致,不允许 detach。
06 / 速查

截图保存
这一页。

三张小表:改造深度对照表 + Figma Variables ↔ 代码 token 对应 + 工作量预估。

· 改造深度速查

深度做什么工作量
浅改只改 token (颜色 / 圆角 / 字体)半天
中改token + 调几个 base 组件细节1-2 天
深改token + base + 删/加组件 + 拼复合组件1-2 周
90% 的项目浅改就够了 —— 别一上来就深改,反而失控。

· Figma 变量 ↔ 代码 token

Figma代码 token
Color variable--primary / --background
Number (spacing)--spacing-4 / --padding-x
Number (radius)--radius
Text style--font-size-sm / --font-family
Variable mode同一组 token 的预设(light / dark / brand)
一一对应。你在 Figma 里定的 variables,前端在代码里就这么命名。

· 跟前端这样说

你想要说法 (这样说前端就懂了)
全站换主色 "改 --primary token"
圆角全变大 "改 --radius token"
所有按钮加 4px 高度 "改 button.tsx"(token 改不了高度,要改源码)
单个页面圆角不一样 "在那个组件上加 className override"
公司要用统一一套 "打包成私有 npm 包 @company/ui"
把对的术语说出来,前端就知道你不是外行 —— 沟通效率立刻翻倍。
